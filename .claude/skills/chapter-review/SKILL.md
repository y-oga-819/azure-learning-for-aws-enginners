# chapter-review — 章単位の技術書レビュー

『AWS経験者のためのAzure実践入門』シリーズの 1 章分の原稿を、**技術書の章として**レビューします。
対象の章ファイル(例:`manuscripts/ch04.md`)が指定されていない場合は、まず対象を確認してください。

## 事前に読むもの

レビュー前に必ず次を読み、判断基準を揃えます。

1. 対象章の `docs/outlines/` 該当箇所(構成が凍結アウトラインと一致しているか)
2. `shared/glossary.md`(用語・AWS⇔Azure 対訳・表記ルール)
3. `shared/style-guide.md`(文体・レイアウト規約)
4. `shared/state.md`(前章までの到達状態。この章の前提)
5. 対象章に対応する `handson-logs/chXX/`(ハンズオンの根拠ログ)

機械チェックで拾える指摘は先に潰します:`npm run lint`、`npm run check:links`、`npm run check:todo`。
本スキルは、それらでは拾えない「書籍としての質」を人間の代わりに深くレビューします。

## レビュー観点(チェックリスト)

### A. 構成(5 部構成の遵守)

- 「①このリソースは何か → ②AWS との対比 → ③土台への適用(Before/After) → ④設計の勘所 → ⑤ハンズオン」の順になっているか。
- 各パートが役割を果たしているか(対比パートが薄い、勘所が箇条書きの羅列で終わっている等)。
- アウトライン(凍結)から逸脱していないか。逸脱があれば、それは意図的か。

### B. AWS 経験者への価値(本シリーズの核)

- AWS の対応サービスとの**対比**が具体的か(対応表・「似ている点/決定的に違う点」)。
- AWS 経験者が**つまずくギャップ**(例:リソースグループに真の対応物がない、ID がテナントにある、PaaS が既定でパブリック)を、why まで踏み込んで説明しているか。
- AWS 知識を「使い回せる箇所」と「捨てるべき先入観」を切り分けているか。

### C. ハンズオンの信頼性

- 手順・コマンド・出力が `handson-logs/chXX/` の**検証済みログに基づいている**か。ログにない創作がないか。
- 破壊的操作・課金が発生する操作に注意(`<div class="note">`)があるか。
- 手順が最小限で再現可能か。前提(前章の state)と矛盾しないか。

### D. 正確性・鮮度

- 料金・SKU・上限値・プレビュー/GA 状況などに `<!-- 要検証: 公式Doc確認 -->` が付いているか。付いていない断定がないか。
- 一般に古くなりやすい記述(ポータル UI の文言、プレビュー機能)への配慮があるか。

### E. 用語・文体・可読性

- 用語が `glossary.md` に準拠し、初出時のみ英語併記になっているか。
- です・ます調、コードブロックの言語指定、図の命名規則(`chXX-figYY-説明.png`)を満たすか。
- 一文が長すぎないか、指示語の乱用・二重否定・論理の飛躍がないか。
- 読者が「今どこにいるか」を見失わない導線(節の冒頭のつなぎ、Before/After 図)があるか。

### F. 読者体験(到達感)

- 章の終わりに「何ができるようになったか」が明確か。
- 次章・続巻への接続(伏線)が自然に置かれているか(過剰な煽りになっていないか)。

## 出力フォーマット

1. **総評**(2〜3 文。章の狙いが達成できているか)
2. **重要度順の指摘**。各指摘は次を含める:
   - `重要度`(blocker / major / minor / nit)
   - `場所`(`manuscripts/chXX.md:行` の形式で、クリックできるように)
   - `問題`(何が・なぜ問題か)
   - `提案`(具体的な直し方。可能なら書き換え例)
3. **良い点**(伸ばすべき箇所を必ず 1〜3 点)
4. **マージ判定**:`<!-- 要検証 -->` が残っていないか、blocker がないかを踏まえた可否

指摘は率直に、しかし書き手の成長を後押しする姿勢で。断定できないものは「要確認」として理由を添えます。
