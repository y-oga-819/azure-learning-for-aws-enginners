# book-consistency — 書籍全体の整合性レビュー

『AWS経験者のためのAzure実践入門』を、**1 冊の本(かつ全4冊シリーズの一部)**として通しで点検します。
個々の章の良し悪しではなく、**章と章の間・本文と付録の間・巻と巻の間**で生じる齟齬を見つけるのが役割です。

## 事前に読むもの

- `docs/outlines/`(全アウトライン。設計意図と、章をまたぐ伏線・接続点の設計)
- `shared/glossary.md` / `prh.yml`(用語・表記の単一情報源)
- `shared/state.md`(各章終了時点の環境状態の台帳)
- `manuscripts/` 一式(本文・付録・まえがき・奥付)

まず `npm run lint` を通し、prh で機械的に拾える表記ゆれを潰してから、機械では拾えない整合性を見ます。

## レビュー観点(チェックリスト)

### A. 用語・表記の一貫性(通し)

- 同じ概念が章によって別語で呼ばれていないか(例:「土台」と「基盤」と「ランディングゾーン」の揺れ)。
- `glossary.md` にない重要語が本文で多用されていないか(あれば glossary へ追加を提案)。
- 英語併記のルール(初出のみ)が本全体で一貫しているか。同じ語を各章で毎回展開していないか。
- 略語(NSG・RBAC・VNet 等)の初出定義が、本の中で最初に現れる箇所にあるか。

### B. 章間の接続(ストーリーの連続性)

- 各章の「土台への適用(Before/After)」が、前章の After と噛み合っているか(段差・飛躍がないか)。
- 前章で「後で説明する」とした事項が、後の章で回収されているか(未回収の予告がないか)。
- 逆に、まだ説明していない概念を前提にして書いていないか(前方参照の事故)。

### C. state.md との一致

- 各章末で `shared/state.md` に追記された「到達状態」が、実際の章のハンズオン結果と一致しているか。
- 章 N の記述が、`state.md` の「章 N-1 終了時点」を正しい前提にしているか。
- 第7章の到達状態(巻末チェックリスト/構成図)が、各章の積み上げの総和と一致しているか。

### D. 本文 ⇔ 付録の整合

- 付録 A(AWS⇔Azure 対応表)・付録 B(用語対訳)が本文・`glossary.md` と矛盾しないか。
- 付録 C(命名規則)が、本文のハンズオンで実際に使っている命名と一致しているか。
- 付録 E(リソースグループ分離への読み替え)が、全章のサブスクリプション分離手順を漏れなくカバーしているか。

### E. シリーズ(続巻)への接続

- 第7章の「続巻への分岐ガイド」と、各続巻のアウトラインの前提が食い違っていないか。
- 続巻の序章に再掲される想定の「到達状態の構成図」が、本書の完成形と一致しているか。
- 本書で「続巻で扱う」とした伏線(例:Private Endpoint の重要性)が、後続の期待と整合しているか。

### F. 重複・矛盾・過不足

- 同じ説明が複数章で重複していないか(意図的な再掲か、冗長か)。
- 数値・設計判断が章によって食い違っていないか(例:CIDR 設計、リージョン選定)。
- 章の分量バランス(各章 約25p 想定)から大きく外れた章はないか。

## 出力フォーマット

1. **総括**(本として通っているか。最も重大な整合性リスク上位 3 点)
2. **カテゴリ別の指摘**(A〜F)。各指摘に:
   - `場所`(関係する複数箇所を `manuscripts/chXX.md:行` 形式で。整合性は「どこと、どこが」矛盾かを両方示す)
   - `齟齬の内容`
   - `解決案`(どちらに合わせるべきか、glossary/state のどちらを直すか)
3. **単一情報源の更新提案**(`glossary.md` / `prh.yml` / `state.md` に反映すべき事項をまとめる)
4. **全体マージ可否**(通し校正としての結論)

矛盾は「どちらが正か」を独断で決めず、根拠(アウトライン・state・公式仕様)を示して推奨を述べます。判断に確証がない箇所は「要確認」とし、確認すべき対象を具体的に挙げます。
