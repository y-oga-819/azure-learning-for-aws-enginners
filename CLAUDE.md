# CLAUDE.md — AI アシスタント向け執筆ルール

このリポジトリで原稿を書く・直すときは、以下のルールに必ず従ってください。これは人間のレビュー負荷を下げ、シリーズ全体の一貫性を保つための取り決めです。

## 0. リポジトリの目的

『AWS経験者のためのAzure実践入門』シリーズ(全4冊)の原稿を Markdown で管理し、Vivliostyle で PDF 組版します。本リポジトリは第1冊【共通基礎編】です。

## 1. 執筆前に必ず読むファイル

章を書き始める前に、以下を必ず読んでから着手してください。

1. `docs/outlines/` の**該当章のアウトライン**(設計書として凍結扱い。勝手に構成を変えない)
2. `shared/glossary.md`(用語の単一情報源・AWS⇔Azure 対訳・表記ルール)
3. `shared/style-guide.md`(文体・レイアウトの約束事)
4. `shared/state.md`(前章までに構築済みの環境状態)

## 2. 章の構成(必須の 5 部構成)

すべての章は、必ず次の順序で構成します。順序の入れ替え・省略はしません。

1. **このリソースは何か** — Azure における位置づけと設計思想
2. **AWS との対比** — 対応サービス、似ている点、決定的に違う点
3. **土台への適用(Before / After)** — 前章までの土台がどう育つか
4. **設計の勘所** — セキュリティ・統制・コストのチェックリスト
5. **ハンズオン** — 手を動かす最小手順

## 3. ハンズオンの根拠(創作の禁止)

- ハンズオン手順は、`handson-logs/chXX/` にある**検証済みログを根拠**とします。
- ログにない手順・コマンド・出力を**創作しない**でください。
- 根拠となるログがまだない箇所には、ドラフトに `<!-- 要検証 -->` コメントを付けます。

## 4. 未確定情報のマーキング

- 料金・SKU 名・上限値・プレビュー / GA 状況などの**数値・製品情報**には、必ず `<!-- 要検証: 公式Doc確認 -->` を付けます。
- これらのマーカーの残存件数は CI(`npm run check:todo`)で集計されます。**`<!-- 要検証 -->` が残ったままの章はマージしません。**

## 5. 用語と文体

- 用語は `shared/glossary.md` に従います。**初出時のみ英語併記**(例:「マネージド ID(Managed Identity)」)。
- 文体は**です・ます調**に統一します。
- 表記ゆれは `prh.yml` と textlint で機械チェックされます(`npm run lint`)。
- コードブロックには**必ず言語指定**を付けます。

## 6. 章を書き終えたら

- `shared/state.md` の該当章セクションに、「**この章終了時点で存在するリソースと設定**」(作成したリソース / RBAC・ポリシー変更 / ネットワーク変更 / コスト影響)を追記します。
- 次章はこの state を前提に書くため、更新漏れは致命的です。

## 7. 提示の粒度(レビュー可能な単位で)

- **1 つの返答で章全体を書き上げない**でください。
- **節(5 部構成の 1 パート、または小見出し)単位**で提示し、人間のレビューを待ってから次へ進みます。

## 8. コミット前のローカルチェック

原稿を変更したら、コミット前に以下が通ることを確認してください(CI でも実行されます)。

```bash
npm run lint         # textlint(表記ゆれ・文体・技術文書ルール)
npm run check:links  # ローカルリンク・画像参照の切れチェック
npm run check:todo   # 「要検証」マーカーの残存集計(レポート)
npm run build        # Vivliostyle で PDF がビルドできるか
```

## 9. レビュー用スキル

執筆・校正の際は、次のプロジェクトスキルを活用できます(`.claude/skills/`)。

- **`chapter-review`** — 1 章分を「技術書の章」として多角的にレビュー(5 部構成・AWS 対比・ハンズオン根拠・用語・要検証)。
- **`book-consistency`** — 章をまたいだ整合性(前章との接続・state.md との一致・表記ゆれ・伏線回収・続巻への接続)をレビュー。
