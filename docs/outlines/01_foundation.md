# 第1章 Azureの全体像 — AWSと何が根本的に違うのか

- **説明**:テナント/管理グループ/サブスクリプション/リソースグループの4階層、ARM(すべてのデプロイが通る共通基盤)、リージョン・可用性ゾーン・リージョンペア
- **AWS対比**
  - AWSアカウント ⇔ サブスクリプション(課金・ポリシー境界としての使われ方の差)
  - リソースグループには真の対応物がない(タグ+CFnスタックの中間)
  - 「IaaSから育ったAWS、エンタープライズID管理から育ったAzure」という文化的背景(why)— この違いが以降すべての章の伏線になる
  - 東日本/西日本リージョンペアという概念がDR設計に与える影響(AWSにない制約)
- **土台への適用**:dev/prod分離方式の設計判断。本書は「サブスクリプション分離」を主軸とし、個人学習・小規模向けに「リソースグループ分離」への読み替え表を提示(社内DX編はこちらを採用する旨を予告)
- **設計の勘所**:CAF準拠の命名規則を最初に決める(後から変えられない)、リージョン選定
- **ハンズオン**:アカウント作成、サブスクリプション確認、命名規則シートの作成、最初のリソースグループ作成

# 第2章 Microsoft Entra ID と RBAC — IAMとは別物と思え

- **説明**:Entra ID(テナント全体のIDディレクトリ)、ユーザー/グループ(セキュリティグループ・動的グループ)、Azure RBAC(スコープ×ロール×プリンシパル)、マネージドID、PIM(特権のJIT昇格)概説
- **AWS対比**
  - IAMユーザー/ロール/ポリシーとの対応と決定的な違い:IDがアカウント内ではなくテナントに存在する
  - JSONで細粒度制御するIAMポリシー vs 「組み込みロール+スコープ」で考えるRBACへの発想転換
  - マネージドID ≒ インスタンスプロファイル/IRSAだが適用範囲が広く、「キー認証を根絶する」文化の中心にある
  - コラム:オンプレADとの同期(Entra Connect)の存在(実案件の頻出前提知識)
- **土台への適用**:管理者/開発者/閲覧者グループを作成し、dev/prodに異なるRBACを割り当てる(devは開発者に共同作成者、prodは閲覧者のみ、等)
- **設計の勘所**:最小権限、個人への直接割り当て禁止(グループ経由原則)、本番の特権はPIMで一時昇格
- **ハンズオン**:グループ作成→RBAC割り当て→権限差の動作確認

# 第3章 ガバナンス — 管理グループと Azure Policy

- **説明**:管理グループ階層の設計、Azure Policy(監査/拒否/自動修復)、イニシアティブ、タグ戦略
- **AWS対比**:Organizations+SCP+AWS Configとの対応。SCPは「上限の設定」、Policyは「監査・強制・修復」まで担う守備範囲の違い。Config Rules⇔Policy監査モードの対応
- **土台への適用**:管理グループ配下にdev/prodを整理し、「必須タグ」「リージョン制限」「パブリックIP作成の制限(prodのみ)」を適用。**devは緩く・prodは厳しく、をポリシーで表現する**のが本章の核
- **設計の勘所**:いきなり拒否にせず監査→拒否の段階導入、除外(exemption)の管理
- **ハンズオン**:必須タグポリシー適用→違反リソースの作成が拒否されることを確認

# 第4章 仮想ネットワーク — VNetで土台のネットワーク骨格を作る

- **説明**:VNet、サブネット、NSG/ASG、ルートテーブル(UDR)、VNetピアリング、NAT Gateway、Hub-Spokeトポロジ
- **AWS対比**
  - VPC/サブネット/SG/NACLとの対応表
  - 重要な違い:パブリック/プライベートサブネットの明確な区別がない、インターネットGWの明示アタッチ不要、NSGはサブネットとNIC両方に付く
  - Hub-Spoke ⇔ Transit Gateway構成の対比
  - 予告:AzureのPaaSは既定でパブリック公開のため、続巻で学ぶPrivate Endpointの重要度がAWS以上に高い(本シリーズ最重要の伏線)
- **土台への適用**:将来のピアリング・ハイブリッド接続を見越したCIDR設計表を作り、`vnet-dev` / `vnet-prod`(app用/data用サブネット+デフォルト拒否寄りのNSG)を構築
- **設計の勘所**:アドレス空間の重複禁止、NSGの管理単位、サブネットの切りすぎ注意
- **ハンズオン**:VNet+NSG構築、一時VM+Bastionで疎通確認(VM自体の詳説は続巻へ)

# 第5章 監視とコストの土台 — Log Analytics ワークスペースと予算

- **説明**:Azure Monitorの全体像(メトリック/ログの二層)、Log Analyticsワークスペース、診断設定、アクティビティログ、KQL入門、Cost Management と予算アラート
- **AWS対比**:CloudWatch(Metrics/Logs)+CloudTrail+Cost Explorerとの対応。「ワークスペースに全ログを集約する」設計文化、KQL⇔Logs Insightsクエリの読み替え
- **土台への適用**:dev/prod共通の監視用リソースグループにワークスペースを作成し、「新規リソースは必ず診断設定でここへ送る」規約を確立(ポリシーによる自動化も紹介)。月額予算アラートを設定
- **設計の勘所**:ワークスペースを分けるか集約するかの判断基準、ログ保持期間とコスト、学習用の課金事故防止(本書読者の実利として重要)
- **ハンズオン**:ワークスペース作成→アクティビティログ送信→KQLで操作履歴を検索→予算アラート設定

# 第6章 Bicep入門 — IaCの「型」だけ先に覚える

- **説明**:ARMテンプレートとBicepの関係、リソース定義・パラメータ・出力の基本文法、what-ifデプロイ、ステートファイルが不要な理由(ARMが状態を持つ)
- **AWS対比**:CloudFormation/CDKとの対応。「Bicep ≒ CFnの人間に優しいDSL」、ドリフト・スタックの考え方の違い、Terraform azurermの位置づけ(本シリーズはBicep主軸の理由)
- **土台への適用**:第4章で作ったVNet・リソースグループをBicep化し、「コードから土台を再現できる」状態にする。**本章はあくまで「型」の習得に留め、モジュール化・パイプライン統合などの本格運用は各続巻の終盤で扱う**(二段構えの明示)
- **設計の勘所**:ポータルで作る→エクスポート→Bicep整形、という現場の現実的ワークフロー
- **ハンズオン**:VNetのBicep化→一度削除→コードから再構築(再現性の体験)

# 第7章 総仕上げ — ミニランディングゾーンの完成と続巻への分岐

- **説明**:完成した土台の全体構成図、CAFエンタープライズ・ランディングゾーンとの対応(自分たちが作ったものが何の縮小版なのかの種明かし)、Well-Architected Framework(両クラウド)の紹介
- **AWS対比**:Control Tower / Landing Zone Acceleratorとの対応。「AWSでマルチアカウント戦略を組んだ経験」がAzureのどこに活きるかの総まとめ
- **到達状態の確認**:構成図1枚+チェックリスト(この図が続巻すべての序章に再掲される)
- **続巻への分岐ガイド**:
  - 公開サービス・SaaSを作る → サービス公開編へ
  - 社内業務アプリを作る → 社内DX編へ
  - 両方読む場合の推奨順序と、重複箇所の読み飛ばしマップ
- **ハンズオン**:到達状態チェックリストによるセルフ監査、土台一式のBicepコードのタグ付けとリポジトリ整理

# 補足:本書の設計判断(トレードオフ)

1. **「アプリなしの土台だけ」で1冊にする**:退屈化のリスクがあるため、「ミニランディングゾーン完成」という独自ゴールと、AWSマルチアカウント経験との対比(第7章)で到達感を作る。アプリを少しでも載せる案は、続巻との重複と本書の焦点ぼけを招くため棄却。
2. **Bicepを基礎編に入れる(型のみ)**:シリーズ当初の「ポータル先行」方針とは半分矛盾するが、最単純リソース(VNet・RG)で型だけ体験→本格IaC化は各続巻終盤、の二段構えで学習負荷を抑えた。土台の再現性(削除→再構築)は基礎編のゴールとも相性が良い。
3. **監視・コストを基礎編に前倒し**:従来は本番化フェーズの話題だったが、「診断設定を最初から規約化する」方が現実の統制に近く、学習者の課金事故防止という実利もあるため移動。
4. **想定分量**:7章×約25p+付録 ≒ 200p前後。
