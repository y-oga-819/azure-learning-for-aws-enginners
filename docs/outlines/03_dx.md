# 0. 本書の位置づけと設計方針

## 前提と読み方(基礎編との接続)

- 前提:【共通基礎編】で構築したミニランディングゾーン(RBAC設計、ポリシー、VNet、Log Analyticsワークスペース、Bicepの型)
- **序章に以下を収録(シリーズ共通の緩和策)**:
  1. **「基礎編の到達状態」1枚構成図** — 本書はこの状態から開始する
  2. **用語ミニ対訳表**(15語程度)
  3. **スキップ判定チェックリスト10問** — 実技寄りの判定(例:マネージドIDでキーレス接続を構成できる、Policyで必須タグを強制できる)。解答集は付録にあり、基礎編の該当章へ誘導
  4. **本書の環境方針**:社内アプリの規模感に合わせ「1サブスクリプション+dev/prodリソースグループ分離」を採用。基礎編(サブスクリプション分離)からの読み替えは基礎編・付録Eの対応表を使用
  5. **環境準備ハンズオン**:部署を模したEntraグループ(全社員/管理部署/機密取扱部署)の作成 — 後半の認可章への伏線

## 題材システム

架空の「社内申請・承認ポータル」(利用者:全社500名、うち管理部署30名)

- 構成:Webアプリ(App Service)+ RDB + 添付ファイル(Blob)+ 通知・自動処理(Functions/Logic Apps)
- 認証認可のユースケースを要件に織り込む:閲覧は全社員/承認・マスタ編集は管理部署のみ/機密申請は特定部署のみ閲覧/管理部署にだけ管理メニュー表示
- AWSでの相当構成(読者の既存イメージ):内部ALB + IAM Identity Center/Cognito + Beanstalk + RDS + S3 + Lambda

## 本書の主役(公開サービス編との違い)

- 公開サービス編:スケーラビリティ、公開Web向けセキュリティ(WAF・Front Door)
- **本書:Entra IDによる認証・認可、社内向け公開パターンの選択、App Service、小規模運用のコスト最適化**

# Phase 1: アプリ構築 — App Serviceを主軸に

## 第1章 App Service 徹底解説 — 本書のコンピューティング基盤

- **説明**:App Service Planとアプリの分離、SKU選定、デプロイスロット、ヘルスチェック、Always On。**VNet統合の二方向(送信=Regional VNet Integration/受信=Private Endpoint・アクセス制限)の区別**(本書最重要の基礎)
- **AWS対比**:Elastic Beanstalk/App Runnerとの対応、スロット⇔Blue/Green、「AzureではApp Serviceが主役級」という文化。公開サービス編がコンテナを選んだ理由との対比で、本件でApp Serviceを選ぶ理由(小規模・運用体制・PaaSの手軽さ)を示す
- **環境への適用**:基礎編のvnet-dev(読み替え後)にApp Service(スロット付き)を構築し、サンプルアプリをデプロイ
- **設計の勘所**:社内アプリのSKU選定(Basic〜Premium v3)、自動スケールは控えめで十分という判断基準
- **ハンズオン**:デプロイ→スロットスワップ体験

## 第2章 データベース — 小規模社内アプリに最適な選択

- **説明**:Azure SQL Database(サーバーレス・自動一時停止)、PostgreSQL Flexible Server(停止機能)
- **AWS対比**:RDS/Aurora Serverlessとの対応。夜間・休日ほぼ無負荷の社内アプリでのサーバーレス/自動停止のコストメリット
- **環境への適用**:題材DBを構築し、App ServiceからVNet統合(送信)経由で接続
- **設計の勘所**:Entra ID認証でのDB接続(SQLログインを作らない)、バックアップとPITR
- **ハンズオン**:DB作成→Entra ID認証で接続→自動一時停止の挙動確認

## 第3章 Blob と Key Vault — 添付ファイルとシークレットレス構成

- **説明**:ストレージアカウント(Blob中心)、SAS、Key Vault(RBACモード)
- **AWS対比**:S3/Secrets Manager+KMSとの対応(要点のみ)、「アカウント」概念の違い
- **環境への適用**:申請の添付ファイル保存先としてBlob追加。接続情報をKey Vaultへ、アクセスはすべてマネージドID(パスワード・アクセスキーゼロの構成に)
- **設計の勘所**:アクセスキー無効化、ソフトデリート、機密文書のライフサイクル
- **ハンズオン**:マネージドIDでのBlobアクセス、SASでの一時ダウンロードURL発行

## 第4章 Functions と Logic Apps — 社内業務の自動化

- **説明**:Functions(タイマー/キュートリガー)、Logic Apps(ローコード連携)、両者の使い分け
- **AWS対比**:Lambda/Step Functions/EventBridgeとの対応。「Logic Apps+コネクタ文化」はAWSにない強み(Teams通知・Outlook連携が数クリック)
- **環境への適用**:承認時のTeams/メール通知、夜間の集計バッチを追加
- **設計の勘所**:Logic Apps・Power Platformとの住み分け(コラム:Power Apps/Automateとの境界線 — 社内DXで必ず問われる)
- **ハンズオン**:承認イベント→Teams通知のフロー構築
- **Phase 1完了時点の構成図**:dev一式が動くが、まだ「誰でもアクセスできる」状態 → Phase 2へ

# Phase 2: 公開・認証・認可 — 本書の核

## 第5章 社内アプリの公開パターン設計 — 4つの選択肢と決め方

手を動かす前に、選択肢の全体像と決定基準を示す設計章。

- **パターンA:インターネット公開 + Entra ID認証 + 条件付きアクセス(ゼロトラスト型)**
  - メリット:VPN不要でリモートワークに強い、構築運用コスト最小、権限管理がEntraで完結
  - デメリット:URLは外部から見える、アプリ脆弱性が直接露出、「閉域必須」規程の企業では不可
- **パターンB:完全閉域 + VPN / ExpressRoute(境界防御型)**
  - メリット:ネットワーク到達性ごと遮断する堅牢さ、監査・規程対応が容易、機密データ向き
  - デメリット:VPN Gateway/ExpressRouteのコストと運用、リモートワークUX低下、結局認証も必要
- **パターンC:ハイブリッド(Front Door Premium + Private Link)**
  - メリット:オリジン閉域+インターネットアクセス両立、社外公開への発展性
  - デメリット:コスト高、500名規模にはオーバースペックになりがち
- **パターンD:App Serviceアクセス制限(会社出口IPのみ許可)**
  - メリット:最安最速 / デメリット:ゼロトラスト観点で非推奨、暫定措置の位置づけ
- **セオリー**:近年の主流はA(ゼロトラスト型)。機密区分・社内規程・既存NW資産でBを選ぶ判断をデシジョンツリー化
- **AWS対比**:A⇔ALB+Cognito(ただし条件付きアクセスの細かさはEntraの独壇場)、B⇔内部ALB+Direct Connect/Client VPN
- **環境への適用**:題材はAを主軸採用、第8章でBへの拡張も実装する二段構成
- **ハンズオン**:なし(設計章)。自組織の要件でデシジョンツリーを辿る演習

## 第6章 認証 — Easy Auth と Entra ID アプリ登録

- **説明**:Easy Auth(コード変更ゼロで全リクエスト認証必須)、アプリ登録とエンタープライズアプリケーション、OpenID Connectフロー、MSAL、条件付きアクセス(MFA必須・準拠デバイス・国外ブロック・リスク評価)
- **AWS対比**:Cognito+ALB認証との対応。「Cognitoにユーザーを登録する」のではなく「既にいる全社員がそのまま使える」違い(why:社内DXでEntraが圧倒的に楽な理由)。External ID(公開サービス編第8章)との違い=顧客ID基盤 vs 社員ID基盤、の整理表
- **環境への適用**:Easy Authを構成し、未認証アクセスが全てEntraログインへリダイレクトされる状態に。条件付きアクセスでMFA強制
- **設計の勘所**:トークンの取り扱い(X-MS-CLIENT-PRINCIPAL)、セッション有効期限、ゲスト(B2B)を許可するか
- **ハンズオン**:Easy Auth有効化→社員アカウントでログイン→クレーム確認

## 第7章 認可 — 「誰に何をどこまで」を設計する

ユースケース駆動の本書ハイライト章。

- **基本部品**:App Roles、グループクレーム、スコープ、Azure RBAC(インフラ用)との役割分担
- **ユースケース別実装パターン**:
  1. 閲覧は全社・編集は特定部署のみ → App Roles(`Reader`/`Approver`)をグループに割り当て
  2. 機密申請は特定部署のみ閲覧(データレベル認可) → グループクレーム+アプリ側データフィルタ。**トークンだけでは足りず、アプリ実装が必要になる境界を明確化**(本書はインフラ側=クレーム設計まで。アプリ実装の詳細はアプリアーキテクチャ編第3章と同型の考え方)
  3. 特定部署にだけ機能・画面を追加表示 → ロールによるUI出し分け+App Configurationフィーチャーフラグ
  4. 管理者権限の一時付与 → PIMの応用
  5. 代理承認・委任 → アプリで委任テーブルを持つ(IDaaSの守備範囲外と示す)
- **設計原則**:グループ直接参照よりApp Roles経由を推奨する理由(グループ乱立・トークン肥大回避)、認証と認可の責務分離
- **AWS対比**:Cognitoグループ+Verified Permissionsとの対応。「グループ→App Roles→クレーム」パイプラインの素直さ
- **環境への適用**:`Reader`/`Approver`/`Admin`のApp Rolesを定義し、序章で作った部署グループに割り当て。承認ボタンと管理メニューの出し分け
- **ハンズオン**:App Roles定義→割り当て→ロール別の画面差確認→フィーチャーフラグで新機能を管理部署だけに公開

## 第8章 閉域化 — パターンBへの拡張(Private Endpoint と VPN/ExpressRoute)

- **説明**:Private Endpoint/プライベートDNSゾーン、App Serviceの受信閉域化、VPN Gateway(P2S/S2S)、ExpressRouteの概要とコスト感
- **AWS対比**:VPC Endpoint/Client VPN/Direct Connectとの対応。**PaaSが既定パブリックのAzureでは閉域化を意識的に行う**(基礎編からの伏線、公開サービス編第11章と共通の最重要ポイント)
- **環境への適用**:prod環境を構築し、DB・Blob・Key VaultをPrivate Endpoint化。App Service受信も閉域化する構成(パターンB)を追加実装し、P2S VPNで接続確認
- **設計の勘所**:「A採用でもデータ層のPrivate Endpoint化は必須」というハイブリッドな現実解
- **ハンズオン**:DB閉域化→パブリック経路遮断確認→P2S VPN接続体験

# Phase 3: 運用と完成

## 第9章 監視とコスト最適化 — 小さく運用する技術

- **説明**:Application Insights(要点)、**Entra側の監視(サインインログ・監査ログ)— 社内アプリならではの主役**、コスト最適化(サーバーレスDB・Plan共有・dev夜間停止自動化・予算アラート)
- **AWS対比**:CloudWatch+CloudTrail、IAM Access Analyzerとの対応
- **環境への適用**:基礎編のワークスペースに診断設定を集約し、「誰がいつログインし何を承認したか」を追える監査基盤を構築
- **ハンズオン**:KQLで監査ログ検索(特定ユーザーの操作追跡)、dev自動停止の構成

## 第10章 CI/CD と IaC — 少人数チームの現実解

- **説明**:GitHub Actions+OIDC連携、スロットを使った安全なリリース、環境のBicep化
- **「ポータルで作ったものをIaC化する」現実的ワークフローを正面から扱う**:基礎編第6章で覚えた「型」を使い、エクスポート→Bicep整形→what-if→取り込みの手順を確立(ポータル先行で作ってきた本書の流れと整合)
- **AWS対比**:CodePipeline/CloudFormationとの対応(要点)
- **環境への適用**:dev自動デプロイ→スロット検証→承認→prodスワップのパイプライン、環境一式のBicep化
- **ハンズオン**:一連のパイプライン体験

## 第11章 総仕上げ — 社内DXアーキテクチャパターン集

- **説明**:完成構成図とWell-Architectedセルフレビュー(セキュリティ・コスト柱を重点)
- **発展パターン集**:M365/Teams統合、Power Platformとの共存、SharePoint/OneDrive連携、全社ポータルへの成長(A→C移行パス)、複数アプリのPlan共有戦略
- **シリーズ接続**:利用拡大でスケーラビリティが必要になったら(コンテナ化・Front Door)公開サービス編へ、アプリ内の認可・機能設計を深めるならアプリアーキテクチャ編へ
- **ハンズオン**:構成図作成とセルフアセスメント

# 補足:本書の設計判断(トレードオフ)

1. **基礎トピックは基礎編に委ね、全11章構成に(約220p)**:土台の圧縮再掲を持たない分、App Serviceと認証認可という本書の核に集中でき、焦点が明確になっている。
