# 0. 本書の前提と設計方針

## 各編の責務分離

| 編 | 責務 | 問いの形 |
|---|---|---|
| サービス公開編 | インフラ構築(サービス公開型) | 「リソースをどう立て、どう安全に繋ぐか」 |
| 社内DX編 | インフラ構築(社内DX型) | 「社内アプリをどう公開し、誰に何を許すか」 |
| **アプリケーションアーキテクチャ編(本書)** | **アプリケーションアーキテクチャ** | **「そのインフラの上で、機能をどう設計・実装するか」** |

- 前提:サービス公開編の完成環境(Container Apps + PostgreSQL + Blob + Key Vault + External ID + Front Door)を出発点とする
- ただし本書も**単体で読める**よう、第1章で前提構成を圧縮再掲する(社内DX編と同じ方針)
- インフラ視点(SKU・ネットワーク・プロビジョニング)は各章冒頭で最小限触れ、詳細はサービス公開編へ参照を付す

## 題材システムの発展

サービス公開編の「タスク管理SaaS」を**マルチテナントSaaSとして本格化**させる:

- 複数企業(テナント)が契約して使う
- Free / Standard / Enterprise の3プラン(プランで機能・上限が変わる)
- 全文検索、リアルタイム更新、通知、月次の利用量集計と請求
- AWSでの相当スタック(読者の既存イメージ):Cognito + API Gateway + ElastiCache + SQS/SNS/EventBridge + DynamoDB + OpenSearch

## 各章の共通フォーマット(本書版)

1. **解決したい設計課題** — 題材SaaSで直面する具体的な要求から始める
2. **設計パターンの選択肢** — 定番パターンとトレードオフ
3. **Azureでの実現** — どのサービスのどの機能で実装するか(AWSでの実現との対比)
4. **実装** — コード付きハンズオン(C#/.NET主軸、要点はTypeScript併記)
5. **落とし穴と運用** — 本番で踏みがちな問題

# Phase 1: 骨格 — マルチテナントとID・APIの設計

## 第1章 出発点の確認とアプリケーションアーキテクチャの地図

- サービス公開編の完成構成の圧縮再掲(構成図1枚+各リソースの役割3行ずつ)
- 「インフラは同じでも、アプリ設計で性能・コスト・保守性は桁で変わる」ことを示す導入例
- 本書で扱う設計判断の全体マップ(各章がどの判断に対応するか)
- AWS対比:Well-Architected(両クラウド)のアプリ寄り項目の読み比べ
- ハンズオン:題材リポジトリのセットアップ、サービス公開編環境の再現(Bicep一発デプロイ)

## 第2章 マルチテナント設計 — 分離モデルという最重要判断

- **設計課題**:テナント間のデータ漏洩ゼロ・ノイジーネイバー対策・テナント別のコスト把握
- **選択肢**:①DB分離(テナント毎DB)②スキーマ分離 ③行レベル分離(共有DB+tenant_id)④ハイブリッド(EnterpriseだけDB分離)
- **Azureでの実現**:PostgreSQLの行レベルセキュリティ(RLS)、Azure SQLのエラスティックプール(テナント毎DBのコスト問題を解く仕組み)、Cosmos DBのパーティションキー=テナントIDという設計
- AWS対比:RDS+RLS/DynamoDBのテナント分離パターン、AWS SaaS Lensとの対応
- 実装:行レベル分離を主軸に、テナントコンテキストの伝搬(ミドルウェアでtenant_idを解決しRLSに接続)
- 落とし穴:tenant_idの付け忘れを型で防ぐ、テナント横断クエリ(管理画面)の扱い

## 第3章 認証・認可の実装 — External IDトークンをアプリで捌く

- **設計課題**:サービス公開編で立てたExternal ID基盤から受け取ったトークンを、テナント・ロール・プランの判定にどう使うか(サービス公開編との接続点=「トークンを受け取った後」から本書の担当)
- **選択肢**:トークンにどこまで載せるか(クレーム肥大 vs 都度DB参照)、認可の実装位置(ミドルウェア/ドメイン層/DB)
- **Azureでの実現**:External IDのカスタムクレーム、アプリ側のポリシーベース認可(.NET Authorization Policy)、テナント管理者によるメンバー招待フロー(B2Bとの違い)
- AWS対比:Cognitoのカスタムクレーム+Lambda Authorizer/Verified Permissionsとの対応
- 実装:「テナント管理者/一般メンバー」ロール、招待→参加フロー
- 落とし穴:トークンキャッシュと権限剥奪の遅延、テナント切り替え(1ユーザーが複数テナント所属)

## 第4章 APIアーキテクチャ — API Management で公開APIを製品化する

- **設計課題**:自社フロントエンド用APIと、顧客向け公開API(Enterpriseプラン特典)を同じバックエンドから安全に提供する
- **選択肢**:APIM経由に統一するか、公開APIのみAPIMか。BFFパターンの要否
- **Azureでの実現**:API Management(製品/サブスクリプションキー/ポリシーによるレート制限・変換)、バージョニング戦略、OpenAPI連携
- AWS対比:API Gateway(usage plan / APIキー / ステージ)との対応。APIMの「製品」概念はAWSにない粒度
- 実装:プラン別レート制限(Free: 60req/分、Enterprise: 無制限)をAPIMポリシーで実現
- 落とし穴:APIMのコスト(小規模ではオーバースペックになる境界)、認証の二重化を避ける構成
- ※インフラとしてのAPIM構築手順はサービス公開編を参照

# Phase 2: データと状態 — キャッシュ・NoSQL・検索・非同期

## 第5章 キャッシュ戦略 — Azure Cache for Redis を使い倒す

- **設計課題**:ダッシュボードの重い集計クエリ、セッション状態、多重実行の防止
- **選択肢**:cache-aside / write-through、キャッシュ対象と TTL の決め方、「キャッシュしない」判断基準
- **Azureでの実現**:Azure Cache for Redis / Azure Managed Redis(新世代)の使い分け、テナント毎のキー設計(`tenant:{id}:...`)、分散ロック
- AWS対比:ElastiCache(Redis OSS/Valkey)との対応。Valkeyフォーク以降のエンジン事情とAzure側の対応状況(執筆時要確認の注記)
- 実装:集計APIへのcache-aside導入とヒット率計測、Redisセッションプロバイダ
- 落とし穴:キャッシュ不整合(更新時の無効化戦略)、テナント間のキー衝突、サンダリングハード

## 第6章 NoSQLモデリング — Cosmos DB をDynamoDB経験者の目で設計する

- **設計課題**:アクティビティフィード(時系列大量書き込み・テナント毎読み出し)をRDBから分離する
- **選択肢**:RDBに留まる/Cosmos DB/テーブルストレージ。ポリグロット化の損益分岐
- **Azureでの実現**:パーティションキー設計(=DynamoDBのPK設計と同じ思考が使える)、RU/sというキャパシティモデル、変更フィード
- AWS対比:DynamoDBとの徹底比較 — シングルテーブル設計の思想はCosmosでどう変わるか、RU vs RCU/WCU、GSI vs 複合インデックス、DynamoDB Streams vs 変更フィード
- 実装:フィードをCosmosに実装、RUの実測とコスト見積もり
- 落とし穴:クロスパーティションクエリの罰金、ホットパーティション(tenant_id偏り)対策

## 第7章 全文検索 — Azure AI Search で検索機能を作る

- **設計課題**:タスク・コメント・添付ファイル本文を横断する全文検索(日本語対応)
- **選択肢**:DBのLIKE/pg_trgmで粘る境界、専用検索エンジンへの移行判断
- **Azureでの実現**:AI Searchのインデックス設計、日本語アナライザー、インデクサー(DB/Blobからの自動取り込み)vs 自前同期(変更イベント駆動)、セキュリティフィルタ(テナント分離をfilterで強制)
- AWS対比:OpenSearch(マネージド/Serverless)との対応。「クラスタを運用するOpenSearch vs フルマネージドのAI Search」という運用哲学の差、コストモデルの違い
- 実装:検索インデックス構築、テナントフィルタ付き検索API
- 落とし穴:インデックス遅延とUX、再インデックス戦略、ベクトル検索・RAGへの拡張余地(概説)

## 第8章 非同期処理とイベント駆動 — Service Bus / Event Grid を軸に

- **設計課題**:通知送信・重い集計・外部連携を同期APIから引き剥がし、失敗に強くする
- **選択肢**:キュー(コマンド)とイベント(事実の通知)の使い分け、コレオグラフィ vs オーケストレーション
- **Azureでの実現**:Service Bus(キュー/トピック、セッション、DLQ)、Event Grid(イベントルーティング)、Event Hubs(ストリーム)の三兄弟の使い分け、Durable Functions(サーガ/長時間処理)
- AWS対比:SQS/SNS/EventBridge/Kinesis/Step Functionsとの対応表(本書で最も需要の高い対比)。Storage Queue(サービス公開編)との使い分け基準
- 実装:Transactional Outboxパターンで「DB更新とイベント発行」の整合を取り、通知処理をService Bus経由に移行。冪等コンシューマの実装
- 落とし穴:少なくとも1回配信と重複処理、順序保証が必要な場面(セッション)、DLQ監視の運用

# Phase 3: 品質と体験 — レジリエンス・リアルタイム・リリース・可観測性

## 第9章 レジリエンス設計 — 分散システムの失敗を飼いならす

- **設計課題**:依存先(DB・Redis・外部API)の一時障害でサービス全体を落とさない
- **選択肢**:リトライ/タイムアウト/サーキットブレーカー/バルクヘッド/フォールバックの適用判断
- **Azureでの実現**:.NET Polly(標準Resilience API)、各Azure SDKの組み込みリトライとの重複を避ける設計、Azure Chaos Studio での障害注入テスト
- AWS対比:AWS SDKのリトライ設計・Fault Injection Serviceとの対応
- 実装:外部API連携にサーキットブレーカー導入、Chaos StudioでRedis障害を注入して挙動確認
- 落とし穴:リトライストーム(指数バックオフ+ジッタ)、タイムアウトの多層整合

## 第10章 リアルタイム機能 — Azure SignalR Service

- **設計課題**:タスクの更新を他メンバーの画面に即時反映する
- **選択肢**:ポーリング/SSE/WebSocketの判断基準、スケールアウト時の接続管理問題
- **Azureでの実現**:SignalR Service(マネージド接続基盤)、グループ=テナント/ボードの設計、Functionsバインディングとの連携
- AWS対比:API Gateway WebSocket API / AppSyncとの対応。「接続状態管理を自前DynamoDBでやるAWS vs マネージドで持つSignalR」の違い
- 実装:更新イベント(第8章のEvent Grid)→SignalR→クライアント反映
- 落とし穴:認可(テナント外のグループ購読防止)、再接続と取りこぼし補償

## 第11章 フィーチャーフラグとリリース戦略

- **設計課題**:大機能を安全に段階公開したい(社内→一部テナント→全体)
- **選択肢**:デプロイとリリースの分離、カナリア/リング/ダークローンチ
- **Azureでの実現**:Azure App Configuration(フィーチャーフラグ、ターゲティング=特定テナント/割合)、Container Appsのリビジョン分割トラフィックとの併用
- AWS対比:AppConfig(feature flags)/CodeDeployカナリアとの対応
- 実装:新ダッシュボードを「特定テナント10%→50%→100%」で段階公開
- 落とし穴:フラグの寿命管理(消し忘れ負債)、フラグ組合せ爆発のテスト

## 第12章 可観測性のアプリ実装 — 「調査できるアプリ」を作る

- **設計課題**:「あるテナントのある操作が遅い」を5分で特定できる状態にする(サービス公開編のインフラ監視の上に載せるアプリ側の作法)
- **選択肢**:ログに何を構造化して載せるか(tenant_id/相関ID/ユーザー文脈)、サンプリング戦略
- **Azureでの実現**:Application Insights + OpenTelemetry、分散トレースがService Bus/SignalRを跨いで繋がる実装、KQLでのテナント別分析、可用性テスト
- AWS対比:X-Ray+ADOT/CloudWatch EMFとの対応
- 実装:全ログへのtenant_id注入、非同期境界を跨ぐトレース、SLOダッシュボード
- 落とし穴:PII(個人情報)をログに漏らさない設計、Application Insightsのコスト暴発(サンプリング)

# Phase 4: ビジネスと完成

## 第13章 プラン制御と利用量メータリング

- **設計課題**:Free/Standard/Enterpriseの機能差・上限(ユーザー数、ストレージ、API呼出数)をシステムとして強制し、利用量を集計する
- **選択肢**:上限チェックの実装位置(APIM/アプリ/DB制約)、メータリングの集計方式(同期カウント vs イベント集計)
- **Azureでの実現**:第4章APIMポリシー+第8章イベント集計+第11章ターゲティングの総合演習として構成。月次集計バッチ(Durable Functions)
- AWS対比:API Gateway usage plans/課金メータリングのAWS定番構成との対応
- 実装:利用量イベント→集計→上限超過時の挙動(ソフトリミット警告→ハードリミット)
- 落とし穴:課金に直結する集計の冪等性・監査可能性、締め処理とタイムゾーン

## 第14章 課金連携と外部SaaS統合(概説+設計)

- **設計課題**:Stripe等の決済SaaSとの連携、Webhook受信の信頼性
- **Azureでの実現**:Webhook受信の定番構成(即時200+キュー投入)、シークレット管理、リプレイ攻撃対策。※決済実装の詳細は本書の範囲外とし、設計原則に絞る(社内DX編の教訓=範囲の線引きを明示)
- AWS対比:EventBridge Partner Event Sourcesという思想の違い
- 実装:Stripe Webhookのモック受信→Service Bus→サブスクリプション状態更新

## 第15章 総仕上げ — リファレンスアーキテクチャと設計判断の振り返り

- 完成した全体アーキテクチャ図(インフラ層=サービス公開編/アプリ層=本書の二層で描く)
- 全設計判断の一覧表(章・判断・採用案・棄却案・理由)— 実案件でそのまま使える意思決定記録のテンプレとして

# 補足:本書の設計判断(トレードオフ)

1. **サービス公開編とのすみ分け**:Cache for Redis / Service Bus / APIM は「リソース構築・接続・閉域化」をサービス公開編、「設計パターンと実装」を本書が担当。AI SearchとCosmos DBは設計とプロビジョニングが不可分なため本書主担当とし、サービス公開編は対応表+概説のみ。
2. **実装言語はC#/.NET主軸**:Azureとの親和性・情報量で最良。ただしAWS経験者はNode.js/Python比率が高い可能性があるため、パターンの要点はTypeScript併記+付録Dで補う。全編二言語併記は分量倍増のため棄却。
3. **マルチテナントSaaSを題材に固定**:シングルテナント読者には過剰な面もあるが、テナント分離を貫く題材の方が認可・キャッシュ・検索すべての章で「分離」という一貫した緊張感を持てる。第15章の縮退パターンでシングルテナント読者を救済。
4. **課金(第14章)は概説に絞る**:決済実装まで踏み込むと本のスコープが崩壊する。社内DX編で得た「インフラ/アプリの線引きを明示する」原則をここでも適用。
