# 環境状態台帳(state.md)

各章を書き終えた時点で「その章終了時点に存在するリソースと設定」を追記していく台帳です。次章の執筆・レビュー時は、まずこのファイルで前章までの到達状態を確認します。

- 各章のセクションには「作成したリソース / RBAC・ポリシー変更 / ネットワーク変更 / コスト影響」を表で記録します。
- ハンズオンの内容と食い違いが出ないよう、章のマージ前に必ず更新します(CLAUDE.md のルール)。

## 第2章 終了時点

第2章では、第1章で作った器の骨格に「ID と権限の層」を足しました。テナント(Entra ID)に管理者/開発者/閲覧者の 3 グループを作成し、dev/prod へ強さの違う RBAC を割り当てました。dev は緩く、prod は厳しく、という差を権限で表現しています。個人への直接割り当てを避け、グループ経由で付与する原則を確立しました。

### 決めた方針(以降の章の前提)

| 項目 | 決定内容 |
|---|---|
| 権限付与の原則 | 個人へ直接割り当てず、グループ経由で付与する |
| ロールの既定 | 既定は共同作成者。所有者は絞る |
| 本番特権の扱い | 本番の管理者権限は常時付与せず PIM で一時昇格(実運用時に導入。P2 前提) |
| 割り当てスコープ | 本書はまずサブスクリプション スコープで割り当てる |

### 作成したリソース

| リソース種別 | 名前 | スコープ | 備考 |
|---|---|---|---|
| セキュリティグループ | `grp-foundation-admins` | テナント(Entra ID) | 管理者。本番所有者は PIM で一時昇格する運用を想定 |
| セキュリティグループ | `grp-foundation-developers` | テナント(Entra ID) | 開発者。dev=共同作成者 / prod=閲覧者 |
| セキュリティグループ | `grp-foundation-viewers` | テナント(Entra ID) | 閲覧者。dev/prod とも閲覧者 |

### RBAC・ポリシー変更

| `grp-foundation-developers` | 閲覧者(Reader)を割り当て | サブスクリプション `sub-foundation-prod` |
| `grp-foundation-viewers` | 閲覧者(Reader)を割り当て | `sub-foundation-dev` / `sub-foundation-prod` |
| `grp-foundation-admins` | 本番所有者は常時付与せず PIM で一時昇格(本章では常時割り当てを実施せず) | `sub-foundation-prod` |

## 第3章 終了時点

第3章では、器と権限の層に「ガバナンス(ポリシー)の層」を足しました。管理グループ `mg-foundation` に共通ポリシー(必須タグ・許可リージョン)を、prod サブスクリプションに追加の厳しいポリシー(パブリック IP 作成の禁止)を割り当て、継承で配下へ効かせます。第2章の合言葉「dev は緩く、prod は厳しく」を、今度は権限でなく「作れるリソースの範囲」で表現しました。必須タグは監査で現状把握してから拒否へ引き上げる、段階導入の型を確立しています。

### 決めた方針(以降の章の前提)

| 項目 | 決定内容 |
|---|---|
| ポリシー導入の順序 | 監査(Audit)で現状把握 → 既存違反を修正 → 拒否(Deny)へ引き上げ |
| 割り当てスコープの原則 | 共通ルールは管理グループへ、環境差(prod 限定)は個別サブスクリプションへ |
| 例外の扱い | 除外(Exemption)で対象・理由・期限を添えて登録。恒久的な穴を作らない |
| 必須タグの強制 | 第1章で項目のみ決めた `env` / `owner` を、本章でポリシーとして強制 |

### 作成したリソース

| リソース種別 | 名前 | スコープ | 備考 |
|---|---|---|---|
| ポリシー割り当て | `require-tag-env-deny`(必須タグ env) | サブスクリプション `sub-foundation-prod` | 監査で導入後、拒否へ引き上げ。ハンズオンで実施 |

### RBAC・ポリシー変更

| 対象 | 変更内容 | スコープ |
|---|---|---|
| 必須タグ(`env` / `owner`) | 監査 → 拒否で割り当て(共通) | 管理グループ `mg-foundation` |
| 許可リージョンを東日本に限定 | 拒否で割り当て(共通) | 管理グループ `mg-foundation` |
| パブリック IP の作成を禁止 | 拒否で割り当て(prod 限定) | サブスクリプション `sub-foundation-prod` |

> ハンズオンでは必須タグ(env)の prod への割り当てを実施。管理グループへの共通ポリシーと prod のパブリック IP 禁止は、同じ要領で割り当てる設計として提示。

## 第4章 終了時点

第 4 章では、統制の層(第 1〜3 章)の上に「ネットワークの層」を足しました。将来の Hub-Spoke 化・オンプレミス接続を見越した CIDR 設計を先に固め、dev/prod へ VNet を 1 つずつ構築しています。各 VNet は app 用・data 用の 2 サブネットと Bastion 専用サブネットを持ち、NSG はサブネット単位で管理します。既定を拒否寄りにし、data サブネットは app サブネットからのみ受信を許可する設計です。dev では一時 VM と Azure Bastion で疎通を確認しました(確認後に削除)。prod は骨格の構築までとし、Bastion を使う疎通確認は行っていません(第 3 章のパブリック IP 禁止ポリシーのため)。

### 決めた方針(以降の章の前提)

| 項目 | 決定内容 |
|---|---|
| CIDR 設計 | `10.0.0.0/8` を用途別に分割。ハブ=`10.0.0.0/16`(将来)、dev=`10.10.0.0/16`、prod=`10.20.0.0/16`、予備=`10.30.0.0/16` 以降 |
| サブネット構成 | 各 VNet に app 用・data 用の 2 サブネット + `AzureBastionSubnet` |
| NSG の管理単位 | サブネットに寄せる(NIC 側 NSG は原則使わない) |
| NSG の既定方針 | 拒否寄り。data サブネットは app サブネットからのみ受信許可 |
| 本番の運用アクセス | Bastion(パブリック IP 必要)は prod のポリシーと衝突。踏み台・Private 接続は続巻で設計 |

### 作成したリソース

| リソース種別 | 名前 | スコープ | 備考 |
|---|---|---|---|
| 仮想ネットワーク | `vnet-foundation-dev` | `rg-foundation-dev` | `10.10.0.0/16`。東日本。タグ `env=dev` |
| サブネット | `snet-app-dev` / `snet-data-dev` / `AzureBastionSubnet` | `vnet-foundation-dev` | `10.10.1.0/24` / `10.10.2.0/24` / `10.10.255.0/26` |
| NSG | `nsg-app-dev` / `nsg-data-dev` | `rg-foundation-dev` | 各サブネットへ関連付け。data は app からのみ許可 |
| 仮想ネットワーク | `vnet-foundation-prod` | `rg-foundation-prod` | `10.20.0.0/16`。東日本。タグ `env=prod` |
| サブネット | `snet-app-prod` / `snet-data-prod` / `AzureBastionSubnet` | `vnet-foundation-prod` | `10.20.1.0/24` / `10.20.2.0/24` / `10.20.255.0/26` |
| NSG | `nsg-app-prod` / `nsg-data-prod` | `rg-foundation-prod` | 各サブネットへ関連付け |

> 疎通確認で使った一時 VM(`vm-nettest-dev`)・Azure Bastion(`bastion-foundation-dev`)・パブリック IP(`pip-bastion-dev`)は、確認後に削除済み(常設リソースではない)。
