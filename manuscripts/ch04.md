## 概要 ── VNet・サブネット・NSG

第 1 章で器の骨格(4 階層と ARM)を、第 2 章で ID と権限の層(Entra ID と RBAC)を、第 3 章でガバナンスの層(管理グループと Azure Policy)を重ねてきました。ここまでは「**誰が**」「**何を**」作ってよいかという、統制の話でした。この章で足すのは、そこに置くリソースが実際に通信するための道——**ネットワーク**の層です。使う中心の道具が **仮想ネットワーク(VNet)** です。

VNet は、Azure 上に引く自分専用のプライベートなネットワーク空間です。この中にサブネットを切り、仮想マシンやマネージドサービスを接続し、通信の許可・拒否を **NSG(Network Security Group、ネットワークセキュリティグループ)** で制御します。まだこの章ではアプリを 1 つも載せませんが、続巻で載せるアプリがすべて乗る「土台のネットワーク」を、ここで先に敷いておきます。

AWS 経験者であれば、VNet は **VPC** に読み替えれば大枠は掴めます。プライベートな IP アドレス空間を確保し、その中をサブネットで区切る——この発想は同じです。ただし、後の節で見るように、細部の作法はいくつも違います。まずは VNet を構成する主な要素を、用語として並べておきます。

| 要素 | 何を表すか | AWS での相当 |
|---|---|---|
| 仮想ネットワーク(VNet) | プライベートな IP アドレス空間 | VPC |
| サブネット(Subnet) | VNet を区切った小さなアドレス範囲 | サブネット |
| NSG | 通信を許可・拒否するルールの集合 | セキュリティグループ / NACL |
| ルートテーブル(UDR) | 通信の経路を上書きする定義 | ルートテーブル |
| VNet ピアリング | VNet 同士を直結する接続 | VPC ピアリング |
| NAT Gateway | 送信専用のインターネット出口 | NAT Gateway |

<p class="caption">図4-1 VNet の構成要素と AWS での相当</p>

このうち、この章で実際に手を動かすのは **VNet・サブネット・NSG** の 3 つです。ルートテーブル(UDR、User-Defined Route、ユーザー定義ルート)・ピアリング・NAT Gateway は、設計の見通しを立てるうえで概念を押さえますが、構築は続巻の Hub-Spoke 構成で本格的に扱います。

<div class="column">
<span class="column-title">コラム:Hub-Spoke とは</span>

**Hub-Spoke(ハブ&スポーク)** は、Azure でのネットワーク構成の定番の型です。共有機能——オンプレミス接続・共有ファイアウォール・DNS など——を集約した中心の VNet が**ハブ**です。そこへ、各環境やワークロードの VNet を**スポーク**としてピアリングでぶら下げます。共通基盤をハブへ一元化すれば、スポークはアプリを載せる場所に専念できます。

AWS でいえば、**Transit Gateway 構成**に相当します。ただし、AWS の Transit Gateway が専用の中継サービスなのに対し、Azure のハブの実体はただの VNet です(次節の AWS 対比で改めて触れます)。本書ではハブを作らず、続巻で構築します。この章で `10.0.0.0/16` をハブ用に空けておくのは、将来の重複を避けるためです。

</div>

### NSG はサブネットと NIC の両方に付く

NSG は、通信を **許可・拒否**するルールをまとめたものです。送信元・宛先・ポート・プロトコルを指定した規則を並べ、パケットがそこを通ってよいかを判定します。ここまでは AWS のセキュリティグループと同じ感覚で読めます。

決定的に違うのは、**付けられる場所**です。NSG は、サブネットと、個々の NIC(Network Interface Card、ネットワークインターフェイス)のどちらにも関連付けられます。両方に付けた場合、パケットは受信・送信の両方向で二重に評価されます。

```text
【受信の評価順】インターネット/他サブネット
        │
        ▼
   サブネットの NSG   ← まずここで判定
        │
        ▼
     NIC の NSG      ← 次にここで判定
        │
        ▼
   仮想マシン(VM)

  ※ 両方で「許可」されて初めて到達する
```

<p class="caption">図4-2 NSG はサブネットと NIC の両方に付き、二重で評価される</p>

この二層構造は便利ですが、後述するとおり「どちらに付けるか」を決めずに両方へ付けると、通信が通らないときの切り分けが難しくなります。本書では**管理単位をサブネットに寄せる**方針を採ります。理由は「設計のポイント」で述べます。

## AWSとの対応と相違 ── VPCと何が違うか

ネットワークまわりを AWS と対応づけます。大枠は VPC と同じでも、AWS の習慣のまま進めると足をすくわれる違いがいくつかあります。まず対応表で全体像を掴み、そのあと決定的な違いを 3 点掘り下げます。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| 仮想ネットワーク | VPC | 仮想ネットワーク(VNet) | プライベート空間を確保する点は同じ |
| 区画 | サブネット | サブネット | Azure は「パブリック/プライベート」の区別を持たない |
| ステートフルな制御 | セキュリティグループ | NSG | NSG はサブネットと NIC の両方に付く |
| ステートレスな制御 | NACL | (NSG が兼ねる) | Azure に NACL 相当の独立した層はない |
| 経路制御 | ルートテーブル | ルートテーブル(UDR) | 既定経路は自動。UDR は上書き用 |
| インターネット入口 | インターネットゲートウェイ | (明示アタッチ不要) | ゲートウェイを貼る操作がない |
| ハブ接続 | Transit Gateway | Hub-Spoke + VNet ピアリング | ハブを VNet として自前で持つ |

<p class="caption">図4-3 AWS のネットワーク群と Azure の対応</p>

### 決定的な違い1 ── パブリック/プライベートサブネットの区別がない

AWS でネットワークを設計するとき、まず頭にあったのは「**このサブネットはパブリックか、プライベートか**」という区別だったはずです。ルートテーブルの既定経路(`0.0.0.0/0`)をインターネットゲートウェイに向けたサブネットが「パブリック」、NAT Gateway に向けた(あるいはどこにも向けない)サブネットが「プライベート」でした。この区別は、AWS では設計の出発点でした。

Azure には、この区別が**ありません**。サブネットに「パブリック用/プライベート用」という属性はなく、どのサブネットも既定では同じ性質を持ちます。では何が通信の可否を決めるのかというと、**NSG のルール**と、リソースにパブリック IP を割り当てるかどうかです。「サブネットの種類」ではなく「そのリソースにグローバル IP が付いているか」「NSG が通すか」で公開範囲が決まります。

```text
【AWS】サブネットの種類で決まる
  パブリックサブネット ── IGW への既定経路あり ── 公開されうる
  プライベートサブネット ─ IGW への経路なし ──── 公開されない

【Azure】サブネットに種類はない。リソース単位で決まる
  同じサブネット内でも…
    VM-A(パブリック IP あり) ── 公開されうる
    VM-B(パブリック IP なし) ── 外からは到達しない
  → 公開範囲は「パブリック IP の有無 + NSG」で決まる
```

<p class="caption">図4-4 パブリック/プライベートサブネットの考え方の違い</p>

この違いは、第 3 章で prod に効かせた「**パブリック IP の作成を禁止**」ポリシーの意味を、いま一度際立たせます。Azure では公開範囲がリソース単位のパブリック IP に強く依存するため、「そもそもパブリック IP を作らせない」という統制が、AWS の「プライベートサブネットに置く」に近い効き方をします。サブネットの種類で守るのではなく、IP の付与を止めて守る——これが Azure 流です。

### 決定的な違い2 ── インターネットゲートウェイを貼る操作がない

AWS では、VPC を作っただけでは外に出られませんでした。インターネットゲートウェイを作成して VPC にアタッチし、ルートテーブルで `0.0.0.0/0` をそこへ向けて、はじめて通信の道が開きました。

Azure の VNet には、この「ゲートウェイを貼る」操作が**ありません**。VNet を作れば、内部のリソースは既定で外向き(送信)のインターネット通信ができます。NSG の既定ルールが送信を許可しているためです。逆に言えば、AWS のように「IGW を貼り忘れて外に出られない」という詰まり方はしませんが、「**気づいたら外に出られてしまう**」方向に既定が倒れています。この既定の向きの違いは、後述の NAT Gateway やセキュリティ設計に効いてきます。

<div class="column">
<span class="column-title">コラム:既定の送信経路と NAT Gateway</span>

VNet の既定では、送信インターネット通信に **Azure が用意する既定の出口**が使われます。ただしこの既定の出口は、送信元 IP が固定されない・将来的な廃止が案内されているなど、本番で頼るには不安があります。<!-- 要検証: 公式Doc確認 — 既定の送信アクセス(default outbound access)の廃止時期と推奨移行先 -->

本番で送信 IP を固定したい・出口を 1 か所に束ねたいときは、**NAT Gateway** をサブネットに関連付けます。役割は AWS の NAT Gateway と同じで、多数のリソースの送信を 1 つのグローバル IP に集約します。本書の土台では出口の固定までは踏み込みませんが、「送信は明示的に設計するもの」という感覚だけ持っておいてください。

</div>

### 決定的な違い3 ── NSG がセキュリティグループと NACL を兼ねる

AWS では、通信制御に 2 つの層がありました。**セキュリティグループ**(ステートフル、インスタンス単位)と **NACL**(ステートレス、サブネット単位)です。セキュリティグループは戻りの通信を自動で許可します。一方 NACL は、往復を別々に書く必要がありました。

Azure では、この 2 つの役割を **NSG** が兼ねます。NSG は**ステートフル**で、許可した通信の戻りは自動で通ります。この挙動はセキュリティグループと同じです。そのうえで、NSG はサブネットとリソース(NIC)のどちらにも付けられるため、AWS で NACL が担っていた「サブネット単位の制御」も同じ道具でこなせます。NACL のようなステートレスの独立した層は、Azure には基本的にありません。

つまり AWS 経験者は、「NSG = ステートフルなセキュリティグループが、サブネットにも付けられるようになったもの」と捉えるのが近道です。NACL のことは、いったん忘れてかまいません。

<div class="note">
<span class="note-title">予告:PaaS は既定でパブリック公開 — 続巻最重要の伏線</span>

VNet の中に置く仮想マシンは、パブリック IP を付けなければ外から到達しません。ところが Azure の **PaaS**(ストレージ・データベース・Key Vault など)は、既定で**インターネットに公開されたパブリックなエンドポイント**を持ちます。VNet の外にいるのです。

そのため Azure では、「PaaS をどうやって VNet の内側へ引き込み、パブリック公開をやめさせるか」が、AWS 以上に重要なテーマとなります。その主役が **Private Endpoint**(プライベートエンドポイント)です。AWS の VPC エンドポイントに相当しますが、Azure では**必須級**の重要度を持ちます。本書では土台のネットワークを敷くところまでを扱い、Private Endpoint の実装は続巻で本格的に学びます。この章の VNet は、そのための受け皿でもあります。

</div>

## 全体構成に組み込む ── ネットワーク層に vnet-dev/prod を足す

第 3 章までで、器(管理グループ+サブスクリプション+リソースグループ)・権限(RBAC)・ガバナンス(ポリシー)がそろいました。ただし、ここまでに**ネットワークは 1 つもありません**。この章では、dev と prod のそれぞれに VNet を 1 つずつ作り、続巻のアプリが乗る土台のネットワークを敷きます。

ネットワークで最初に決めるべきは、個々のリソースではなく **アドレス設計(CIDR 設計)** です。ここを行き当たりばったりで決めると、後から VNet 同士をピアリングしたい・オンプレミスとつなぎたいというときに、アドレスが重複して**つなげなくなります**。CIDR は後から変えるのが極めて難しいため、将来を見越して先に表で決めておきます。

### 将来を見越した CIDR 設計

本書では、将来の Hub-Spoke 化(続巻)とオンプレミス接続を見越し、大きな枠として `10.0.0.0/8` の中を用途ごとに分けます。ハブ用・dev 用・prod 用の範囲を重ならないように割り当てておくのが要点です。

| 用途 | 範囲 | 状態 | 備考 |
|---|---|---|---|
| ハブ(将来) | `10.0.0.0/16` | 未作成 | 続巻で Hub-Spoke の中心に置く |
| vnet-dev | `10.10.0.0/16` | 本章で作成 | 開発環境 |
| vnet-prod | `10.20.0.0/16` | 本章で作成 | 本番環境 |
| 予備 | `10.30.0.0/16` 以降 | 未使用 | 追加環境・追加リージョン用に空けておく |

<p class="caption">図4-5 土台全体の CIDR 設計。範囲を重ねないことが最優先</p>

| | `snet-data-dev` | `10.10.2.0/24` | データ層(続巻で使用) |
| | `AzureBastionSubnet` | `10.10.255.0/26` | Bastion 専用(疎通確認用) |

<p class="caption">図4-6 vnet-dev / vnet-prod のサブネット構成</p>

命名は、第 1 章で決めた CAF 準拠の規則(`{種別}-{ワークロード}-{環境}`、ワークロード名は `foundation`)に従います。VNet は `vnet-foundation-dev` / `vnet-foundation-prod` とします。サブネットと NSG は、用途が一目で分かるよう `snet-app-dev` / `nsg-app-dev` のように用途名を含めます。

### Before / After ── ネットワークの層が加わる

第 3 章を終えた時点では、器・権限・ポリシーはありますが、通信の道はまだありません。

```text
【Before】第3章の到達状態(ネットワークはまだ無い)
管理グループ: mg-foundation
├─ ◆ ポリシー: 必須タグ / 許可リージョン(継承)
│
├─ サブスクリプション: sub-foundation-dev
│  └─ リソースグループ: rg-foundation-dev
└─ サブスクリプション: sub-foundation-prod
   │  ◆ ポリシー: パブリック IP 作成の禁止(prod 限定)
   └─ リソースグループ: rg-foundation-prod
```

<p class="caption">図4-7 第3章の到達状態(Before)</p>

第 4 章を終えると、dev/prod のリソースグループに VNet が 1 つずつ入り、それぞれの中に app/data サブネットと NSG が並びます。

```text
【After】第4章の到達状態(ネットワーク層を追加)
管理グループ: mg-foundation
├─ サブスクリプション: sub-foundation-dev
│  └─ リソースグループ: rg-foundation-dev
│     └─ VNet: vnet-foundation-dev(10.10.0.0/16)
│        ├─ snet-app-dev(10.10.1.0/24)  + nsg-app-dev
│        ├─ snet-data-dev(10.10.2.0/24) + nsg-data-dev
│        └─ AzureBastionSubnet(10.10.255.0/26)
└─ サブスクリプション: sub-foundation-prod
   └─ リソースグループ: rg-foundation-prod
      └─ VNet: vnet-foundation-prod(10.20.0.0/16)
         ├─ snet-app-prod(10.20.1.0/24)  + nsg-app-prod
         ├─ snet-data-prod(10.20.2.0/24) + nsg-data-prod
         └─ AzureBastionSubnet(10.20.255.0/26)
```

<p class="caption">図4-8 第4章の到達状態(After)。dev/prod に VNet とサブネット、NSG が加わる</p>

これで、統制の層(第 1〜3 章)の上に、通信の土台が乗りました。まだアプリはありませんが、続巻で app サブネットに公開 SaaS のコンテナを、data サブネットの先にデータベースを置いていく——その受け皿が整います。ハブ用の範囲を空けてあるので、続巻で Hub-Spoke に発展させるときも、アドレスを引き直さずに済みます。

## 設計のポイント ── アドレス設計とNSGの管理単位

ネットワークの土台で事故が起きやすいのは、**アドレス設計**と **NSG の管理**の 2 か所です。どちらも「後から直しにくい」点が共通します。要点をチェックリストにまとめます。

| 分類 | 原則 | 理由 |
|---|---|---|
| アドレス | 範囲を**重複させない**。将来分も先に予約する | ピアリング・オンプレ接続時に重複すると繋げない |
| アドレス | サブネットを**切りすぎない** | Azure は各サブネットで先頭・末尾の数 IP を予約する |
| NSG | 管理単位を**サブネットに寄せる** | サブネットと NIC の二重管理は切り分けを難しくする |
| NSG | 既定は**拒否寄り**にし、必要な通信だけ開ける | 「気づいたら開いていた」を防ぐ |
| セキュリティ | prod ほどルールを厚くする | 本番の公開事故は影響が大きい |
| コスト | VNet・サブネット・NSG 自体は**無料** | 課金は Bastion・NAT Gateway など上物から始まる |

<p class="caption">図4-9 ネットワーク設計のチェックリスト</p>

### アドレスは重複させない、切りすぎない

最重要は、**アドレス範囲を重複させない**ことです。いま単独で使う VNet でも、いずれハブや別環境、オンプレミスとつなぐ日が来ます。そのとき範囲が 1 ビットでも重なると、ピアリングや VPN でつなげなくなります。前節で `10.0.0.0/8` を用途ごとに分け、ハブ用・予備まで先に予約したのは、この事故を未来にわたって防ぐためです。

もう 1 つの落とし穴が、**サブネットの切りすぎ**です。細かく区切れば管理しやすい気がしますが、Azure は各サブネットの先頭・末尾のいくつかの IP を内部用に予約します。<!-- 要検証: 公式Doc確認 — サブネットで予約される IP アドレス数(先頭・末尾)の正確な仕様 --> 小さなサブネットを大量に作ると、予約分だけで利用可能な IP が目減りします。「意味のある単位で、少なめに」区切るのが基本です。本書が app/data の 2 つに絞っているのも、この考え方に沿っています。

### NSG の管理単位はサブネットに寄せる

NSG はサブネットと NIC のどちらにも付けられますが、**両方で作り込むと管理が破綻します**。通信が通らないとき、サブネットと NIC のどちらの NSG が原因か、二重に調べる手間が生じるからです。

本書では、**管理単位をサブネットに寄せます**。「この app サブネットに入ってよい通信」「この data サブネットに入ってよい通信」をサブネットの NSG に集約し、NIC 側の NSG は原則使いません。サブネット単位で見れば、そのネットワークで何が許可されているかが 1 か所で読み取れます。個々のリソースごとに例外を足したくなったときだけ、NIC の NSG を検討します。

### 既定は拒否寄り、必要な通信だけ開ける

NSG には既定のルールがあり、**同じ VNet 内の通信は許可**、インターネットからの受信は拒否、という状態から始まります。この既定を土台に、本書では「必要な通信だけを明示的に開ける」方針を採ります。app/data の各サブネットに対し、次のように最小限の許可を足します。

| サブネット | 許可する受信 | 送信 | ねらい |
|---|---|---|---|
| `snet-app` | Bastion サブネットから 22/3389(疎通確認時のみ) | 既定(拒否寄りに調整) | 運用アクセスだけを通す |
| `snet-data` | `snet-app` から DB ポートのみ(例: 5432) | 既定 | アプリ層以外からデータ層へ入れない |
| `AzureBastionSubnet` | Bastion に必要な規定の通信 | Bastion に必要な規定の通信 | Bastion の要件に従う |

<p class="caption">図4-10 NSG のルール設計(最小許可の方針)</p>

ここで data サブネットの受信を「app サブネットからのみ」に絞っているのが要点です。続巻でデータベースを置いたとき、アプリ層を経由しない直接アクセスを、ネットワークの段階で塞げます。まだ中身は空でも、「データ層はアプリ層の後ろに隠す」という設計を、先にネットワークで表現しておきます。

<div class="note">
<span class="note-title">注意:Bastion サブネットの規定ルールは緩めない</span>

`AzureBastionSubnet` に付ける NSG は、Azure Bastion が求める受信・送信のルールを満たす必要があります。要件を満たさない NSG を付けると Bastion が動きません。この専用サブネットの NSG は、公式ドキュメントの規定に厳密に従い、独自の判断で緩めたり削ったりしないでください。<!-- 要検証: 公式Doc確認 — AzureBastionSubnet の NSG に必要な受信・送信ルールの一覧 -->

</div>

<div class="column">
<span class="column-title">コラム:NSG フローログは第5章の監視へ</span>

NSG は「どの通信を許可・拒否したか」を **フローログ**として記録できます。このログの送り先が、次章で作る Log Analytics ワークスペースです。本書では第 5 章で「新しく作るリソースの診断ログは、必ず 1 つのワークスペースへ集約する」という規約を立てます。VNet や NSG も、その対象に含めていきます。ネットワークを敷いた今は「ログを出せる状態にした」ところまでを押さえ、集約の設定は第 5 章でまとめて行います。

</div>

## 構築手順 ── VNetとNSGを作り疎通確認

ここからは手を動かします。目的は 3 つです。第 1 に dev の VNet とサブネットを作ること、第 2 に NSG を作ってサブネットへ関連付けること、第 3 に一時的な仮想マシンと Azure Bastion で**実際に疎通を確認**することです。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

作業対象は **dev サブスクリプション**です。第 2 章で開発者グループに dev の共同作成者を割り当てたので、dev では自由にリソースを作れます。疎通確認には Azure Bastion が必要で、Bastion はパブリック IP を使います。第 3 章のパブリック IP 禁止ポリシーは **prod 限定**のため、dev では問題になりません。prod 側は VNet・サブネット・NSG の構築までを同じ要領で行い、Bastion を使う疎通確認は dev だけで実施します。

以下のコマンド・出力は、`handson-logs/ch04/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch04 の検証ログ未取得。コマンド・出力・所要時間を実機で確認する -->

### 1. 対象サブスクリプションを選ぶ

まず操作対象を dev サブスクリプションに合わせます。

```bash
# dev サブスクリプションを対象にする
az account set --subscription "sub-foundation-dev"
```

### 2. VNet とサブネットを作る

第 1 章で作成済みの `rg-foundation-dev` の中に、VNet を作ります。作成と同時に、最初のサブネットとして `snet-app-dev` を切ります。タグは第 3 章の必須タグポリシーに合わせて `env` を付けます。

```bash
# VNet と app サブネットを同時に作成する
az network vnet create \
  --resource-group "rg-foundation-dev" \
  --name "vnet-foundation-dev" \
  --location "japaneast" \
  --address-prefixes "10.10.0.0/16" \
  --subnet-name "snet-app-dev" \
  --subnet-prefixes "10.10.1.0/24" \
  --tags env=dev owner="you@example.com"
```

<!-- 要検証: handson-logs/ch04 の検証ログ未取得。az network vnet create の引数名(--address-prefixes / --subnet-prefixes)を実機で確認する -->

続けて、data 用サブネットと Bastion 専用サブネットを追加します。

```bash
# data サブネットを追加する
az network vnet subnet create \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --name "snet-data-dev" \
  --address-prefixes "10.10.2.0/24"

# Bastion 専用サブネット(名前は固定)
az network vnet subnet create \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --name "AzureBastionSubnet" \
  --address-prefixes "10.10.255.0/26"
```

### 3. NSG を作り、サブネットへ関連付ける

app 用と data 用の NSG を作ります。NSG も `rg-foundation-dev` の中に置きます。

```bash
# app 用・data 用の NSG を作成する
az network nsg create \
  --resource-group "rg-foundation-dev" \
  --name "nsg-app-dev" \
  --location "japaneast" \
  --tags env=dev owner="you@example.com"

az network nsg create \
  --resource-group "rg-foundation-dev" \
  --name "nsg-data-dev" \
  --location "japaneast" \
  --tags env=dev owner="you@example.com"
```

data サブネットには、「app サブネットからの DB ポートのみ許可」というルールを 1 つ足します。ここではデータベースの例として PostgreSQL の `5432` を使います。

```bash
# data サブネット: app サブネットからの 5432 のみ許可
az network nsg rule create \
  --resource-group "rg-foundation-dev" \
  --nsg-name "nsg-data-dev" \
  --name "allow-app-to-db" \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes "10.10.1.0/24" \
  --destination-port-ranges "5432"
```

<!-- 要検証: handson-logs/ch04 の検証ログ未取得。az network nsg rule create の引数名と既定の拒否ルールとの優先順位を実機で確認する -->

作った NSG を、それぞれのサブネットへ関連付けます。

```bash
# app サブネットへ nsg-app-dev を関連付ける
az network vnet subnet update \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --name "snet-app-dev" \
  --network-security-group "nsg-app-dev"

# data サブネットへ nsg-data-dev を関連付ける
az network vnet subnet update \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --name "snet-data-dev" \
  --network-security-group "nsg-data-dev"
```

### 4. 一時的な仮想マシンと Bastion で疎通を確認する

ネットワークが期待どおりに通るかを、実際の通信で確かめます。app サブネットに**パブリック IP を付けない**仮想マシンを 1 台立て、Azure Bastion 経由でサインインします。Bastion は、パブリック IP を持たない VM へブラウザから安全に接続するためのマネージドサービスです。VM 自体の詳しい扱いは続巻に譲り、ここでは疎通確認のための最小構成にとどめます。

```text
【疎通確認の構成】
  管理者のブラウザ
      │ HTTPS
      ▼
  Azure Bastion(AzureBastionSubnet / パブリック IP あり)
      │ プライベート通信
      ▼
  一時 VM(snet-app-dev / パブリック IP なし)
      │ 5432 を試す
      ▼
  snet-data-dev(NSG で app からのみ許可)
```

<p class="caption">図4-11 疎通確認の構成。Bastion 経由で VM に入り、data 側への通信を試す</p>

パブリック IP を持たない VM を app サブネットに作成します。

```bash
# app サブネットにパブリック IP なしの一時 VM を作成する
az vm create \
  --resource-group "rg-foundation-dev" \
  --name "vm-nettest-dev" \
  --vnet-name "vnet-foundation-dev" \
  --subnet "snet-app-dev" \
  --public-ip-address "" \
  --image "Ubuntu2204" \
  --admin-username "azureuser" \
  --generate-ssh-keys \
  --tags env=dev owner="you@example.com"
```

<!-- 要検証: handson-logs/ch04 の検証ログ未取得。--public-ip-address "" でパブリック IP を無効化する指定と、--image の正式なエイリアスを実機で確認する -->

次に、`AzureBastionSubnet` へ Azure Bastion を作成します。Bastion にはパブリック IP が 1 つ必要です。

```bash
# Bastion 用のパブリック IP を作成する
az network public-ip create \
  --resource-group "rg-foundation-dev" \
  --name "pip-bastion-dev" \
  --sku Standard \
  --location "japaneast"

# Azure Bastion を作成する(数分かかる)
az network bastion create \
  --resource-group "rg-foundation-dev" \
  --name "bastion-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --public-ip-address "pip-bastion-dev" \
  --location "japaneast"
```

<!-- 要検証: handson-logs/ch04 の検証ログ未取得。Bastion 作成の所要時間と必要な SKU、公開 IP の SKU 要件(Standard)を実機で確認する -->

作成できたら、ポータルの VM 画面から「接続 → Bastion」を選び、ブラウザ経由でサインインします。VM に入れれば、「パブリック IP なしの VM へ Bastion 経由で到達できる」ことが確認できます。VM の中から data サブネット側への通信を試し、NSG のルール(app からのみ許可)が意図どおり効いているかを合わせて確認します。<!-- 要検証: handson-logs/ch04 の検証ログ未取得。Bastion 接続手順と data 側への疎通テストの具体的コマンド・結果を実機で確認する -->

<span class="note-title">注意:Bastion と VM は確認後に必ず消す</span>

Azure Bastion と仮想マシン、パブリック IP は、置いておくだけで課金が続きます(VNet・サブネット・NSG は無料ですが、これらは有料です)。<!-- 要検証: 公式Doc確認 — Azure Bastion / 仮想マシン / パブリック IP の課金体系(時間課金の有無) --> 疎通確認が済んだら、忘れずに削除します。第 5 章・付録 D で扱う予算アラートと合わせ、学習用の課金事故を防いでください。

```bash

az network bastion delete --resource-group "rg-foundation-dev" --name "bastion-foundation-dev"
az vm delete --resource-group "rg-foundation-dev" --name "vm-nettest-dev" --yes
az network public-ip delete --resource-group "rg-foundation-dev" --name "pip-bastion-dev"
```

### 5. prod 側も同じ要領で作る(疎通確認は除く)

prod サブスクリプションでも、手順 2・3 と同じ要領で `vnet-foundation-prod`(`10.20.0.0/16`)とサブネット、NSG を作成します。ただし prod では、第 3 章のパブリック IP 禁止ポリシーにより Bastion のパブリック IP が作れません。そのため、Bastion を使う疎通確認は dev だけで行い、prod は骨格の構築までとします。本番へ運用アクセスする方法(踏み台の置き方や Private な接続)は、続巻で改めて設計します。

以上で、dev/prod に VNet とサブネット、NSG を敷き、dev では Bastion 経由の疎通まで確認できました。パブリック IP を持たない VM へ到達でき、データ層はアプリ層の後ろに隠れる——土台のネットワークが、設計どおりに動くことを確かめられました。

**この章のまとめ**

- **ネットワーク**の層は **VNet** を中心に、サブネットで区切り、**NSG** で通信を制御します。大枠は AWS の VPC と同じですが、作法はいくつも違います。
- Azure には**パブリック/プライベートサブネットの区別がなく**、公開範囲は「パブリック IP の有無 + NSG」で決まります。インターネットゲートウェイを貼る操作もなく、**NSG はセキュリティグループと NACL を兼ね**、サブネットと NIC の両方に付きます。
- **PaaS は既定でパブリック公開**のため、それを VNet に引き込む **Private Endpoint** の重要度が AWS 以上に高い——これが続巻最重要の伏線です。
- 土台には、将来のピアリング・オンプレ接続を見越した **CIDR 設計**を先に固め、dev/prod へ `vnet-foundation-dev` / `vnet-foundation-prod` を敷きました。アドレスは重複させず、NSG の管理はサブネットに寄せ、既定は拒否寄りにするのが要点です。
- この VNet と NSG は、第 5 章で診断ログを Log Analytics へ集約し、第 6 章で Bicep 化して「コードから再現できる」状態へ育てていきます。
