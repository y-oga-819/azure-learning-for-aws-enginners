## 概要 ── テナントから続く4階層とARM

AWS では、環境づくりの出発点は「アカウント」でした。Azure にも出発点にあたる境界はありますが、その手前に **テナント(Tenant)** という、より大きな器が存在します。この一段の違いが、Azure の設計思想を象徴しています。AWS が IaaS(Infrastructure as a Service、仮想サーバー貸し)から育ったのに対し、Azure は企業の ID 管理基盤から育ちました。そのため Azure は「まず組織と ID があり、その配下にリソースがぶら下がる」という発想で全体が組み立てられています。

本章では、まずこの器の入れ子構造——**4階層**——と、すべての操作が通る共通基盤 **ARM(Azure Resource Manager)** を押さえます。ここを最初に理解しておくと、以降の章(ID・ガバナンス・ネットワーク・監視)がなぜその順序で積み上がるのかが腑に落ちます。

### 4階層:テナント/管理グループ/サブスクリプション/リソースグループ

Azure のリソースは、次の4つの階層に必ず属します。上から順に器が小さくなり、上位で決めた統制(ポリシーや権限)が下位へ**継承**されます。

```text
テナント(= Microsoft Entra ID / ルート管理グループ)
└─ 管理グループ
   ├─ サブスクリプション: dev
   │  └─ リソースグループ
   │     └─ VNet / Log Analytics / ...
   └─ サブスクリプション: prod
      └─ リソースグループ
         └─ VNet / Log Analytics / ...

   ↑ 上位で決めた統制(ポリシー・権限)が下位へ継承される
```

<p class="caption">図1-1 テナントから続く Azure の4階層(統制は上位から下位へ継承される)</p>

| 階層 | 役割 | AWS でいうと |
|---|---|---|
| テナント | 組織全体の ID 境界(Entra ID のディレクトリと 1 対 1) | 相当物なし(組織全体) |
| 管理グループ(Management Group) | サブスクリプションを束ね、ポリシー・権限を継承させる統制の器 | Organizations / OU(Organizational Unit、組織単位) |
| サブスクリプション | 課金とポリシーの境界。リソースの量的な上限もここ | アカウント |
| リソースグループ | 同じライフサイクルのリソースをまとめる論理コンテナ | 完全な対応物なし |

<p class="caption">図1-2 4階層それぞれの役割と AWS での対応</p>

それぞれの器を、外側(上位)から順に見ていきます。

- **テナント(Tenant)** — 組織全体を表す、いちばん外側の器です。Microsoft の ID 基盤である **Entra ID** のディレクトリと 1 対 1 で対応し、「誰がこの組織のユーザーか」を管理します。テナントの ID は Entra ID のディレクトリ ID と同じ値です。すべての管理グループとサブスクリプションは、最終的にテナントに 1 つだけ存在する**ルート管理グループ**へ畳み込まれます。
- **管理グループ(Management Group)** — 複数のサブスクリプションを束ね、ポリシーや権限をまとめて継承させるための統制の器です。「本番系」「開発系」のようにサブスクリプションをグループ分けし、上位で 1 度決めたルールを配下すべてに効かせます。AWS でいえば、Organizations の中でアカウントをまとめる **OU(組織単位)** に相当します。ルート管理グループの下に階層を作れますが、実務では「3〜4 階層まで、なるべくフラットに」が推奨です。<!-- 要検証: 公式Doc確認 — 管理グループの階層深さ・サブスクリプション数の上限 -->
- **サブスクリプション(Subscription)** — 課金とポリシーの境界であり、リソースの量的な上限(クォータ)が効く単位です。AWS のアカウントにいちばん近い器で、「ここまでが 1 つの請求・1 つの統制のまとまり」を表します。ただし ID の境界ではない点が AWS アカウントとの決定的な違いで、これは次節「AWSとの対応と相違」で詳しく扱います。
- **リソースグループ(Resource Group)** — 同じライフサイクル(まとめて作成・更新・削除する単位)のリソースを入れる、いちばん内側の論理コンテナです。すべてのリソースは、必ずどれか 1 つのリソースグループに属します。どこにも属さないリソースは存在しません。リソースグループ自体も、権限の割り当てやコスト集計のスコープになります。

本書では、この 4 階層のうち「管理グループ+サブスクリプション」で dev/prod を分ける方式を主軸に据えます(詳細は本章「全体構成に組み込む」で扱います)。

### ARM:すべての操作が通る一枚の関所

Azure でリソースを操作する方法は複数あります。ポータル(画面での操作)、Azure CLI(`az`)、Azure PowerShell、各言語の SDK(Software Development Kit)、そして REST API です。しかし、どの入り口を使っても、要求は必ず **ARM(Azure Resource Manager)** という単一の層を通ります。

```text
  ポータル / Azure CLI / PowerShell / SDK / REST API
                     │
                     ▼
     ARM(すべての要求が通る関所)  ── 認証・認可 ──
                     │
                     ▼
  リソースプロバイダー(Microsoft.Network / Microsoft.Compute / ...)
                     │
                     ▼
                実際のリソース
```

<p class="caption">図1-3 すべての操作は ARM を通ってリソースへ届く</p>

ARM は受け取った要求を**認証・認可**し、問題がなければ担当の**リソースプロバイダー**(`Microsoft.Network`、`Microsoft.Compute` など、リソース種別ごとの実体)へ転送します。すべての要求が同じ API を通るため、どのツールを使っても結果と権限の効き方が一貫します。「ポータルでできることは CLI でも PowerShell でもできる」という安心感は、ここから生まれています。

この「一枚の関所」という性質は、本書全体の伏線です。第 2 章の RBAC(Role-Based Access Control。誰が何をできるか)も、第 3 章の Azure Policy(何を許し・何を拒むか)も、ARM を通る要求に対して効きます。つまり ID とガバナンスは、ARM という共通の通り道の上に成立しています。この関係を最初に掴んでおくと、以降の章が「別々の機能」ではなく「同じ土台の上の別の顔」だと見えてきます。

<div class="column">
<span class="column-title">コラム:リージョン・可用性ゾーン・リージョンペア</span>

リソースを配置する物理的な単位が **リージョン(Region)** です。東日本(Japan East)や西日本(Japan West)がこれにあたります。多くのリージョンは内部に複数の **可用性ゾーン(Availability Zone)** を持ち、データセンター単位の障害に対する冗長性を提供します。東日本リージョンは可用性ゾーンを備えたリージョンです。<!-- 要検証: 公式Doc確認 — 東日本リージョンの可用性ゾーン数 -->

さらに Azure には **リージョンペア(Region Pair)** という概念があり、東日本は西日本とペアを組みます。一部のサービスがこのペアを使って地理冗長やディザスタリカバリ(DR)を実現します。ただし近年の新しいリージョンはペアを持たず、可用性ゾーンを冗長性の主役とする方向へ変わりつつあります。リージョンとゾーンの設計判断は本章「設計のポイント」で扱います。<!-- 要検証: 公式Doc確認 — リージョンペアの最新の位置づけ・非ペアリージョンの扱い -->

## AWSとの対応と相違 ── アカウントとサブスクリプションの距離感

AWS 経験者が Azure で最初に行う「読み替え」は、たいてい **アカウント = サブスクリプション** です。この対応づけは半分正しく、残り半分が落とし穴です。課金とポリシーの境界という点で、両者はよく似ています。しかし、その境界が「ID の境界」と一致するかどうかで、両クラウドは決定的に分かれます。まずは 4 階層に関わる用語の対応を整理します。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| アカウント体系 | アカウント | サブスクリプション | どちらも課金・ポリシー境界。Azure はさらに上位にテナント/管理グループがある |
| 組織管理 | Organizations / OU | 管理グループ | 階層で統制を継承する点は同じ |
| ID 基盤 | IAM(アカウント内) | Microsoft Entra ID(テナント) | ID がアカウント内ではなくテナントに存在する |
| 論理グルーピング | CloudFormation スタック+タグ | リソースグループ | AWS に完全な対応物はない |
| リージョン冗長 | 任意リージョンで自前設計 | リージョンペア+可用性ゾーン | ペアの相手があらかじめ決まっている |

<p class="caption">図1-4 4階層に関わる用語の AWS ⇔ Azure 対応</p>

### 似ている点 ── 課金とポリシーの境界、階層の継承

サブスクリプションは、AWS アカウントと同じく**課金の集計単位**であり、**ポリシーや権限を適用する境界**でもあります。リソースの量的な上限(クォータ)もサブスクリプション単位で効きます。この「お金と統制の入れ物」という感覚は、そのまま持ち込んで構いません。

管理グループも、AWS の Organizations / OU とよく似ています。上位で決めた統制が下位のサブスクリプションへ継承される仕組みは、OU に SCP(Service Control Policy)を適用する感覚に近く、AWS でマルチアカウント戦略を組んだ経験がそのまま活きます。

### 決定的に違う点 ── ID の位置とリソースグループ

**1つ目は ID の位置です。** AWS では、IAM ユーザーやロールは**アカウントの中**に閉じています。別アカウントのリソースを触るにはクロスアカウントの `AssumeRole` が要りました。Azure では、ID は個々のサブスクリプションの中ではなく、その上の**テナント(Entra ID)に一元的に存在**します。1 人のユーザーが、同じ ID のまま複数のサブスクリプションに権限を持てます。この差が、第 2 章「Entra ID と RBAC」で扱う認可モデルの出発点になります。

**2つ目はリソースグループです。** これには AWS の完全な対応物がありません。強いて言えば、**CloudFormation スタックのライフサイクル**(まとめて作成・削除できる)と、**タグによる論理グルーピング**の、ちょうど中間にあたります。決定的な違いは、リソースグループが**任意ではなく必須**である点です。AWS ではスタックを使わずにリソースを直接作れましたが、Azure ではすべてのリソースがいずれかのリソースグループに属さなければなりません。しかもリソースグループ自体が RBAC やコスト集計のスコープになります。

<div class="column">
<span class="column-title">コラム:なぜこの違いが生まれたのか(本書全体の伏線)</span>

AWS は **IaaS(仮想サーバー貸し)** から育ちました。まず計算資源があり、その上に ID や統制を後付けした歴史です。一方 Azure は、Microsoft の **エンタープライズ ID 管理(Active Directory)** の系譜から育ちました。まず「組織と ID」があり、その配下にリソースをぶら下げる発想です。

だから Azure では、最上位にあるのがリージョンやアカウントではなく **テナント(=組織の ID)** なのです。この「ID が主役」という文化は、本書のすべての章に通底します——キー認証を根絶するマネージド ID(第 2 章)、テナント全体に効くガバナンス(第 3 章)。「Azure は ID から考える」——この一言を頭の片隅に置いておくと、以降の設計判断がずっと読みやすくなります。

</div>

### 最初のアカウント作成 ── ルートユーザーとテナントの発行

AWS と Azure の違いがいちばんはっきり出るのは、メールアドレスで初めてサインインした瞬間です。

AWS の場合、メールアドレスで AWS アカウントを作ると、そのアドレスが**ルートユーザー**になります。作成直後はまだ何も動いておらず、ここから「どのリージョンに、どのサービスを作るか」を進めます。ID(IAM ユーザーやロール)も、このアカウントの中に作ります。アカウントと ID が一体で、境界は 1 つです。

Azure の場合、サインインする先はテナントです。アカウントではありません。個人が Microsoft アカウントで Azure に**サインアップ**すると、Azure は既定のディレクトリ(テナント)を自動で作り、本人をそのテナントの**グローバル管理者(Global Administrator)**にします。課金の器であるサブスクリプションは、このテナントに紐づけて用意されます。「まず ID の器ができ、その中で作業を始める」——この順序こそ、AWS との出発点の違いです。<!-- 要検証: 公式Doc確認 — Azure サインアップ時の既定ディレクトリ自動作成とグローバル管理者付与 -->

| 観点 | AWS | Azure |
|---|---|---|
| サインインする先 | アカウント | テナント(Entra ID) |
| 最初にできる管理者 | ルートユーザー(アカウント内) | テナントのグローバル管理者 |
| 初回サインアップで起きること | アカウントができるだけ | ID の器(テナント)が発行される |
| ID の置き場所 | アカウントの中 | テナント(サブスクリプションの外側) |

<p class="caption">図1-5 メールアドレスで初めてサインインしたときの違い</p>

企業で使う場合は、話が変わります。会社が Microsoft 365 や Entra ID を導入していれば、入社時に配られるメールアドレス(例:`you@contoso.com`)は、**会社の Entra ID テナントの中に作られた職場アカウント**です。最初からテナントに所属しているため、Azure にサインインしても新しいテナントは作られません。管理者がそのユーザーにサブスクリプションの RBAC を割り当てれば、すぐに作業へ入れます。

既存の個人 Microsoft アカウントを、あとから会社のテナントへ参加させることもできます。その場合は、会社のテナントに**ゲスト(外部ユーザー)として招待**し、RBAC を割り当てます(Entra ID の B2B(Business-to-Business)機能。詳しくは第 2 章)。<!-- 要検証: 公式Doc確認 — Microsoft アカウントを B2B ゲストとして招待する手順 -->

<div class="note">
<span class="note-title">注意:意図しない「野良テナント」に気をつける</span>

自社ドメインを **Entra ID にまだ登録していない**とします。この状態で、社員がそのドメインのメールアドレスを使い、Microsoft のクラウドサービスへ個人的にサインアップしたとします。すると、**管理者不在のテナント(unmanaged tenant、いわゆる野良テナント)**が自動で作られてしまいます。放置すると、そのドメインを正規のテナントで使おうとしたときに衝突します。防ぐには、自社ドメインを早めに Entra ID で検証・確保しておくこと、そして ID を会社が一元管理することです。すでにできてしまった野良テナントは、**管理者による引き取り(admin takeover)**で正規のテナントに取り込めます。<!-- 要検証: 公式Doc確認 — unmanaged テナントの発生条件と admin takeover の手順 -->

</div>

### リージョンペアという制約 ── DR設計での立ち位置

もう 1 つ、AWS にない考え方が **リージョンペア** です。AWS では、DR 設計のときに任意の 2 リージョンを自分で選び、レプリケーションも自分で組みました。Azure では、東日本と西日本のように**ペアの相手があらかじめ決められています**。地理冗長ストレージ(GRS)などの一部サービスは、このペア先へ自動的にデータを複製します。プラットフォーム側の計画メンテナンスも、ペアの片方ずつ順番に行われる配慮があります。

つまり Azure の DR 設計では、「相方が決まっている」ことを前提に考えます。ただし前節のコラムで触れたとおり、近年の新しいリージョンはペアを持たず、**可用性ゾーンを冗長性の主役**とする方向へ移りつつあります。日本国内で設計する場合は、東日本を主・西日本をペアとする従来型と、東日本のゾーン冗長で完結させる新しい型の、どちらを採るかを最初に意識しておくとよいでしょう。<!-- 要検証: 公式Doc確認 — リージョンペアの提供状況と新規リージョンの冗長方針の最新動向 -->

## 全体構成に組み込む ── dev/prod 分離の方式を決める

本書は、章を追うごとに 1 つの土台(ミニランディングゾーン)を育てていきます。第 1 章の役割は、その**器の骨格**を決めることです。まだネットワークや監視はありませんが、「dev と prod をどう分けるか」という最初の分岐だけは、ここで確定させます。ここを曖昧にすると、以降の章で作るリソースの置き場所が定まらないからです。

### Before / After ── 骨格ができるまで

サインアップ直後は、既定のサブスクリプションが 1 つあるだけの、ほぼ空の状態です。

```text
【Before】サインアップ直後
テナント
└─ (ルート管理グループ)
   └─ サブスクリプション: 既定 1 つ
      └─ リソースグループ: なし
```

<p class="caption">図1-6 サインアップ直後(Before)の構成</p>

第 1 章を終えると、統制の器である管理グループの下に dev/prod を並べ、それぞれに最初のリソースグループが載った状態になります。

```text
【After】第1章の到達状態
テナント
└─ 管理グループ: mg-foundation
   ├─ サブスクリプション: sub-foundation-dev
   │  └─ リソースグループ: rg-foundation-dev
   └─ サブスクリプション: sub-foundation-prod
      └─ リソースグループ: rg-foundation-prod
```

<p class="caption">図1-7 第1章を終えた時点(After)の構成</p>

この骨格の上に、第 2 章で RBAC(dev を緩く・prod を厳しく)、第 3 章でポリシー、第 4 章で仮想ネットワーク(VNet)、と要素を足していきます。

### 分離方式の設計判断 ── サブスクリプション分離を主軸に

dev/prod の分け方には、大きく 2 つの方式があります。本書は **サブスクリプション分離** を主軸に据えます。境界が最も強く、実務のランディングゾーンに近いためです。一方、学習用にサブスクリプションを複数作れない読者のために、**リソースグループ分離**への読み替えも用意します(全章の対応は付録 E)。

| 観点 | サブスクリプション分離(本書の主軸) | リソースグループ分離(読み替え) |
|---|---|---|
| 境界の強さ | 課金・ポリシー・クォータが完全に分離 | 同一サブスクリプション内での緩い分離 |
| コストの見え方 | サブスクリプション単位で明確 | タグやリソースグループで集計 |
| 権限の分離 | サブスクリプション スコープで RBAC | リソースグループ スコープで RBAC |
| 向いている読者 | 組織・チームでの利用 | 個人学習・小規模 |
| 採用する編 | 本書(主軸)・公開サービス編 | 社内 DX 編 |

<span class="note-title">注意:サブスクリプションを複数作れない場合</span>

無料アカウントや個人の従量課金では、サブスクリプションを自由に増やせないことがあります。その場合は本文の「dev/prod サブスクリプション」を「dev/prod リソースグループ」に読み替えてください。以降の各章も同じ読み替えで通せるよう、対応表を付録 E にまとめています。<!-- 要検証: 公式Doc確認 — 個人向けアカウントで作成可能なサブスクリプション種別・数の制限 -->

</div>

なお、続巻の「社内 DX 編」は、あえてリソースグループ分離を主軸に採用します。1 つのサブスクリプションの中で App Service を中心に組む構成では、そのほうが現実的だからです。本書で両方式の勘所を押さえておくと、続巻でも迷いません。

## 設計のポイント ── 命名規則とリージョンを最初に決める

第 1 章で決めるべきことの多くは、**後から変えにくい**という共通点を持ちます。リソース名は原則として作成後に変更できず、リージョンも移動できません。だからこそ、手を動かす前に「命名規則」と「リージョン」の 2 つだけは腰を据えて決めます。

### 命名規則 ── CAF準拠のパターンを最初に固定する

Azure の推奨は、CAF(Cloud Adoption Framework)が示す次のパターンです。

```text
{リソース種別}-{ワークロード}-{環境}-{リージョン}-{連番}
   例) rg-foundation-dev / vnet-foundation-dev-001
```

<p class="caption">図1-9 CAF 準拠の命名パターン</p>

先頭の**リソース種別**は、CAF が公開している略語表に従います(リソースグループは `rg`、仮想ネットワークは `vnet` など)。すべてを毎回フルに書く必要はありませんが、**種別・ワークロード・環境**の 3 つは最低限そろえると、名前を見ただけで「何の・どの環境のリソースか」が分かります。本書では `foundation`(土台)をワークロード名とし、環境は `dev` / `prod` を使います。具体的な略語やそのまま使えるシートは付録 C にまとめます。<!-- 要検証: 公式Doc確認 — CAF の命名パターンとリソース種別略語の最新版 -->

### リージョン選定 ── 主リージョンとDR方針を決める

リージョンは、次の観点で選びます。本書は主リージョンを**東日本(Japan East)** とします。

| 観点 | 見るポイント |
|---|---|
| データ所在 | 法令・社内規程で国内保持が必要か。国内なら東日本/西日本 |
| レイテンシ | 利用者に近いリージョンほど低遅延。国内利用なら東日本 |
| サービス提供状況 | 新しいサービス・SKU は特定リージョンで先行提供される場合がある |
| DR 方針 | リージョンペア(東日本↔西日本)を使うか、東日本のゾーン冗長で完結させるか |
| コスト | 同じサービスでもリージョンにより価格が異なることがある |

<p class="caption">図1-10 リージョン選定の観点</p>

### 最初に決めておくチェックリスト

以降の章をスムーズに進めるために、この章の段階で次を確定させておきます。セキュリティ・統制・コストの観点を最初から織り込むのが狙いです。

| 分類 | 決めること | なぜ今か |
|---|---|---|
| 統制 | 命名規則(CAF パターン)を固定する | リソース名は後から変更できない |
| 統制 | 必須タグの項目(例:`env` / `owner` / `costcenter`) | 第 3 章でポリシーとして強制する土台になる |
| 統制 | dev/prod の分離方式(サブスクリプション or リソースグループ) | 以降のリソースの置き場所が決まる |
| セキュリティ | ルート管理グループ直下にリソースを置かない | 統制の効かない場所にリソースが漏れるのを防ぐ |
| コスト | 主リージョンと DR 方針 | リージョンは後から移動できない |
| コスト | 学習用の予算枠と課金アラートの前提 | 消し忘れによる課金事故を防ぐ(第 5 章・付録 D) |

<p class="caption">図1-11 第1章で決めておくことのチェックリスト</p>

<div class="column">
<span class="column-title">コラム:タグは「今すぐ全部」でなくてよい</span>

必須タグは、この章で**項目だけ**決めておけば十分です。実際に「タグが無いリソースは作らせない」と強制するのは、Azure Policy を学ぶ第 3 章の役割です。ここでは `env`(環境)と `owner`(持ち主)くらいから始め、運用しながら増やすのが現実的です。最初から項目を盛りすぎると、かえって形骸化します。

</div>

## 構築手順 ── サブスクリプション確認と最初のリソースグループ

ここからは手を動かします。第 1 章のハンズオンは、大きく作り込むことより「土台の器を 1 つ、命名規則どおりに正しく作る」ことを目的とします。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

<div class="note">
<span class="note-title">注意:課金事故を防ぐ</span>

学習用のサブスクリプションでは、消し忘れが思わぬ課金につながります。本章で作るリソースグループ自体に料金はかかりませんが、この先の章で作る VNet などには費用が発生し得ます。予算アラートと消し忘れ対策は第 5 章と付録 D で扱います。まずは「作ったら消す」を習慣にしましょう。

</div>

以下のコマンド・出力は、`handson-logs/ch01/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch01 の検証ログ未取得。コマンド・出力を実機で確認する -->

### 1. サインインとサブスクリプションの確認

まず Azure にサインインし、いま自分がどのサブスクリプションを操作対象にしているかを確認します。

```bash
# サインイン(ブラウザが開く)
az login

# 現在の操作対象サブスクリプションを表示
az account show --output table

# 利用できるサブスクリプションの一覧
az account list --output table
```

複数のサブスクリプションがある場合は、操作対象を明示的に切り替えます。

```bash
az account set --subscription "sub-foundation-dev"
```

### 2. 命名規則シートを用意する

コマンドを打つ前に、付録 C の命名規則シートに沿って、この章で使う名前を書き出しておきます。手を動かしながら名前を考え始めると、規則が崩れます。

| 種別 | 環境 | 名前 |
|---|---|---|
| 管理グループ | 共通 | `mg-foundation` |
| リソースグループ | dev | `rg-foundation-dev` |
| リソースグループ | prod | `rg-foundation-prod` |

<p class="caption">図1-12 第1章で使う名前(命名規則シート)</p>

### 3. 最初のリソースグループを作成する

命名規則に沿って、dev 用のリソースグループを作成します。リージョンに東日本(`japaneast`)を指定し、タグは最小限に `env` と `owner` を付けます。

```bash
az group create \
  --name rg-foundation-dev \
  --location japaneast \
  --tags env=dev owner=learning
```

作成できたら、内容を確認します。

```bash
# 単一リソースグループの詳細
az group show --name rg-foundation-dev --output table

# リソースグループの一覧
az group list --output table
```

同じ要領で、prod 用のリソースグループ `rg-foundation-prod` も作成しておきます(タグは `env=prod`)。

### 4. 後片付け(任意)

学習の区切りで不要になったら、リソースグループごと削除できます。リソースグループを消すと、**その中のリソースもすべて消えます**。

<div class="note">
<span class="note-title">注意:削除は取り消せません</span>

`az group delete` は、リソースグループ内のすべてのリソースをまとめて削除します。取り消しはできません。名前をよく確認してから実行してください。

</div>

```bash
az group delete --name rg-foundation-dev --yes --no-wait
```

以上で、命名規則に従った最初の器が 1 つでき、Azure の 4 階層のいちばん内側「リソースグループ」を実際に触れました。次章では、この器に対して「誰が何をできるか」を Entra ID と RBAC で設計していきます。

**この章のまとめ**

- Azure はテナント/管理グループ/サブスクリプション/リソースグループの **4 階層**でできており、上位の統制が下位へ継承されます。
- すべての操作は **ARM** という一枚の関所を通り、RBAC やポリシーがここに効きます。
- AWS との最大の違いは、**ID がアカウント内ではなくテナントに存在する**ことと、**リソースグループが必須の器**であること。背景には「ID から育った Azure」という文化があります。
- 第 1 章では、**dev/prod の分離方式・命名規則・リージョン**を先に固定し、最初のリソースグループを 1 つ作りました。
