## 概要 ── ARMテンプレートと Bicep

第 1 章から第 5 章まで、器の骨格・ID と権限・ガバナンス・ネットワーク・監視とコストの層を、一段ずつ重ねてきました。土台はほぼ出来上がっています。ですが、ここまでのリソースは**すべて手作業**——ポータルの画面と Azure CLI(`az`)のコマンドで作ってきました。もし誤ってこの VNet を消してしまったら、あなたは記憶と手順書を頼りに、また同じ操作を最初からなぞることになります。この章で足すのは、その不安をなくす最後のピース——**コードから土台を再現できる状態**、すなわち **Infrastructure as Code(IaC、インフラをコードで管理する考え方)**です。

Azure で IaC の中心を担う道具が **Bicep** です。Bicep は、Azure のリソースを宣言的に記述するための専用言語です。「こういうリソースが、こういう構成で存在してほしい」という**あるべき姿**を書くと、その状態になるようデプロイしてくれます。第 1 章で「すべての操作が通る一枚の関所」として押さえた **ARM(Azure Resource Manager)** を思い出してください。Bicep で書いたコードも、最終的にはこの ARM を通ってリソースになります。Bicep は、ARM に渡すための設計図を、人間に優しい書き方で用意する道具だといえます。

### Bicep は ARM テンプレートの「読みやすい書き方」

ARM には、もともと **ARM テンプレート**という設計図の形式があります。これは JSON で書かれた、リソースの定義ファイルです。ただ、この JSON は機械には素直でも、人間には冗長で読みにくいものでした。波括弧とダブルクォートが深く入れ子になり、ちょっとした VNet 1 つでも画面を何度もスクロールする長さになります。

**Bicep** は、この ARM テンプレートを**人間が読み書きしやすくした専用言語**です。特定分野の記述に特化した言語、いわゆる DSL(Domain-Specific Language、ドメイン固有言語)の一種にあたります。Bicep で書いたファイルは、デプロイ時に等価な ARM テンプレート(JSON)へ**変換(トランスパイル)**されてから ARM へ渡されます。つまり Bicep と ARM テンプレートは、同じ設計図を「読みやすい書き方」と「機械向けの書き方」で表したもので、**中身は同じ**です。

```text
【Bicep から ARM への流れ】
  Bicep ファイル(.bicep / 人間が読み書きする)
      │ 変換(トランスパイル)
      ▼
  ARM テンプレート(.json / 機械向け)
      │
      ▼
  ARM(第1章の「関所」)── 認証・認可・ポリシー評価 ──
      │
      ▼
  実際のリソース(VNet・NSG など)
```

<p class="caption">図6-1 Bicep は ARM テンプレートへ変換されてから ARM を通る</p>

この関係を押さえると、以降の話が腑に落ちます。Bicep で作ったリソースにも、第 2 章の RBAC も第 3 章の Azure Policy も**そのまま効きます**。なぜなら、入り口が画面でも CLI でも Bicep でも、要求は必ず同じ ARM を通るからです。IaC は「統制をすり抜ける裏口」ではなく、あくまで**同じ関所を通る、もう 1 つの入り口**です。

### 最小の Bicep ── リソース・パラメータ・出力

Bicep の文法は、覚えるべき要素が驚くほど少なく始められます。まずは 3 つ——**リソース定義**・**パラメータ**・**出力**——だけ押さえます。次は、VNet を 1 つ定義するだけの、最小の Bicep です。

```bicep
// パラメータ:デプロイ時に外から渡す値
param location string = 'japaneast'
param env string = 'dev'

// リソース定義:あるべきリソースを宣言する
resource vnet 'Microsoft.Network/virtualNetworks@2023-09-01' = {
  name: 'vnet-foundation-${env}'
  location: location
  tags: {
    env: env
  }
  properties: {
    addressSpace: {
      addressPrefixes: [
        '10.10.0.0/16'
      ]
    }
  }
}

// 出力:デプロイ後に受け取りたい値
output vnetId string = vnet.id
```

<p class="caption">図6-2 最小の Bicep。パラメータ・リソース定義・出力の 3 要素</p>

<!-- 要検証: 公式Doc確認 — Microsoft.Network/virtualNetworks の最新の安定 API バージョン(@2023-09-01 の妥当性) -->

読み方はこうです。**`param`** は、デプロイのたびに外から差し替えたい値です(ここでは location と env)。`= 'japaneast'` のような既定値を付けておけば、指定しなければその値が使われます。**`resource`** が本体で、「`Microsoft.Network/virtualNetworks` という種別のリソースが、この名前・この構成で存在してほしい」と宣言します。`@2023-09-01` は**リソースプロバイダーの API バージョン**で、どの世代の仕様で作るかを固定します。**`output`** は、デプロイ後に受け取りたい値です。作成したリソースの ID などを、後続の処理へ渡すときに使います。

大切なのは、Bicep が**手順ではなく状態**を書く言語だということです。「まず VNet を作り、次にサブネットを足す」という命令の並びではなく、「最終的にこうあってほしい」という完成形を書きます。あとは Bicep と ARM が、現状との差分を見て必要な操作を組み立ててくれます。同じファイルを 2 回デプロイしても、2 個目の VNet ができたりはしません——すでにあるべき姿になっていれば、何も変わらないだけです。この性質を**冪等性(べきとうせい)**と呼び、IaC の安心感の核になります。

### なぜ「ステートファイル」が要らないのか

AWS で Terraform を使ったことがあるなら、`terraform.tfstate` という**状態ファイル**の存在を覚えているはずです。Terraform は「自分が何を作ったか」を、この外部ファイルに記録して管理します。ところが Bicep には、これに当たるファイルがありません。

理由は明快です。**状態を持っているのは ARM 自身**だからです。Azure では、リソースグループやサブスクリプションといった**スコープそのものが「いま何が存在するか」の唯一の真実**です。Bicep は、デプロイのたびに「あるべき姿(コード)」と「現にある姿(ARM が持つ実物)」を突き合わせ、差分だけを適用します。管理すべき状態ファイルが外部にないので、「状態ファイルが実物とずれる」「状態ファイルをうっかり消す・ロックする」といった、Terraform 特有の悩みが、そもそも発生しません。

<div class="column">
<span class="column-title">コラム:状態が「実物側」にある強み</span>

状態ファイルを持たない設計は、一見おおざっぱに見えて、実は堅牢です。真実は常に実物(ARM が管理するリソース)の側にあります。そのため、チームの誰かがポータルで手直ししても、次のデプロイでコードと実物の差分はそのまま見えます。状態ファイルを共有ストレージに置いてロックを取り合う、といった運用も不要です。もっとも、これは「手直しがコードへ自動で書き戻される」という意味ではありません。実物を正とするか、コードを正とするか——その運用ルールは人間が決めなければなりません。この話は次節の「ドリフト」で改めて触れます。

</div>

### この章で扱う範囲 ── 「型」だけを先に

最後に、この章の**射程**をはっきりさせます。本章のゴールは、Bicep の**型(かた)を体に入れること**だけです。具体的には、第 4 章で手作業のまま作った VNet を、Bicep へ書き直します。それを一度削除し、コードから再構築して、「**同じものが甦る**」感覚を得るところまでです。

本格運用は、この章では扱いません。具体的には、複数の `.bicep` ファイルを部品として組み合わせるモジュール化や、コードの変更を自動でデプロイするパイプライン統合(CI/CD)です。これらは、載せるアプリの形が決まってから設計するほうが自然なため、各続巻の終盤でまとめて扱います。本章は「型」、続巻は「運用」——この**二段構え**を意識してください。ここで無理に作り込まず、まず素朴な 1 ファイルで再現性を体験することが、遠回りに見えて近道です。

## AWSとの対応と相違 ── CloudFormation/CDK との距離

IaC まわりを AWS と対応づけます。AWS では、CloudFormation(JSON/YAML のテンプレート)と AWS CDK(Cloud Development Kit)がありました。CDK は、プログラミング言語でテンプレートを生成する道具です。その周辺では、Terraform も広く使われていました。Azure でも道具立ては似ていますが、「状態をどこが持つか」と「スタックという単位があるか」で、AWS 経験者の習慣とずれる点があります。まず対応表で全体像を掴みます。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| テンプレート(宣言) | CloudFormation(JSON/YAML) | ARM テンプレート(JSON) | どちらも宣言的な設計図。ARM テンプレートは冗長 |
| 人に優しい記述 | AWS CDK(各言語) | Bicep(専用 DSL) | CDK は汎用言語、Bicep は Azure 専用の軽量言語 |
| サードパーティ IaC | Terraform(aws プロバイダー) | Terraform(azurerm プロバイダー) | Azure でも使えるが、本シリーズは Bicep 主軸 |
| 状態管理 | CFn がスタックで管理 | ARM(スコープ)が管理 | Bicep に状態ファイルはない |
| まとめて削除する単位 | スタック | (デプロイ履歴はあるが同義ではない) | 「スタック削除で一括削除」の感覚は通じない |

<p class="caption">図6-3 IaC 系サービスの AWS と Azure の対応</p>

### Bicep は「CloudFormation の人に優しい書き方」に近い

いちばん近い対応は、**Bicep ≒ CloudFormation を人に優しくした DSL** という捉え方です。どちらも「あるべきリソースを宣言する」宣言的 IaC で、手続きを書くのではなく完成形を書きます。CloudFormation の JSON/YAML に対して、Bicep はさらに記述量を削り、変数・パラメータ・参照を簡潔に書けるようにした言語です。

AWS CDK と混同しないよう注意してください。CDK は TypeScript や Python といった**汎用プログラミング言語**でテンプレートを組み立てる道具でした。Bicep は汎用言語ではなく、**Azure リソースを書くことに特化した軽量な専用言語**です。ループや条件分岐といった最小限のプログラマブルな要素は持ちますが、あくまで「宣言を書きやすくするため」のものです。CDK のようにアプリのロジックと同じ言語で書く、という発想とは立ち位置が異なります。

### 決定的な違い1 ── 状態ファイルがない、ドリフトの捉え方が変わる

概要でも触れたとおり、最大の違いは**状態ファイルの有無**です。Terraform は状態ファイル(`tfstate`)を持ち、CloudFormation はスタックが状態を持ちます。Bicep は、状態を ARM(スコープの実物)に委ねるため、**管理する状態ファイルがありません**。

この違いは、**ドリフト(drift、コードと実物のずれ)**の扱いに表れます。CloudFormation では「スタックが記録した状態」と「実際のリソース」のずれをドリフト検出で調べました。Bicep では、比較の基準がコードそのものです。デプロイ前に現状との差分を確かめれば、それがそのままドリフトの確認になります。この「デプロイ前に差分を見る」手段が、次に述べる what-if です。

### 決定的な違い2 ── what-if で「変更前に差分を見る」

Bicep(および ARM テンプレート)には、**what-if デプロイ**という機能があります。これは、コードを実際に適用する**前に**、そのデプロイで何が新規作成され・変更され・削除されるかを、一覧で見せてくれる仕組みです。CloudFormation の**変更セット(Change Set)**に相当する、事故を防ぐための安全装置です。

```bash
# 実際に適用する前に、変更内容だけを確認する
az deployment group what-if \
  --resource-group "rg-foundation-dev" \
  --template-file "./vnet-foundation-dev.bicep"
```

<!-- 要検証: handson-logs/ch06 の検証ログ未取得。az deployment group what-if の引数名と出力形式(+/-/~ の記号の意味)を実機で確認する -->

本番に近い環境ほど、この what-if を**必ず先に実行する**習慣が効きます。「削除される」と表示されるはずのないリソースが削除対象に出ていたら、それは書き間違いのサインです。適用してから気づくのでは遅い操作を、事前に発見できます。

### 決定的な違い3 ── 「スタック削除で一括削除」の感覚は通じない

CloudFormation では、スタックという単位が「そのテンプレートで作ったリソース群」を束ね、**スタックを削除すれば中身も一括で消える**という運用に馴染んでいたはずです。Azure の Bicep デプロイには、標準ではこれと同じ「削除まで面倒を見るスタック」の感覚がありません。デプロイの履歴は残りますが、それは「いつ何を適用したか」の記録であって、リソースの所有者ではないのです。<!-- 要検証: 公式Doc確認 — デプロイスタック(deployment stacks)機能の GA/プレビュー状況と、CFn スタックに相当する一括削除の可否 -->

では Azure で「まとめて消す」をどう実現するか。答えは第 1 章から一貫しています——**リソースグループ**です。関連するリソースを 1 つのリソースグループにまとめておけば、そのリソースグループを削除することで中身をまとめて消せます。CloudFormation のスタックが担っていた「ライフサイクルをまとめる器」の役割を、Azure ではリソースグループが担う——この対応で捉えると、後の構築手順(削除して再構築する)も素直に読めます。

### なぜ本シリーズは Bicep を主軸にするか

Azure では Terraform(azurerm プロバイダー)も広く使われており、マルチクラウドを 1 つの言語で扱いたい現場では有力な選択肢です。それでも本シリーズが Bicep を主軸に据えるのは、次の理由からです。第 1 に、**Azure 純正**で追加の状態管理が要らず、学習の入り口として構成要素が少ないこと。第 2 に、新しいリソース種別が出た**当日から対応**しやすいこと。第 3 に、ポータルからのエクスポートや ARM テンプレートとの相互変換など、Azure の他の道具と地続きなことです。まず純正の Bicep で「型」を固め、必要に応じて Terraform へ広げる——この順序が、AWS 経験者にとっても遠回りになりません。

## 全体構成に組み込む ── VNet/RG をコード化する

第 5 章までで、器・権限・ガバナンス・ネットワーク・監視がそろいました。ですが、そのすべては手作業の産物です。土台は完成していても、「**この土台を、もう一度同じ手順で作れますか**」と問われると、心もとない。この章で足すのは、新しいリソースではありません。既にある VNet・リソースグループを**コードで表し直し**、「壊れても甦る」再現性を土台に組み込みます。

### コード化するのは VNet とリソースグループに絞る

本章で Bicep 化する対象は、第 4 章で作った **dev の VNet 一式**(VNet・サブネット・NSG)と、それを収める**リソースグループ**に絞ります。ID(第 2 章)やポリシー(第 3 章)、監視(第 5 章)まで一気にコード化しないのは、二段構えの方針どおり「型」の習得に集中するためです。ネットワークは構造がはっきりしていて、Bicep の練習素材として最適です。まずここで再現性を体験し、対象を広げるのは各要素の運用設計とあわせて続巻で行います。

### Before / After ── 手作業の土台に「再現性」が加わる

第 5 章を終えた時点では、リソースはすべて手作業で作られ、その作り方は手順書と記憶の中にしかありません。

```text
【Before】第5章の到達状態(すべて手作業。コードはまだ無い)
管理グループ: mg-foundation
├─ サブスクリプション: sub-foundation-dev
│  └─ rg-foundation-dev
│     └─ vnet-foundation-dev(10.10.0.0/16)+ NSG  ← 手作業で作成
└─ サブスクリプション: sub-foundation-prod
   └─ rg-foundation-prod
      └─ vnet-foundation-prod(10.20.0.0/16)+ NSG  ← 手作業で作成
```

<p class="caption">図6-4 第5章の到達状態(Before)。作り方は手順書と記憶の中にしかない</p>

第 6 章を終えると、dev の VNet 一式が **Bicep ファイル**として表現され、リポジトリで管理されます。実物は同じでも、その隣に「同じものを再現するための設計図」が並びます。この設計図があれば、消しても、別環境へ写しても、同じ構成が甦ります。

```text
【After】第6章の到達状態(VNet 一式を Bicep 化)
管理グループ: mg-foundation
├─ サブスクリプション: sub-foundation-dev
│  └─ rg-foundation-dev
│     └─ vnet-foundation-dev + NSG  ◀── vnet-foundation-dev.bicep(設計図)
│                                          az deployment group create で再現
└─ サブスクリプション: sub-foundation-prod
   └─ rg-foundation-prod             (同じ設計図を env=prod で流用可能)
      └─ vnet-foundation-prod + NSG
```

<p class="caption">図6-5 第6章の到達状態(After)。VNet 一式に「コードからの再現性」が加わる</p>

注目してほしいのは、After でも**新しいリソースは 1 つも増えていない**ことです。増えたのは、リポジトリ側の `.bicep` ファイルだけです。IaC は、土台に機能を足す章というより、**既にある土台を「再現可能な資産」に変える**章だといえます。そして dev で書いた設計図は、パラメータを `env=prod`・アドレス空間を `10.20.0.0/16` に差し替えれば、prod の VNet 生成にもほぼそのまま流用できます。手作業では「dev と prod で同じ構成を、二度手で作る」しかなかったものが、コードでは「1 つの設計図を、パラメータ違いで 2 回使う」に変わります。

## 設計のポイント ── ポータルで作りエクスポートして整形

IaC を土台へ入れるとき、判断を迷いやすい点が 3 つあります。**どうやって最初の Bicep を用意するか**、**適用前の安全確認**、そして**コードと実物のどちらを正とするか**です。要点をチェックリストにまとめます。

| 分類 | 原則 | 理由 |
|---|---|---|
| 作り方 | まず**ポータル/CLI で作り**、エクスポートして整形する | ゼロから書くより速く、正しい API バージョンが得られる |
| 安全確認 | 適用前に**必ず what-if** で差分を見る | 意図しない変更・削除を適用前に捕まえる |
| 冪等性 | 同じコードは**何度流しても同じ結果**になるように書く | 再実行が怖くなくなる。これが IaC の安心の核 |
| 正とする側 | **コードを正**とし、手直しはコードへ戻す | ドリフトを放置しない。実物だけの変更を残さない |
| 統制 | Bicep デプロイにも**ポリシーは効く**前提で書く | ARM を通る以上、第3章の拒否ポリシーに従う |
| コスト | IaC の道具立て**自体は無料** | 課金はデプロイされるリソースからのみ発生する |

### 現場の現実 ── ゼロから書かず「作って・出して・整える」

Bicep を白紙から書き起こすのは、慣れないうちは骨が折れます。正しいプロパティ名や API バージョンを、いちいち調べることになるからです。実務では、もっと横着だが確実な方法が好まれます。その手順は、**まずポータル(または CLI)で目的のリソースを作り、その定義をエクスポートし、Bicep へ整形する**という流れです。

```text
【現実的なワークフロー】
  ① ポータル/CLI で目的のリソースを作る(動く実物ができる)
      ▼ エクスポート(ARM テンプレート / JSON)
  ② ARM テンプレートを取り出す
      ▼ 変換(bicep decompile)
  ③ Bicep へ整形する(パラメータ化・命名整理・不要プロパティ削除)
```

<p class="caption">図6-7 ポータルで作り、エクスポートして Bicep へ整形する流れ</p>

この方法の利点は 2 つあります。第 1 に、実際に動いた構成が出発点なので、**正しい API バージョンとプロパティ**が最初から手に入ること。第 2 に、エクスポートされた生の JSON には環境固有の値や冗長な設定が混ざっているので、それを**削り・パラメータへ抜き出す**過程そのものが、Bicep の良い練習になることです。エクスポートしたままの JSON は「動くが読めない」状態です。ここから env やアドレス空間をパラメータへ切り出し、不要なプロパティを落として、はじめて**再利用できる設計図**になります。第 4 章の VNet がすでにあるので、この流れは次の構築手順でそのままなぞれます。

<div class="column">
<span class="column-title">コラム:エクスポートは「たたき台」であって完成品ではない</span>

ポータルの「テンプレートのエクスポート」で得られる JSON は、あくまで**たたき台**です。実運用に耐える Bicep へ仕上げるには、ハードコードされた値をパラメータへ、繰り返しをループへ、環境差を変数へ——という整形が要ります。逆に言えば、エクスポートしたものをそのまま「IaC 化できた」と満足しないことが大切です。整形して初めて、dev と prod で使い回せる、レビューに耐える設計図へと仕上がります。本章では VNet 1 つ分でこの整形を体験し、整形の勘所(何を残し、何をパラメータにするか)を掴みます。

</div>

### 統制はコードでもすり抜けられない

IaC を導入すると、つい「コードなら自由に作れる」と錯覚しがちですが、そうではありません。Bicep のデプロイも ARM を通るので、**第 3 章で設定した Azure Policy は等しく効きます**。たとえば prod に対する「パブリック IP の作成を禁止」ポリシーは、ポータルで作ろうと Bicep で作ろうと、同じように拒否します。必須タグ(`env` / `owner`)も同様で、タグを書き忘れた Bicep は、prod では拒否ポリシーに弾かれます。

これはむしろ IaC の強みです。**コードのレビュー段階(what-if)で、ポリシー違反を早期に発見できる**からです。手作業なら「作ろうとして初めて弾かれる」ところを、コードなら差分確認の時点で気づけます。統制と IaC は対立せず、同じ ARM の上で補い合います。Bicep を書くときも、第 3 章で決めた必須タグを**コードにも必ず含める**——この一点を忘れないでください。

### コストは「道具ではなくリソース」から

コスト面は単純です。Bicep も、ARM も、what-if も、**道具立て自体には課金がありません**。費用が発生するのは、あくまで**デプロイされた実物のリソース**からです。この章で作る VNet・サブネット・NSG は、第 4 章で確認したとおりネットワークの器自体は無料です。<!-- 要検証: 公式Doc確認 — VNet/サブネット/NSG の課金有無 --> ただし、次の構築手順では「削除して再構築」を行います。削除と再作成を繰り返すこと自体は無料でも、**間違って別の課金リソースを再作成しない**よう、what-if で差分を確認してから適用する習慣が、ここでもコスト事故を防ぎます。

## 構築手順 ── 削除して再構築し、再現性を体験

ここからは手を動かします。目的は 4 つです。Bicep のツールを準備する、既存の VNet を Bicep 化する、what-if で差分を確認してデプロイする、そして**一度削除してコードから再構築して再現性を体感する**——この順で進めます。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

作業対象は **dev サブスクリプション**です。第 2 章で開発者グループに dev の共同作成者を割り当てたので、dev では自由にリソースを作り直せます。prod は同じ設計図をパラメータ違いで流用できますが、削除・再構築の練習は影響の少ない dev で行います。

以下のコマンド・出力は、`handson-logs/ch06/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch06 の検証ログ未取得。コマンド・出力・所要時間を実機で確認する -->

### 1. Bicep のツールを準備する

Bicep は Azure CLI に統合されており、多くの場合は追加インストール不要で使えます。バージョンを確認し、必要なら導入・更新します。

```bash
# Bicep CLI のバージョンを確認する(未導入なら az bicep install)
az bicep version
```

<!-- 要検証: 公式Doc確認 — Azure CLI に Bicep が同梱される条件と最小バージョン -->

まず操作対象を dev サブスクリプションに合わせます。

```bash
# dev サブスクリプションを対象にする
az account set --subscription "sub-foundation-dev"
```

### 2. 既存の VNet を Bicep 化する

設計のポイントで示したとおり、ゼロから書かず「作って・出して・整える」で進めます。第 4 章で作った `vnet-foundation-dev` は既にあるので、そのリソースグループの定義をエクスポートし、Bicep へ変換します。

```bash
# rg-foundation-dev の現在の構成を ARM テンプレート(JSON)としてエクスポート
az group export \
  --resource-group "rg-foundation-dev" \
  > rg-foundation-dev.json

# エクスポートした JSON を Bicep へ変換(decompile)する
az bicep decompile --file rg-foundation-dev.json
```

<!-- 要検証: handson-logs/ch06 の検証ログ未取得。az group export / az bicep decompile の引数名と、生成される .bicep のファイル名を実機で確認する -->

<div class="note">
<span class="note-title">注意:エクスポート直後の Bicep はそのまま使わない</span>

`az bicep decompile` が生成する Bicep は、あくまで機械的な変換結果です。環境固有の ID や冗長なプロパティ、警告などが残ることもあり、そのままでは再利用に向きません。次のステップで、必要な部分だけを抜き出し、`env` やアドレス空間をパラメータへ整形します。エクスポートは**たたき台**、整形して初めて設計図になる——設計のポイントで述べたとおりです。

</div>

整形の結果として、次のような `vnet-foundation-dev.bicep` を用意します。第 4 章の構成(VNet・app/data/Bastion サブネット・NSG・data は app からの 5432 のみ許可)を、`env` とアドレス空間でパラメータ化したものです。

```bicep
// パラメータ:環境ごとに差し替える値
param location string = 'japaneast'
param env string = 'dev'
param vnetAddressPrefix string = '10.10.0.0/16'
param appSubnetPrefix string = '10.10.1.0/24'
param dataSubnetPrefix string = '10.10.2.0/24'
param bastionSubnetPrefix string = '10.10.255.0/26'
param owner string

// app サブネット用 NSG
resource nsgApp 'Microsoft.Network/networkSecurityGroups@2023-09-01' = {
  name: 'nsg-app-${env}'
  location: location
  tags: {
    env: env
    owner: owner
  }
}

// data サブネット用 NSG(app サブネットからの 5432 のみ許可)
resource nsgData 'Microsoft.Network/networkSecurityGroups@2023-09-01' = {
  name: 'nsg-data-${env}'
  location: location
  tags: {
    env: env
    owner: owner
  }
  properties: {
    securityRules: [
      {
        name: 'allow-app-to-db'
        properties: {
          priority: 100
          direction: 'Inbound'
          access: 'Allow'
          protocol: 'Tcp'
          sourceAddressPrefix: appSubnetPrefix
          destinationPortRange: '5432'
          sourcePortRange: '*'
          destinationAddressPrefix: '*'
        }
      }
    ]
  }
}

// VNet と 3 つのサブネット(app/data/Bastion)
resource vnet 'Microsoft.Network/virtualNetworks@2023-09-01' = {
  name: 'vnet-foundation-${env}'
  location: location
  tags: {
    env: env
    owner: owner
  }
  properties: {
    addressSpace: {
      addressPrefixes: [
        vnetAddressPrefix
      ]
    }
    subnets: [
      {
        name: 'snet-app-${env}'
        properties: {
          addressPrefix: appSubnetPrefix
          networkSecurityGroup: {
            id: nsgApp.id
          }
        }
      }
      {
        name: 'snet-data-${env}'
        properties: {
          addressPrefix: dataSubnetPrefix
          networkSecurityGroup: {
            id: nsgData.id
          }
        }
      }
      {
        name: 'AzureBastionSubnet'
        properties: {
          addressPrefix: bastionSubnetPrefix
        }
      }
    ]
  }
}

// 出力:再現できたことを確認するための値
output vnetId string = vnet.id
```

<p class="caption">図6-8 整形後の vnet-foundation-dev.bicep(第4章の構成をパラメータ化)</p>

第 4 章で `az network ...` の複数コマンドに分けて作った内容が、1 つの設計図にまとまりました。`env` を `prod`、各アドレス空間を `10.20.x` 系に差し替えれば、そのまま prod 用にも使えます。「dev と prod を二度手で作る」から「1 つの設計図をパラメータ違いで使う」への転換が、ここで形になります。

### 3. what-if で差分を確認してからデプロイする

いきなり適用せず、必ず what-if で差分を見ます。まだ VNet が存在する状態で流すと、「差分なし(あるべき姿と一致)」に近い結果が出るはずです。冪等性の確認も兼ねます。

```bash
# 適用前に差分を確認する(owner は必須タグに合わせて指定)
az deployment group what-if \
  --resource-group "rg-foundation-dev" \
  --template-file "./vnet-foundation-dev.bicep" \
  --parameters owner="you@example.com"
```

差分の内容に納得できたら、実際にデプロイします。

```bash
# Bicep をデプロイする
az deployment group create \
  --resource-group "rg-foundation-dev" \
  --template-file "./vnet-foundation-dev.bicep" \
  --parameters owner="you@example.com"
```

<!-- 要検証: handson-logs/ch06 の検証ログ未取得。az deployment group what-if / create の引数名・出力・所要時間を実機で確認する。必須パラメータ(owner)の渡し方も確認する -->

同じコマンドを 2 回流しても、2 つ目の VNet はできません。すでにあるべき姿になっていれば、Bicep は何も変えないからです。これが冪等性です。手作業では「もう一度実行すると重複して失敗するかも」と身構えたところを、IaC では**気軽に再実行できる**——この安心感を、まず素振りで確かめておきます。

### 4. 一度削除して、コードから再構築する

ここが本章の山場です。VNet を**一度削除**し、いま整えた Bicep だけを頼りに**再構築**して、「コードから同じものが甦る」ことを体感します。前節で述べたとおり、Azure で「まとめて消す」の単位はリソースグループなので、練習も VNet 単体の削除で行い、リソースグループ自体は残します。

<div class="note">
<span class="note-title">注意:削除は取り消せません。対象と環境を必ず確認する</span>

これから実在するリソースを削除します。削除は**取り消せません**。作業前に、対象が **dev**(`sub-foundation-dev` / `rg-foundation-dev`)であること、消すのが `vnet-foundation-dev` であることを、必ず確認してください。prod では行いません。また、第 4 章の疎通確認で使った仮想マシン・Azure Bastion は削除済みで、いま VNet を参照している常設リソースが無いことも前提です。もし何かが VNet を使っていると、削除は失敗するか、想定外の影響が出ます。

</div>

まず、現在の VNet を削除します。

```bash
# dev の VNet を削除する(取り消せない操作)
az network vnet delete \
  --resource-group "rg-foundation-dev" \
  --name "vnet-foundation-dev"
```

<!-- 要検証: handson-logs/ch06 の検証ログ未取得。VNet 削除時に NSG が残るか同時に消えるか、依存関係で削除順序の指定が要るかを実機で確認する -->

削除できたら、VNet が無くなったことを確認します。

```bash
# VNet が消えたことを確認する(一覧に出てこないはず)
az network vnet list \
  --resource-group "rg-foundation-dev" \
  --query "[].name" -o tsv
```

ここで、先ほどの Bicep を what-if にかけると、今度は「**VNet 一式を新規作成する**」という差分が出るはずです。適用前に、この差分が期待どおり(新規作成であり、削除が含まれない)であることを確かめます。

```bash
# 再構築の差分を確認する(新規作成が並ぶはず)
az deployment group what-if \
  --resource-group "rg-foundation-dev" \
  --template-file "./vnet-foundation-dev.bicep" \
  --parameters owner="you@example.com"
```

差分を確認したら、コードからデプロイして再構築します。

```bash
# コードから VNet 一式を再構築する
az deployment group create \
  --resource-group "rg-foundation-dev" \
  --template-file "./vnet-foundation-dev.bicep" \
  --parameters owner="you@example.com"
```

最後に、第 4 章と同じ構成が甦ったことを確認します。VNet・3 つのサブネット・NSG が、削除前と同じ名前とアドレスで並んでいれば成功です。

```bash
# サブネットが 3 つ(app/data/Bastion)再現されたか確認する
az network vnet subnet list \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --query "[].{name:name, prefix:addressPrefix}" -o table
```

<!-- 要検証: handson-logs/ch06 の検証ログ未取得。再構築後のサブネット一覧・NSG 関連付けの確認コマンドと出力を実機で確認する -->

手作業では、削除したリソースを元どおりに戻すのは「手順書を最初からなぞり直す」作業でした。IaC では、**設計図を 1 回デプロイするだけ**です。壊すのが怖くなくなり、壊して直す実験を気軽に試せる——これこそ再現性のいちばんの果実です。第 4 章で手をかけて組んだネットワークが、コード 1 ファイルから甦る。この体験こそ、この章で持ち帰ってほしいものです。

以上で、既存の VNet を Bicep 化し、what-if で差分を確認しながらデプロイし、一度削除してコードから再構築するところまでを体験しました。土台の一部が「再現可能な資産」に変わり、続巻でモジュール化やパイプライン統合へ進む準備が整いました。この章で残るのは、リポジトリ側の `vnet-foundation-dev.bicep` という設計図だけ——実物の構成は、第 5 章の到達状態から変わっていません。

**この章のまとめ**

- **Bicep** は、ARM テンプレート(JSON)を人に優しくした専用言語です。Bicep のコードもデプロイ時に ARM テンプレートへ変換され、第 1 章の「関所」である **ARM(Azure Resource Manager)** を通ります。だから、RBAC やポリシーもそのまま効きます。
- 文法は**リソース定義・パラメータ・出力**の 3 要素から始められます。Bicep は手順ではなく**あるべき状態**を書く言語で、同じコードを何度流しても結果が変わらない**冪等性**を持ちます。
- Bicep に**状態ファイルはありません**。状態を持つのは ARM(スコープの実物)だからです。CloudFormation のスタックによる一括削除に当たる操作は、Azure では**リソースグループの削除**が担います。適用前の差分確認には **what-if**(CFn の変更セット相当)を使います。
- AWS の CloudFormation / CDK / Terraform は、Azure では **ARM テンプレート / Bicep / Terraform(azurerm)**に対応します。本シリーズは、純正・状態管理不要・当日対応という理由で **Bicep を主軸**とします。
- 現場の型は「**ポータル/CLI で作り → エクスポート → Bicep へ整形**」です。本章では第 4 章の VNet 一式を `vnet-foundation-dev.bicep` に整形し、**一度削除してコードから再構築**して再現性を体験しました。モジュール化・パイプライン統合は続巻で扱う二段構えとし、第 7 章では、ここまで積み上げた土台の到達状態を総点検します。
