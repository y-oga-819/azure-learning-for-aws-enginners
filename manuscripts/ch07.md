## 概要 ── 完成した全体構成とCAFとの対応

第 1 章から第 6 章まで、器の骨格・ID と権限・ガバナンス・ネットワーク・監視とコスト・再現性(IaC)を、一段ずつ積み重ねてきました。この章では、新しいリソースを 1 つも足しません。代わりに、ここまで組み上げた土台を**1 枚の全体像として見渡し**、抜けや歪みがないかを点検します。そして、この土台が実務のランディングゾーンの縮小版として筋が通っていることを確かめ、続巻の 3 冊へどう分岐していくかを見晴らします。総仕上げの章です。

まず、6 章分の積み上げを 1 枚に重ねてみます。下から順に、上位の層が下位の層に支えられている——この積層構造が、本書で作った**ミニランディングゾーン**の姿です。

```text
【完成したミニランディングゾーンの6層】

  ⑥ 再現性(IaC)     Bicep で VNet を設計図化(第6章)
  ─────────────────────────────────────────────
  ⑤ 監視とコスト     Log Analytics 1 本に集約 + 予算アラート(第5章)
  ─────────────────────────────────────────────
  ④ ネットワーク     dev/prod の VNet・サブネット・NSG(第4章)
  ─────────────────────────────────────────────
  ③ ガバナンス       Azure Policy(必須タグ・許可リージョン・IP 禁止)(第3章)
  ─────────────────────────────────────────────
  ② ID と権限        Entra ID グループ + RBAC(第2章)
  ─────────────────────────────────────────────
  ① 器の骨格         管理グループ + dev/prod サブスクリプション + RG(第1章)
```

<p class="caption">図7-1 6 章で積み上げたミニランディングゾーンの層構造</p>

大切なのは、この 6 層が**別々の機能の寄せ集めではない**ことです。第 1 章で「すべての操作が通る一枚の関所」として押さえた ARM(Azure Resource Manager)の上に、ID・ガバナンス・ネットワーク・監視が乗り、それらを IaC が丸ごと再現できる——という一本の背骨が通っています。どの層も、下の層があって初めて意味を持ちます。権限(②)は器(①)というスコープがあって効き、ポリシー(③)は継承の階層(①)があって配下へ届き、監視(⑤)は集約先というただ 1 つの置き場所を統制の効いた場所(①〜③)に持ちます。

### CAF ── Microsoft が示す「クラウド導入の地図」

本書が作ってきたものが実務で何と呼ばれるかを、ここで正式に接続しておきます。Microsoft は、クラウド導入の進め方を体系化した **CAF(Cloud Adoption Framework、クラウド導入フレームワーク)** を公開しています。CAF は「戦略 → 計画 → 準備 → 導入 → 統制 → 管理」といった段階で、組織がクラウドをどう採り入れるかを示す地図です。第 1 章で命名規則の根拠として触れた CAF は、実はこの大きな枠組みの一部でした。<!-- 要検証: 公式Doc確認 — CAF の方法論(methodology)の段階名と最新の構成 -->

その CAF の中で、統制の効いた土台を最初に敷くための設計思想が **ランディングゾーン(Landing Zone)** です。ランディングゾーンは、アプリを載せる**前に**、ID・ガバナンス・ネットワーク・監視といった共通の土台を整えておく、という考え方です。CAF は、このランディングゾーンを設計するときに検討すべき観点を**設計領域(design area)**として整理しています。本書の各章は、この設計領域をなぞって進んできました。

| CAF の設計領域(観点) | 本書で対応する層 | 扱った章 |
|---|---|---|
| Azure への課金と組織(サブスクリプション設計) | 器の骨格(管理グループ + サブスクリプション) | 第1章 |
| ID とアクセス管理 | Entra ID グループ + RBAC | 第2章 |
| ガバナンス(統制の自動化) | Azure Policy | 第3章 |
| ネットワークトポロジと接続 | VNet・サブネット・NSG | 第4章 |
| 管理と監視 | Log Analytics + 予算 | 第5章 |
| プラットフォームの自動化(IaC) | Bicep | 第6章 |
| セキュリティ | 全章を貫く横断テーマ | 第2〜5章 |

つまり、本書でばらばらに作ってきたように見えるものは、CAF が「ランディングゾーンを敷くならこの観点を押さえなさい」と示す**チェック項目**を、1 つずつ手で埋めてきた作業でした。この章で全体を見渡すのは、その地図の上で「行くべき場所をひととおり訪れたか」を確かめる作業にあたります。

<div class="column">
<span class="column-title">コラム:なぜ「ミニ」ランディングゾーンなのか</span>

本書が作ったものを、あえて「ミニ」と呼ぶのには理由があります。実務のランディングゾーンでは、共有サービス専用の管理(プラットフォーム)サブスクリプションを分けたり、管理グループを「プラットフォーム」「アプリ着地帯」といった役割別に何層も切ったり、ポリシーを数十本まとめて敷いたりします。本書では、その骨組みだけを最小構成に圧縮しました。管理サブスクリプションの役割は prod が兼ね(第 5 章)、管理グループは 1 つに絞っています。小さく作った分、**中で何が起きているかがありありと分かる**——これがミニで学ぶ最大の利点です。規模を上げる分岐点は、各章で「続巻で切り出す」と書き添えてきました。

## AWSとの対応と相違 ── Control Tower/LZA との対応

本書で「手で積み上げた土台」を、AWS のマネージドな統制サービスと対応づけます。AWS では、統制の効いたマルチアカウント環境を一気に敷く手段として **Control Tower** と **Landing Zone Accelerator(LZA、ランディングゾーン アクセラレータ)** がありました。Azure にも同じ役割の道具立てがあります。まず対応表で全体像を掴みます。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| 統制つき環境の初期構築 | Control Tower | Azure ランディングゾーン(ポータルのアクセラレータ) | どちらも土台一式を短時間で敷く |
| 大規模統制のテンプレート | Landing Zone Accelerator(LZA) | Azure Landing Zone Accelerator(ALZ) | Bicep/Terraform でパラメータ化された参照実装 |
| ガードレール(統制ルール) | Control Tower のガードレール | Azure Policy(組み込み + イニシアティブ) | Azure は Policy が単体で完結 |
| アカウント/サブスクリプション払い出し | Account Factory | サブスクリプション ベンディング | 新しい入れ物を統制つきで自動発行 |
| 統制の継承単位 | OU(組織単位) | 管理グループ | 階層で継承する点は同じ |

<p class="caption">図7-3 大規模統制サービスの AWS ⇔ Azure 対応</p>

<!-- 要検証: 公式Doc確認 — Azure ランディングゾーン アクセラレータ(ALZ)/サブスクリプション ベンディングの正式名称・提供形態の最新情報 -->

### 似ている点 ── 「土台を先に、ガードレール付きで」

発想は AWS と Azure でよく似ています。どちらも、アプリを載せる前に**統制の効いた土台を先に用意**し、その上で各チームに環境を払い出します。Control Tower のガードレールが「やってはいけないこと」を禁じたのと同じ役割を、Azure では第 3 章で扱った **Azure Policy** が担います。OU に統制を継承させる感覚は、管理グループにそのまま持ち込めます。AWS でマルチアカウント戦略を組んだ経験は、Azure のランディングゾーン設計でほぼそのまま活きます。

第 3 章でも触れたとおり、実際の大企業では、管理グループの階層・多数のポリシー・共有ネットワーク・監視までを一括で敷設する **Azure Landing Zone Accelerator(ALZ)** が使われます。これは AWS の LZA に相当する、統制一式の参照実装です。ALZ は Bicep や Terraform で公開されており、パラメータを与えるだけで、企業規模のランディングゾーンをまとめて展開できます。

### 決定的に違う点1 ── 「押しボタン」か「手で積む」か

最大の違いは、**土台をどう手に入れるか**の思想です。AWS の Control Tower は、コンソールから有効化すると、複数アカウント・OU・ガードレール・ログアーカイブまでを**マネージドサービスとして一括セットアップ**してくれる「製品」でした。裏側で何が起きているかは、ある程度隠されています。

Azure のランディングゾーンは、ポータルのアクセラレータで一括展開もできますが、その実体は **ALZ という参照実装(コード)**です。「隠された製品」ではなく「公開された設計図」に近く、中身を読んで、必要なだけ改変して使います。本書があえて Control Tower 相当のボタンを押さず、第 1 章から手で積み上げてきたのは、この「設計図の中身」を 1 行ずつ理解するためでした。

| 観点 | AWS Control Tower | 本書のアプローチ(手で積む) |
|---|---|---|
| 入手方法 | コンソールで有効化(マネージド) | ポータル/CLI で 1 つずつ構築 |
| 中身の見え方 | ある程度隠蔽される | すべて自分で作るので完全に見える |
| 学習効果 | 早く整うが仕組みは掴みにくい | 遅いが各層の意味が腑に落ちる |
| 実務での使い分け | 本番の初期構築を高速化 | 縮小版で原理を学ぶ → 本番は ALZ へ |

<p class="caption">図7-4 Control Tower と本書の「手で積む」アプローチの違い</p>

本書のミニランディングゾーンと、ALZ が展開する企業版とは、**対立するものではありません**。ミニで原理を理解した人が、本番では ALZ の設計図を読み解いて使う——この順序こそ、遠回りに見えて確実です。ボタンで一括生成された土台を「なぜこうなっているのか分からないまま」運用するより、まず小さく手で作った経験のあるほうが、ALZ のパラメータ 1 つの意味も判断できます。

### 決定的に違う点2 ── ガードレールは Azure Policy 単体で完結する

Control Tower のガードレールは、内部で AWS Config ルールや SCP(Service Control Policy、サービスコントロールポリシー)を組み合わせて実現されていました。Azure の場合、第 3 章で見たように **Azure Policy 単体が監査・拒否・修復までを担います**。SCP のような「上限を絞る」層と、Config のような「準拠を評価する」層が、Azure では 1 つの Policy に統合されています。だから本書では、ガードレールを敷くのに複数のサービスを組み合わせる必要がなく、管理グループへ Policy を割り当てれば済みました。この単純さは、ミニで学ぶうえでも大きな利点です。

## 設計のポイント ── Well-Architected で振り返る

到達状態が「存在するか」を確認したら、次は「**良い状態か**」を評価します。ここで使うのが、Microsoft の **Well-Architected Framework(ワークロードの品質を評価するフレームワーク)**です。CAF が「土台を敷くまでの地図」だとすれば、Well-Architected Framework は「敷いた土台の質を測る物差し」にあたります。5 つの柱で、セキュリティ・統制・コストを含む観点から土台を採点します。<!-- 要検証: 公式Doc確認 — Well-Architected Framework の 5 本柱の正式名称と最新の定義 -->

| 柱 | 本書の土台での実装 | 続巻・実務へ送る宿題 |
|---|---|---|
| セキュリティ | Entra ID + RBAC、Policy による拒否、NSG の既定拒否寄り | PIM の有効化、Private Endpoint、マネージド ID の実装 |
| 信頼性 | 東日本リージョンを主、リージョンペア/ゾーンを認識 | DR 設計、可用性ゾーン配置、バックアップ |
| コスト最適化 | 予算アラート、ログ保持は短めから、器は無料と把握 | タグベースのコスト配賦、リザーブド購入の判断 |
| オペレーショナルエクセレンス | 監視の集約、IaC による再現性、命名規則 | CI/CD、`DeployIfNotExists` の本格運用、モジュール化 |
| パフォーマンス効率 | サブネット分離で層を分ける素地 | 実アプリの SKU 選定、スケール設計 |

### できていること ── セキュリティと統制を「土台の段階で」織り込んだ

この土台の一番の美点は、**アプリを 1 つも載せる前に、セキュリティと統制を先に敷いた**ことです。ID はテナントに集約され、権限はグループ経由でのみ付与され、逸脱はポリシーで止まり、通信は既定で絞られています。多くの現場で「後回しにして事故る」順番を、本書は逆にしました。アプリを載せてからセキュリティを足すのではなく、セキュリティの効いた土台の上にアプリを載せる——Well-Architected Framework が繰り返し説く「シフトレフト(前倒し)」を、章の順序そのもので体現しています。

コスト面も、器(管理グループ・サブスクリプション・リソースグループ・VNet・NSG)自体は無料だと各章で確認し、課金は中に置く実物からのみ発生するという原則を通してきました。予算アラートを両環境に置いたので、続巻でアプリを載せて課金が始まっても、最初から見張れます。

### まだできていないこと ── 続巻へ送る宿題を正直に残す

総仕上げでは、**できていないことを隠さない**のが誠実です。図 7-7 の右列が、本書のミニ土台に**あえて残した宿題**です。いくつかは各章で「続巻で扱う」と予告してきました。

- **本番の特権昇格(PIM)** — 第 2 章で方針だけ決め、常時割り当てはしていません。PIM(Privileged Identity Management、特権 ID 管理)の有効化には上位ライセンス(P2)が要り、実運用時に導入します。
- **PaaS のプライベート化(Private Endpoint)** — 第 4 章で「続巻最重要の伏線」として置きました。PaaS が既定でパブリック公開される Azure では、これを VNet へ引き込むのが要になります。
- **DR(Disaster Recovery、災害復旧)設計** — 第 1 章でリージョンペアと可用性ゾーンを認識しましたが、実際の冗長構成は載せるアプリの要件次第です。
- **CI/CD とモジュール化** — 第 6 章で Bicep の「型」までを扱い、パイプライン統合とモジュール化は二段構えで続巻に送りました。

これらを「未完成」と捉える必要はありません。ミニランディングゾーンは、**アプリの形が決まる前に確定できるところまで**を、意図的に切り取った土台です。宿題の多くは「載せるアプリが決まって初めて設計できる」もので、続巻で具体的なアプリと一緒に扱うのが自然です。土台は完成し、その上に何を載せるかで枝分かれする——それが最後の「続巻への分岐」です。

<div class="column">
<span class="column-title">コラム:「完成」とは「これ以上足さないこと」ではない</span>

土台の完成を、「もう何も足さない状態」と誤解しないでください。ここでいう完成は、「**載せるアプリが決まる前に決めておくべきことを、すべて決め終えた**」という意味です。ID の置き方、統制の敷き方、ネットワークの区画、ログの集約先——これらはアプリに依存せずあらかじめ固められる、いわば普遍の土台です。逆に、DR や Private Endpoint や SKU 選定は、アプリの要件が決まってからでないと決められません。本書が「ここで一区切り」とするのは、アプリ非依存の部分を land させ切ったからです。この線引きこそ、ランディングゾーンという考え方の核心です。

</div>

## 構築手順 ── 到達状態のセルフ監査とリポジトリ整理

  handson-logs/ch07/ には検証ログが未取得のため、監査コマンドには要検証を付す。

最後に手を動かします。ただし、この章では**新しいリソースを 1 つも作りません**。代わりに、図 7-5 の全体構成図が実際の Azure 環境と一致しているかを、読み取り専用のコマンドで**セルフ監査**します。あわせて、リポジトリ側(state.md・Bicep・要検証マーカー)を整え、本書を「いつでも読み返せる・引き継げる」状態にします。

セルフ監査は、すべて `list` / `show` 系の**参照コマンド**です。環境を変更しないので、安心して何度でも流せます。以下のコマンド・出力は、`handson-logs/ch07/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch07 の検証ログ未取得。以下のセルフ監査コマンドの引数名・出力形式を実機で確認する -->

### 1. 器の骨格を監査する(第1章の到達状態)

まず、管理グループとサブスクリプション、リソースグループが図 7-5 のとおり並んでいるかを確認します。

```bash
# 管理グループの一覧(mg-foundation があるか)
az account management-group list \
  --query "[].{name:name, displayName:displayName}" -o table

# サブスクリプションの一覧(dev/prod が見えるか)
az account list \
  --query "[].{name:name, id:id, state:state}" -o table

# リソースグループの一覧(rg-foundation-dev/-prod・rg-monitor-foundation)
az group list \
  --query "[].{name:name, location:location, env:tags.env}" -o table
```

<!-- 要検証: handson-logs/ch07 の検証ログ未取得。管理グループ/サブスクリプション/リソースグループの一覧コマンドと出力を実機で確認する -->

図 7-5 の器(`mg-foundation` / dev・prod サブスクリプション / 3 つのリソースグループ)がすべて出そろい、リージョンが東日本、タグ `env` が付いていれば、第 1 章の到達状態は保たれています。

### 2. ID と権限を監査する(第2章の到達状態)

次に、グループと RBAC の割り当てを確認します。個人へ直接割り当てず、グループ経由で付与している原則が守られているかが要点です。

```bash
# dev サブスクリプションのロール割り当て(グループ経由か)
az role assignment list \
  --subscription "sub-foundation-dev" \
  --query "[].{principal:principalName, role:roleDefinitionName, type:principalType}" -o table
```

<!-- 要検証: handson-logs/ch07 の検証ログ未取得。az role assignment list の出力(principalType が Group になっているか)を実機で確認する -->

割り当て先(principalType)が `User` ではなく `Group` になっていること、dev では開発者グループが「共同作成者」、prod では「閲覧者」になっていることを確かめます。ここが `User` だらけなら、第 2 章の原則から外れているサインです。

### 3. ガバナンスを監査する(第3章の到達状態)

続いて、ポリシー割り当てを確認します。共通ルールが管理グループに、prod 限定ルールが prod サブスクリプションに、それぞれ割り当てられているはずです。

```bash
# 管理グループに割り当てた共通ポリシー(必須タグ・許可リージョン)
az policy assignment list \
  --scope "/providers/Microsoft.Management/managementGroups/mg-foundation" \
  --query "[].{name:name, policy:displayName}" -o table

# prod サブスクリプションの追加ポリシー(パブリック IP 禁止・必須タグ拒否)
az policy assignment list \
  --subscription "sub-foundation-prod" \
  --query "[].{name:name, policy:displayName}" -o table
```

<!-- 要検証: handson-logs/ch07 の検証ログ未取得。az policy assignment list のスコープ指定と出力、effect(Audit/Deny)の確認方法を実機で確認する -->

dev と prod の強弱の差が、権限(第 2 章)だけでなくポリシー(第 3 章)でも二重に表現されていること——これが統制の効いた土台の証です。

### 4. ネットワークと監視を監査する(第4・5章の到達状態)

ネットワークとログ集約を確認します。dev/prod の VNet(各 1 つ)・サブネット(各 3 つ)・ログ集約先(ただ 1 つ)が、図 7-5 のとおりであることを確かめます。

```bash
# dev の VNet とサブネット(app/data/Bastion の 3 つ)
az network vnet subnet list \
  --resource-group "rg-foundation-dev" \
  --vnet-name "vnet-foundation-dev" \
  --query "[].{name:name, prefix:addressPrefix}" -o table

# ログ集約先の Log Analytics ワークスペース(ただ 1 つ)
az monitor log-analytics workspace list \
  --resource-group "rg-monitor-foundation" \
  --query "[].{name:name, location:location}" -o table

# dev/prod の予算アラート
az consumption budget list \
  --query "[].{name:name, amount:amount}" -o table
```

<!-- 要検証: handson-logs/ch07 の検証ログ未取得。サブネット一覧・ワークスペース一覧・予算一覧のコマンドと出力を実機で確認する。az consumption budget list の可否も確認する -->

ワークスペースが `log-foundation` の**1 つだけ**であること、両サブスクリプションに予算があること、第 4 章の疎通確認で使った仮想マシン・Azure Bastion・パブリック IP が**残っていない**ことを確認します。消し忘れの課金リソースがないかは、この総仕上げで必ず点検します。
