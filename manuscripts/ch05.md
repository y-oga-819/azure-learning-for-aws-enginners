## 概要 ── Azure Monitor と Log Analytics

第 1 章で器の骨格を、第 2 章で ID と権限の層を、第 3 章でガバナンスの層を、第 4 章でネットワークの層を重ねてきました。ここまでで、リソースを作り・権限を絞り・通信の道を敷く準備が整いました。ですが、その土台の上で「**いま何が起きているか**」「**先月いくら使ったか**」を、まだ 1 か所も見られません。この章で足すのは、土台を運用するための **監視とコストの目**です。

Azure でこの目の役割を担う中心の道具が **Azure Monitor** です。Azure Monitor は、リソースが吐き出すデータを受け止め、検索・可視化・アラートへつなげる、監視の総合窓口です。まず押さえてほしいのは、Azure Monitor の扱うデータが **メトリックとログの二層**に分かれている、という構造です。

| 層 | 何を表すか | 収集のされ方 | 主な保存先 |
|---|---|---|---|
| メトリック | CPU 使用率・要求数などの**数値**を時系列で | 各リソースが**自動で**発行 | Azure Monitor の時系列データベース |
| ログ | 操作履歴・診断イベントなどの**構造化された記録** | **診断設定**で送り先を明示 | Log Analytics ワークスペース |

<p class="caption">図5-1 Azure Monitor が扱うデータの二層(メトリックとログ)</p>

**メトリック**は、リソースを作れば既定で自動収集されます。CPU やメモリ、要求数といった数値が、設定なしで一定間隔で溜まっていきます。ここは AWS の CloudWatch メトリックと同じ感覚で読めます。

一方の**ログ**は、そのままでは 1 か所に集まりません。各リソースの操作履歴や診断イベントを「どこへ送るか」を、**診断設定(Diagnostic Settings)**で明示して初めて集約されます。そして、その送り先の主役が **Log Analytics ワークスペース**です。

### Log Analytics ワークスペースは「ログの集約先」

**Log Analytics ワークスペース**は、Azure 中のログを受け止める器です。仮想マシン、NSG、ストレージ、あるいはサブスクリプションの操作履歴——種類の違うログを、1 つのワークスペースへ流し込みます。集めたログは **KQL(Kusto Query Language)**という問い合わせ言語で横断検索でき、テーブルをまたいで「誰が」「いつ」「何をしたか」を追えます。

この「**全ログを 1 つのワークスペースへ集約する**」という発想が、Azure の監視設計の中心です。AWS のようにサービスごとにロググループが自然に分かれるのではなく、送り先を自分で決めて 1 か所へ寄せる——この設計文化を、本章で土台に組み込みます。

<div class="column">
<span class="column-title">コラム:アクティビティログは最初から溜まっている</span>

「誰がどのリソースを作成・変更・削除したか」という操作の証跡は、**アクティビティログ(Activity Log)**として、設定なしでも各サブスクリプションに記録されています。AWS の CloudTrail に相当する、統制の要となるログです。

ただし、既定の保持は短期間で、横断検索にも向きません。<!-- 要検証: 公式Doc確認 — アクティビティログの既定保持期間 --> そこで本章では、このアクティビティログを**診断設定でワークスペースへ送り**、KQL で長く追える状態にします。「最初から溜まっているが、集約して初めて使える」——この一手間が、監視の土台づくりの第一歩です。

</div>

## AWSとの対応と相違 ── CloudWatch/CloudTrail との対応

監視とコストまわりを AWS と対応づけます。AWS では CloudWatch・CloudTrail・Cost Explorer と、役割ごとにサービスが分かれていました。Azure でも道具立ては似ていますが、「ログを 1 か所へ集約する」設計文化と、そこへ**明示的に送る**という作法が、AWS 経験者の習慣とずれる点です。まず対応表で全体像を掴みます。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| メトリック | CloudWatch メトリック | Azure Monitor メトリック | 自動収集される点は同じ |
| ログ集約 | CloudWatch Logs | Log Analytics ワークスペース | 「全ログを 1 ワークスペースへ集約」する文化 |
| 監査証跡 | CloudTrail | アクティビティログ | 既定で記録。集約は診断設定で明示 |
| ログクエリ | Logs Insights | KQL(Kusto Query Language) | クエリ言語が別物。パイプでつなぐ発想は近い |
| ダッシュボード | CloudWatch ダッシュボード | ブック(Workbooks)/ ダッシュボード | — |
| コスト管理 | Cost Explorer / Budgets | Cost Management + 予算 | 予算アラートの考え方はほぼ同じ |

<p class="caption">図5-2 監視・コスト系サービスの AWS と Azure の対応</p>

### 決定的な違い1 ── ログは「勝手に集まらない」、送り先を規約で決める

AWS では、多くのマネージドサービスがログを CloudWatch Logs のロググループへ、比較的自然に吐き出していました。サービスごとにロググループが分かれ、必要ならそこを見に行く、という運用に馴染んでいたはずです。

Azure の考え方は逆向きです。リソースを作っただけでは、その診断ログはどこにも溜まりません。**診断設定でワークスペースを指定して初めて**、ログが流れ始めます。これは一見面倒ですが、裏を返せば「**送り先を自分で決められる**」ということです。バラバラに散らばるのではなく、最初から 1 つのワークスペースへ寄せる規約を敷けば、横断検索・権限管理・コスト把握のいずれも 1 か所で済みます。

そのため Azure では、「リソースを作ったら診断設定も必ず入れる」という**規約**を、土台の段階で決めておくことが重要になります。本章の「全体構成に組み込む」で、まさにこの規約を立てます。

### 決定的な違い2 ── KQL は Logs Insights とは別物

集めたログを検索する言語も違います。AWS の CloudWatch Logs Insights には独自のクエリ構文がありました。Azure では **KQL(Kusto Query Language)**を使います。名前や文法は別物ですが、「**テーブルを起点に、パイプ(`|`)で絞り込み・整形・集計をつなぐ**」という発想は、Logs Insights に通じるものがあります。

たとえば「直近 24 時間の書き込み操作を新しい順に見る」は、KQL では次のように書きます。テーブル名から始め、`|` で条件を足していく読み口です。

```kql
AzureActivity
| where TimeGenerated > ago(24h)
| where OperationNameValue has "write"
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup
| order by TimeGenerated desc
```

<p class="caption">図5-3 KQL の基本形。テーブルを起点にパイプで絞り込む</p>

<!-- 要検証: handson-logs/ch05 の検証ログ未取得。AzureActivity テーブルの列名(OperationNameValue / ActivityStatusValue / Caller など)を実機で確認する -->

`where` で絞り込み、`project` で表示する列を選び、`order by` で並べ替える——この最小の型を覚えるだけで、操作履歴の追跡はひととおりこなせます。KQL は本シリーズを通じて何度も使うため、ここで「テーブル｜条件｜整形」の骨格に慣れておきます。

### 決定的な違い3 ── コストは Cost Management に集約、予算アラートで事故を防ぐ

コスト管理は、AWS の Cost Explorer / Budgets に相当する **Cost Management + 予算**で行います。考え方はよく似ていて、サブスクリプションや管理グループのスコープで使用量と費用を可視化し、**予算(Budget)**を作ってしきい値を超えたらアラートを飛ばせます。

AWS 経験者にとって新しいのは、コストの境界が**サブスクリプション**である点です。第 1 章で dev/prod をサブスクリプションで分けたので、コストも環境ごとに素直に切り分けて見られます。学習用途では、この予算アラートが「**消し忘れによる課金事故**」を防ぐ命綱になります。本章の最後で、この予算アラートを設定します。

## 全体構成に組み込む ── 診断ログの集約先を規約化する

第 4 章までで、器・権限・ガバナンス・ネットワークがそろいました。ですが、それらのリソースが吐くログは、まだどこにも集まっていません。この章で足すのは、**すべての診断ログを受け止める 1 つのワークスペース**と、「新しく作るリソースは必ずそこへ送る」という**規約**です。

### 監視用リソースグループとワークスペースを 1 つだけ作る

ログの集約先は、**1 つのワークスペースにまとめます**。dev と prod でリソースは分かれていても、監視の窓口は 1 か所にしておくほうが、環境をまたいだ調査やコスト把握が楽になります。そのため、監視専用のリソースグループ `rg-monitor-foundation` を作り、その中にワークスペース `log-foundation` を 1 つだけ置きます。

置き場所は、統制の効いた **prod サブスクリプション**とします。理由は 2 つあります。第 1 に、ログは統制の証跡そのものなので、より厳しく守られた側に置くのが自然だからです。第 2 に、第 2 章で開発者グループには prod の**閲覧者(Reader)**を割り当てたため、開発者は**ログを見られるが消せない**という、監視のログにふさわしい権限の形になるからです。「dev は緩く、prod は厳しく」という本書の合言葉が、ログの集約先にもそのまま効きます。

<div class="column">
<span class="column-title">コラム:本来は「管理サブスクリプション」に置く</span>

実際のランディングゾーンでは、共有サービス(集約ログ・監視・共有ネットワークなど)を専用の**管理(プラットフォーム)サブスクリプション**に寄せ、他のすべてのサブスクリプションからそこへログを送ります。本書のミニ構成では、この管理サブスクリプションを別途は作らず、統制の効いた prod サブスクリプションにその役割を圧縮しています。続巻や実案件で規模が大きくなったら、この 1 ワークスペースを管理サブスクリプションへ切り出す——その分岐点だけ、頭の隅に置いておいてください。

</div>

### 「診断設定は必ずこのワークスペースへ」という規約

ワークスペースを作ったら、次に立てるのが**規約**です。本書ではこう決めます——「**このミニ構成で新しく作るリソースは、診断設定で必ず `log-foundation` へログを送る**」。第 4 章で作った NSG のフローログも、これから作るリソースの診断ログも、送り先はすべてこの 1 か所です。

規約を人手の徹底に頼ると、いつか抜けます。そこで、第 3 章で学んだ **Azure Policy** を使い、「診断設定が無ければ自動で追加する」ところまで踏み込めます。これが `DeployIfNotExists`(存在しなければデプロイする)という種類のポリシーです。第 3 章では監査(Audit)と拒否(Deny)を扱いましたが、Azure Policy には**足りない設定を自動で補う(修復する)**効果もあります。診断設定の付与は、その代表例です。本章では規約の考え方と自動化の入り口までを示し、修復ポリシーの本格運用は続巻に譲ります。

<div class="note">
<span class="note-title">注意:自動化ポリシーはマネージド ID を必要とする</span>

`DeployIfNotExists` のポリシーは、割り当て時に**マネージド ID(Managed Identity)**を作り、その ID の権限でリソースへ診断設定を書き込みます。第 2 章で「キー認証を根絶する」文化の中心として触れたマネージド ID が、ここで統制の自動化の実行役として登場します。割り当て時には、この ID に適切なロールを与える必要があります。<!-- 要検証: 公式Doc確認 — 診断設定の DeployIfNotExists ポリシーが要求するロール(監視共同作成者 等)の正確な名称 -->

</div>

### Before / After ── 監視とコストの目が加わる

第 4 章を終えた時点では、dev/prod にリソースは並んでいますが、そのログを受け止める場所も、費用を見張る仕組みもありません。

```text
管理グループ: mg-foundation

│  └─ rg-foundation-dev
│     └─ vnet-foundation-dev(10.10.0.0/16)+ NSG

   └─ rg-foundation-prod
      └─ vnet-foundation-prod(10.20.0.0/16)+ NSG
```

<p class="caption">図5-4 第4章の到達状態(Before)。ログはまだ 1 か所に集まっていない</p>

第 5 章を終えると、prod サブスクリプションに監視用のリソースグループと 1 つのワークスペースが加わり、dev/prod 双方のリソースと操作履歴が、診断設定を通じてそこへ流れ込みます。あわせて、各サブスクリプションに月額の予算アラートが乗ります。

```text
【After】第5章の到達状態(監視とコストの層を追加)
管理グループ: mg-foundation
├─ サブスクリプション: sub-foundation-dev  ── 予算アラート
│  └─ rg-foundation-dev
│     └─ vnet-foundation-dev + NSG ──┐(診断設定)
│                                     │
├─ サブスクリプション: sub-foundation-prod ── 予算アラート
│  ├─ rg-foundation-prod
│  │  └─ vnet-foundation-prod + NSG ─┤(診断設定)
│  └─ rg-monitor-foundation           │
│     └─ log-foundation(ワークスペース)◀┘
│        ▲
└────────┘ アクティビティログ(dev/prod)を診断設定で送る
```

<p class="caption">図5-5 第5章の到達状態(After)。全リソースのログが 1 ワークスペースへ集約される</p>

これで、統制・ネットワークの土台に、**運用の目**が加わりました。まだアプリはありませんが、続巻でアプリを載せたとき、そのログとメトリックの受け皿はもう用意できています。「作ったら、必ずここへ送る」という規約が先にあるので、リソースが増えても集約先が散らばりません。

## 設計のポイント ── ワークスペース集約と保持期間、課金事故防止

監視とコストの土台では、判断を迷いやすい点が 3 つあります。**ワークスペースの集約と分割**、**ログの保持期間**、そして**学習用の課金事故対策**です。要点をチェックリストにまとめます。

| 分類 | 原則 | 理由 |
|---|---|---|
| 集約 | ワークスペースは**まず 1 つに集約**する | 環境横断の調査・権限・コスト把握が 1 か所で済む |
| 集約 | 分けるのは**明確な理由がある時だけ** | 分割は運用を複雑にする。迷ったら集約 |
| 保持 | 保持期間は**目的から逆算**して決める | 長く持つほど課金が増える |
| コスト | ログは**取り込み量と保持で課金**される | 「全部・無期限」は事故のもと |
| コスト | サブスクリプションごとに**予算アラート**を置く | 消し忘れの課金を早期に検知する |
| 統制 | 診断設定は**規約(できれば自動化)**で徹底する | 人手の徹底はいつか抜ける |

<p class="caption">図5-6 監視とコスト設計のチェックリスト</p>

### 分けるか集約するか ── 迷ったら 1 つに集約

ワークスペースは、いくつも作れます。ですが、まず立てるべき原則は「**1 つに集約する**」です。ログが 1 か所にあれば、dev で見えた異常の関連事象を prod 側まで一気に追え、権限とコストを 1 つの器で管理できます。ワークスペースを分けるほど、横断検索は分断され、運用は複雑になります。

分割を検討してよいのは、次のように**明確な理由がある時だけ**です。判断の目安を挙げます。

| 分ける動機 | 具体例 | 本書の判断 |
|---|---|---|
| データ所在地の要件 | リージョンをまたいで分離する義務がある | 本書は東日本のみ。集約 |
| 権限の完全分離 | 別部門にログを一切見せられない | RBAC で足りる。集約 |
| 桁違いのコスト差 | 特定環境だけログ量が極端 | 学習規模では起きない。集約 |

<p class="caption">図5-7 ワークスペースを分割してよい動機と、本書の判断</p>

本書はミニ構成なので、これらの動機はどれも当てはまりません。よって、**1 つのワークスペースへ集約**します。見た目の整理のためだけに分けないこと。これが運用を軽く保つコツです。

### 保持期間は目的から逆算する

ログは、ワークスペースへ**取り込む量**と、**保持する期間**に応じて課金されます。<!-- 要検証: 公式Doc確認 — Log Analytics の課金体系(取り込み従量・保持期間・無料枠の有無) --> つまり「何でも・無期限に」溜めると、費用は静かに膨らみます。保持期間は、その環境で「**いつまで遡って調べたいか**」から逆算して決めます。

学習用の dev なら、直近の動作確認ができれば十分で、長期保持は要りません。統制の証跡が要る prod は、もう少し長く持ちたくなります。まずは短めの既定から始め、必要になったら延ばす——短いほうから調整するのが、課金の面でも安全です。

<div class="note">
<span class="note-title">注意:「とりあえず全部・長期保持」は課金事故のもと</span>

診断設定では、送れるログの種類を細かく選べます。ここで「よく分からないから全部」「念のため長期保持」と倒すと、使わないログの取り込みと保持に費用がかかり続けます。**本当に見るログだけ**を、**必要な期間だけ**——診断設定と保持期間は、この 2 つを意識して最小限から始めてください。学習段階では特に、次項の予算アラートと合わせて防御します。

</div>

### 学習用の課金事故を予算アラートで防ぐ

本書の読者にとって、監視章の最大の実利は「**課金事故を防ぐこと**」です。第 4 章では Azure Bastion や仮想マシンの消し忘れに注意しました。こうした「置きっぱなし」による想定外の請求を早期に捕まえるのが、**予算アラート**です。

サブスクリプションごとに月額の予算を決め、たとえば **50% / 80% / 100%** といったしきい値を超えたらメールで通知します。金額自体を止める仕組みではありませんが、「気づいたら万単位」という事故を、初期段階で知らせてくれます。dev/prod それぞれに予算を置き、通知先を自分のメールにしておけば、学習中の安心料としては十分です。この設定を、次の構築手順で実際に行います。

## 構築手順 ── ログ集約・KQL検索・予算アラート

ここからは手を動かします。目的は次の 4 つです。ワークスペースを作る、アクティビティログを集約する、KQL で操作履歴を検索する、予算アラートを設定する——この順で進めます。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

以下のコマンド・出力は、`handson-logs/ch05/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch05 の検証ログ未取得。コマンド・出力・所要時間を実機で確認する -->

### 1. 監視用リソースグループとワークスペースを作る

集約先を、prod サブスクリプションに 1 つだけ作ります。まず操作対象を prod サブスクリプションへ合わせます。

```bash
# 集約先は prod サブスクリプションに置く
az account set --subscription "sub-foundation-prod"
```

監視専用のリソースグループを作り、その中に Log Analytics ワークスペースを 1 つ作成します。タグは第 3 章の必須タグポリシーに合わせて `env` / `owner` を付けます。

```bash
# 監視用リソースグループを作成する
az group create \
  --name "rg-monitor-foundation" \
  --location "japaneast" \
  --tags env=prod owner="you@example.com"

# Log Analytics ワークスペースを 1 つ作成する
az monitor log-analytics workspace create \
  --resource-group "rg-monitor-foundation" \
  --workspace-name "log-foundation" \
  --location "japaneast" \
  --tags env=prod owner="you@example.com"
```

<!-- 要検証: handson-logs/ch05 の検証ログ未取得。az monitor log-analytics workspace create の引数名(--workspace-name)と、保持期間を指定する引数(--retention-time 等)を実機で確認する -->

保持期間は、既定のまま短めから始めます。長期保持が必要になったら、あとから延ばせます。<!-- 要検証: 公式Doc確認 — ワークスペースの既定保持期間と、無料枠に収まる保持日数 -->

### 2. アクティビティログを診断設定でワークスペースへ送る

第 1 の集約対象は、各サブスクリプションの**アクティビティログ**です。dev/prod それぞれのアクティビティログを、いま作ったワークスペースへ診断設定で送ります。まず、送り先ワークスペースのリソース ID を控えます。

```bash
# ワークスペースのリソース ID を取得する
WORKSPACE_ID=$(az monitor log-analytics workspace show \
  --resource-group "rg-monitor-foundation" \
  --workspace-name "log-foundation" \
  --query id -o tsv)
```

prod サブスクリプションのアクティビティログを、ワークスペースへ送る診断設定を作成します。

```bash
# prod のアクティビティログをワークスペースへ送る
az monitor diagnostic-settings subscription create \
  --name "to-log-foundation" \
  --location "japaneast" \
  --workspace "$WORKSPACE_ID" \
  --logs '[{"category":"Administrative","enabled":true},
           {"category":"Security","enabled":true},
           {"category":"Policy","enabled":true}]'
```

<!-- 要検証: handson-logs/ch05 の検証ログ未取得。az monitor diagnostic-settings subscription create の引数名と、--logs で指定できるアクティビティログのカテゴリ名(Administrative / Security / Policy 等)を実機で確認する -->

dev サブスクリプションでも、同じ要領で診断設定を作ります。dev の操作対象へ切り替えてから、送り先には**同じ**ワークスペースの ID を指定します(集約先は 1 つに保ちます)。

```bash
# dev へ切り替え、送り先は同じ log-foundation を指定する
az account set --subscription "sub-foundation-dev"
az monitor diagnostic-settings subscription create \
  --name "to-log-foundation" \
  --location "japaneast" \
  --workspace "$WORKSPACE_ID" \
  --logs '[{"category":"Administrative","enabled":true},
           {"category":"Security","enabled":true},
           {"category":"Policy","enabled":true}]'
```

これで、dev/prod 両方のアクティビティログが 1 つのワークスペースへ流れ始めます。第 4 章で作った NSG のフローログなど、個々のリソースの診断ログも、同じ要領で送り先にこのワークスペースを指定します。「新しく作るリソースは必ずここへ送る」という規約を、こうして 1 つずつ形にしていきます。

<div class="column">
<span class="column-title">コラム:規約を人手で守らせない ── 自動化への入り口</span>

診断設定を毎回手で入れるのは、いつか抜けます。第 3 章で扱った Azure Policy には、「診断設定が無いリソースを検知し、自動で付与する」`DeployIfNotExists` 型の組み込みポリシーが用意されています。<!-- 要検証: 公式Doc確認 — 診断設定を強制する組み込みポリシー/イニシアティブの正式名称 --> これを管理グループ `mg-foundation` に割り当てれば、配下のサブスクリプションで新しく作られたリソースに、診断設定が自動で入るようにできます。割り当てにはマネージド ID とロール付与、既存リソースへの修復(Remediation)タスクが必要です。本書では規約の徹底までを手順で示し、この自動化の本格運用は続巻で扱います。

</div>

### 3. KQL で操作履歴を検索する

ログが溜まり始めたら、KQL で検索して集約を体感します。ログが反映されるまで数分の遅延があります。<!-- 要検証: 公式Doc確認 — アクティビティログが診断設定経由でワークスペースに現れるまでの遅延 --> ポータルの「Log Analytics ワークスペース → ログ」を開き、次のクエリを実行します。

まず、直近 24 時間の操作を新しい順に一覧します。

```kql
AzureActivity
| where TimeGenerated > ago(24h)
| project TimeGenerated, Caller, OperationNameValue, ResourceGroup, ActivityStatusValue
| order by TimeGenerated desc
```

<!-- 要検証: handson-logs/ch05 の検証ログ未取得。AzureActivity テーブルの列名を実機で確認する -->

次に、「誰が何回操作したか」を集計してみます。`summarize` で件数をまとめ、多い順に並べます。

```kql
AzureActivity
| where TimeGenerated > ago(24h)
| summarize 操作回数 = count() by Caller
| order by 操作回数 desc
```

<p class="caption">図5-8 KQL で操作履歴を集計する例(直近24時間・操作者別)</p>

クエリは `where` で絞り込み、あとは `project`・`summarize`・`order by` を続けて整形するだけ——第 2 節で示した骨格そのままです。dev 側の診断設定の操作も、prod 側のワークスペースの操作も、**同じ 1 つのテーブル**に並んで見えるはずです。これが「集約」の効果です。バラバラのサービスを見に行くのではなく、1 か所へ問い合わせれば土台全体の動きが追えます。

### 4. 予算アラートを設定する

最後に、課金事故を防ぐ予算アラートを設定します。サブスクリプションごとに月額予算を決め、しきい値を超えたらメールで通知します。ここでは prod に月額 5,000 円の予算を置き、50% / 80% / 100% で通知する例を示します。金額は自分の学習ペースに合わせて調整してください。

```bash
# prod に月額予算とアラートを設定する
az account set --subscription "sub-foundation-prod"
az consumption budget create \
  --budget-name "budget-foundation-prod" \
  --amount 5000 \
  --time-grain Monthly \
  --category Cost \
  --start-date "2026-08-01" \
  --end-date "2027-08-01"
```

<!-- 要検証: handson-logs/ch05 の検証ログ未取得。予算作成の正しいコマンド(az consumption budget create の可否・引数、通知しきい値と連絡先メールの指定方法)を実機で確認する。ポータルの Cost Management での設定手順も併記する -->

dev 側も、同じ要領で予算アラートを設定します。学習中は dev のほうがリソースを作っては消すため、こちらの予算アラートが特に効きます。金額を超えたら止まるわけではないので、**通知を受けたら不要なリソースを消す**——アラートはあくまで気づくための仕組みです。第 4 章で触れた Azure Bastion や仮想マシンの消し忘れも、この予算アラートが早期に知らせてくれます。予算アラートの詳しい設定は、付録 D でも補足します。

<div class="note">
<span class="note-title">注意:予算アラートは「止める」ものではない</span>

予算アラートは、しきい値を超えたことを**通知する**だけで、リソースを自動で停止・削除はしません。通知が来ても放置すれば、課金は増え続けます。アラートは「気づく」ためのもの、実際に止めるのは自分の手——この役割分担を誤解しないでください。学習環境では、使い終わったら消す習慣と予算アラートの両輪で、課金事故を防ぎます。

</div>

以上で、監視用リソースグループと 1 つのワークスペースを作り、dev/prod のアクティビティログを集約しました。さらに KQL で操作履歴を横断検索し、予算アラートで課金事故に備えるところまでを整えました。土台に「運用の目」が付き、続巻でアプリを載せたときも、そのログとコストを最初から 1 か所で見張れます。

**この章のまとめ**

- **Azure Monitor** はデータを**メトリックとログの二層**で扱います。メトリックは自動収集されますが、ログは**診断設定**で送り先を明示して初めて集約されます。
- ログの集約先が **Log Analytics ワークスペース**です。Azure では「**全ログを 1 つのワークスペースへ集約する**」設計文化が中心で、AWS のように勝手には集まりません。送り先を**規約**で決めるのが要点です。
- AWS の CloudWatch / CloudTrail / Cost Explorer は、Azure では Azure Monitor メトリック / アクティビティログ / **Cost Management + 予算**に対応します。ログ検索は **KQL(Kusto Query Language)**で行い、「テーブル｜条件｜整形」の型で読み書きします。
- 土台には、監視用リソースグループ `rg-monitor-foundation` と**ただ 1 つ**のワークスペース `log-foundation` を、統制の効いた prod サブスクリプションへ置きました。dev/prod のアクティビティログをそこへ集約し、「新規リソースは必ずここへ送る」規約を立てました(自動化は `DeployIfNotExists` ポリシーで、本格運用は続巻)。
- 保持期間は目的から逆算して短めに始め、サブスクリプションごとの**予算アラート**で学習用の課金事故を防ぎます。この監視とコストの目は、第 6 章で土台を Bicep 化し、第 7 章で到達状態を総点検する際の前提になります。
