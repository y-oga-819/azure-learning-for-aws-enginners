## 概要 ── テナントの ID とスコープで効く RBAC

第 1 章で、Azure の最上位にあるのはリージョンやアカウントではなく、組織を表す **テナント(Tenant)** だと確認しました。この章では、そのテナントに一元管理される ID の基盤 **Microsoft Entra ID** を扱います。あわせて、その ID へ「何をどこまで許すか」を割り当てる **RBAC(Role-Based Access Control、ロールベースアクセス制御)** を学びます。

AWS 経験者にとって、ここは最もつまずきやすい章です。名前だけ見れば「Entra ID = IAM」と読み替えたくなりますが、**両者は別物**です。IAM がアカウントの中に閉じた仕組みだったのに対し、Entra ID はアカウント(サブスクリプション)の外側、テナントに存在します。そして権限の与え方も、JSON でルールを書く発想から「**組み込みロールをスコープに割り当てる**」という発想へ切り替わります。まずこの 2 つの土台を押さえます。

### Entra ID:テナント全体の ID ディレクトリ

Entra ID は、テナントに属する **ユーザー・グループ・アプリケーション**などの ID を保持するディレクトリです。第 1 章で見たとおり、テナントと Entra ID のディレクトリは 1 対 1 で対応します。つまり「この組織に誰がいるか」は、サブスクリプションごとではなく、テナントに 1 か所だけ存在します。

Entra ID が管理する主な ID を、先に用語として並べておきます。

| 種別 | 何を表すか | 主な用途 |
|---|---|---|
| ユーザー(User) | 人。社員や招待した外部ユーザー | サインインして操作する主体 |
| グループ(Group) | ユーザーの束 | 権限をまとめて割り当てる単位 |
| サービスプリンシパル(Service Principal) | アプリ・自動化に与える ID | CI/CD やスクリプトの実行主体 |
| マネージド ID(Managed Identity) | Azure リソース自身に与える ID | キーを持たずに他リソースへアクセス |

<p class="caption">図2-1 Entra ID が管理する主な ID の種類</p>

このうち**グループ**は本章の主役です。権限は個人に直接ではなくグループに割り当て、人はグループに出入りさせる——これが後述する運用の原則になります。グループには、手動でメンバーを出し入れする**割り当て済みグループ**と、「部署が営業なら自動で入る」のように属性で自動判定する**動的グループ(Dynamic Group)**があります。<!-- 要検証: 公式Doc確認 — 動的グループに必要な Entra ID ライセンス(P1)の要否 -->

### RBAC:スコープ × ロール × プリンシパル

Azure の認可(誰が何をできるか)は、**RBAC** で表現します。RBAC は、次の 3 つを結びつける **ロール割り当て(Role Assignment)** が最小単位です。

```text
   プリンシパル          ロール              スコープ
  (誰が)      ×      (何を)      ×      (どこで)
  ユーザー            所有者              管理グループ
  グループ            共同作成者          サブスクリプション
  サービスプリンシパル  閲覧者              リソースグループ
  マネージド ID        カスタムロール       個別リソース
        └──────────── ロール割り当て ────────────┘
```

<p class="caption">図2-2 RBAC は「プリンシパル × ロール × スコープ」を結びつける</p>

3 つの要素は、それぞれ次を指します。

- **プリンシパル(誰が)** — 権限を受け取る主体です。ユーザー・グループ・サービスプリンシパル・マネージド ID のいずれかです。
- **ロール(何を)** — 許可する操作の集まりです。Azure には多数の**組み込みロール**が用意され、代表的なものが後述の所有者・共同作成者・閲覧者です。足りなければ**カスタムロール**を定義できます。
- **スコープ(どこで)** — 権限が効く範囲です。第 1 章の 4 階層(管理グループ/サブスクリプション/リソースグループ)と個別リソースが、そのままスコープになります。上位スコープで割り当てた権限は、下位へ**継承**されます。

代表的な組み込みロールは、まずこの 3 つだけ覚えれば十分です。

| ロール | できること | 使いどころ |
|---|---|---|
| 所有者(Owner) | リソースの全操作+**権限の付与** | 管理者。付与できる点が強力 |
| 共同作成者(Contributor) | リソースの全操作。ただし権限の付与は不可 | 開発者。作るが権限は配れない |
| 閲覧者(Reader) | 参照のみ。変更不可 | 監査・閲覧。本番を「見るだけ」 |

<p class="caption">図2-3 まず覚える 3 つの組み込みロール</p>

所有者と共同作成者の違いは、**他人に権限を配れるかどうか**です。この一点が統制上とても重要で、「作業はできるが権限は配れない」共同作成者を既定にするのが安全設計の基本になります。

### マネージド ID と PIM:キーレスと一時昇格の予告

本章の後半で設計に使うわけではありませんが、Azure の ID 文化を象徴する 2 つの仕組みを先に名前だけ紹介します。

1 つは **マネージド ID** です。Azure リソース(仮想マシンやアプリ)自身に ID を持たせ、接続文字列やアクセスキーを一切持たずに他のリソースへアクセスさせる仕組みです。「キー認証を根絶する」という Azure の思想の中心にあり、続巻で繰り返し登場します。

もう 1 つは **PIM(Privileged Identity Management、特権 ID 管理)** です。強い権限を常時割り当てておくのではなく、必要なときだけ **JIT(Just-In-Time、必要な時だけ)** で一時的に昇格させます。本番環境の管理者権限は、この PIM で「申請して、承認され、数時間だけ有効」という形にするのが理想です。詳しくは本章「設計のポイント」で扱います。<!-- 要検証: 公式Doc確認 — PIM に必要な Entra ID P2 ライセンスの要否 -->

## AWSとの対応と相違 ── IDがアカウントでなくテナントにある

Entra ID と RBAC を、AWS の IAM(Identity and Access Management)と対応づけます。最初に対応表を置き、そのうえで「似ているようで決定的に違う 3 点」を掘り下げます。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| ID の置き場所 | IAM(アカウント内) | Entra ID(テナント) | ID がアカウントの外側にある |
| 認可の書き方 | IAM ポリシー(JSON) | RBAC(ロール+スコープ) | 「組み込みロール+スコープ」で考える |
| 人の束ね方 | IAM グループ | Entra ID グループ | 割り当ての原則が同じ |
| アプリの ID | IAM ロール(AssumeRole) | サービスプリンシパル | アプリ登録から生まれる |
| ワークロード ID | インスタンスプロファイル / IRSA | マネージド ID | 適用範囲が広い |
| 一時的な特権 | 都度 AssumeRole | PIM による JIT 昇格 | 昇格を仕組み化する |

<p class="caption">図2-4 IAM と Entra ID / RBAC の対応</p>

### 決定的な違い1 ── ID がテナントにある

AWS では、IAM のユーザー・ロール・ポリシーは**アカウントの中**に閉じていました。別アカウントのリソースを操作するには、クロスアカウントの `AssumeRole` を組む必要がありました。マルチアカウント運用で ID を一元化したいときは、IAM Identity Center を上に乗せて中央ディレクトリを作りました。

Azure では、その中央ディレクトリにあたる Entra ID が**最初から土台そのもの**です。ID はサブスクリプションの中ではなく、その外側のテナントに一元的に存在します。1 人のユーザーが、同じ ID のまま dev と prod の両方のサブスクリプションで権限を持てます。第 1 章のコラムで触れた「AWS で IAM Identity Center を入れて実現していた世界が、Azure では既定の姿」という話が、ここで具体化します。

### 決定的な違い2 ── JSON ではなくロール割り当てで考える

AWS の IAM ポリシーは、`Effect` / `Action` / `Resource` / `Condition` を JSON で細かく書く仕組みでした。柔軟な反面、書き手ごとに粒度がばらつき、レビューが難しくなりがちでした。

Azure の RBAC は、発想が逆です。**まず用意された組み込みロールから選び**、それを**どのスコープに割り当てるか**で権限を決めます。「所有者を prod サブスクリプションに」「共同作成者を dev リソースグループに」というように、ロール(何を)とスコープ(どこで)の掛け算で考えます。JSON を書く場面は、組み込みロールで表せない要件が出てきて**カスタムロール**を作るときに限られます。

```text
【AWS】JSON でルールを記述する
  {
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }

【Azure】組み込みロールを、スコープに割り当てる
  共同作成者(ロール)  ×  rg-foundation-dev(スコープ)  →  開発者グループ
```

<p class="caption">図2-5 「JSON を書く」から「ロールを割り当てる」への発想転換</p>

<div class="note">
<span class="note-title">注意:似た名前でも中身が違う</span>

AWS の「ロール」は `AssumeRole` で引き受ける ID そのものでした。一方、Azure の「ロール」は権限の集まり(何ができるか)であり、ID ではありません。Azure でアプリに与える ID は、ロールではなく**サービスプリンシパル**や**マネージド ID** です。名前が同じ「ロール」でも指すものが違うため、読み替えるときは注意してください。

</div>

### 決定的な違い3 ── マネージド ID が「キーレス」の中心にある

AWS では、EC2 にインスタンスプロファイルを付け、EKS のポッドに IRSA(IAM Roles for Service Accounts)を割り当てて、キーを持たせずに権限を渡していました。Azure のマネージド ID は、この発想を Azure リソース全般へ広げたものです。仮想マシンだけでなく、アプリのホスティングサービスやデータ処理サービスなど、多くのリソースが自分自身の ID を持てます。

違いは適用範囲の広さと、文化の強さです。Azure では「接続文字列やアクセスキーをコードや構成に書くのは避け、マネージド ID で認証する」ことが、推奨ではなく**既定の作法**として繰り返し示されます。この「キー認証を根絶する」文化は本シリーズ全体の通奏低音であり、本章はその入り口です。

<div class="column">
<span class="column-title">コラム:オンプレ AD との同期(Entra Connect)</span>

実案件では、多くの企業がすでにオンプレミスの **Active Directory(AD)** を持っています。Entra ID は、この AD と別物です。名前が似ているため混同されがちですが、オンプレの AD はドメインコントローラー上の従来型ディレクトリ、Entra ID はクラウドの ID 基盤です。

両者は **Entra Connect** というツールで**同期**します。オンプレ AD のユーザーやグループを Entra ID へ複製し、社員は同じ ID で社内システムと Azure の両方へサインインできるようになります。パスワードの同期方式や、認証をどこで行うか(パススルー認証・フェデレーション)で複数の選択肢があります。

AWS 経験者にとっては、AD を IAM Identity Center や AWS Managed Microsoft AD と連携させていた構図に近いものです。「オンプレ AD ⇔ クラウドの ID 基盤」を橋渡しする仕組みがある、という前提だけ押さえておけば、実案件の会話についていけます。<!-- 要検証: 公式Doc確認 — Entra Connect の認証方式(PHS/PTA/フェデレーション)の最新名称 -->

</div>

## 全体構成に組み込む ── 管理者/開発者/閲覧者グループを割り当てる

第 1 章で、管理グループ `mg-foundation` の下に dev/prod のサブスクリプションを並べ、それぞれに最初のリソースグループを作りました。まだ「誰が何をできるか」は何も決まっていません。第 2 章では、この骨格に **ID と権限の層**を足します。

具体的には、**管理者・開発者・閲覧者**の 3 つのグループを作り、dev と prod に**異なる強さ**の RBAC を割り当てます。ここでの核は「**dev は緩く、prod は厳しく**」を権限で表現することです。

### Before / After ── 権限の層が加わる

第 1 章を終えた時点では、器はあっても権限は空でした。

```text
【Before】第1章の到達状態(権限は未設定)
テナント(Entra ID)
└─ 管理グループ: mg-foundation
   ├─ サブスクリプション: sub-foundation-dev
   │  └─ リソースグループ: rg-foundation-dev
   └─ サブスクリプション: sub-foundation-prod
      └─ リソースグループ: rg-foundation-prod
```

<p class="caption">図2-6 第1章の到達状態(Before)</p>

第 2 章を終えると、テナントに 3 つのグループができ、dev と prod にそれぞれ強さの違う RBAC が割り当たります。

```text
【After】第2章の到達状態(グループと RBAC を追加)
テナント(Entra ID)
├─ グループ: grp-foundation-admins ──────┐
├─ グループ: grp-foundation-developers ──┤ ロール割り当て
├─ グループ: grp-foundation-viewers ─────┘
│
└─ 管理グループ: mg-foundation
   ├─ サブスクリプション: sub-foundation-dev
   │    developers = 共同作成者 / viewers = 閲覧者
   └─ サブスクリプション: sub-foundation-prod
        developers = 閲覧者 / viewers = 閲覧者
        admins = 所有者(PIM で一時昇格)
```

<p class="caption">図2-7 第2章の到達状態(After)。dev は緩く・prod は厳しく</p>

### 割り当ての設計 ── dev は緩く、prod は厳しく

3 つのグループに、次の方針でロールを割り当てます。同じ開発者グループでも、dev では作れて prod では見るだけ、という**環境ごとの差**を権限で表すのがポイントです。

| グループ | dev サブスクリプション | prod サブスクリプション |
|---|---|---|
| grp-foundation-admins(管理者) | 所有者 | 所有者(常時ではなく PIM で一時昇格) |
| grp-foundation-developers(開発者) | 共同作成者 | 閲覧者 |
| grp-foundation-viewers(閲覧者) | 閲覧者 | 閲覧者 |

<p class="caption">図2-8 グループ別・環境別の RBAC 割り当て方針</p>

開発者は dev では**共同作成者**として自由にリソースを作れますが、prod では**閲覧者**に絞り、誤操作で本番を壊せないようにします。管理者は本番の所有者になれますが、その権限を**常時は持たせず**、PIM で必要なときだけ昇格させます。閲覧者グループは、監査担当や他チームのように「見るだけ」でよい人のための受け皿です。

スコープをサブスクリプションに置くと、その配下のリソースグループ・リソースへ権限が継承されます。より狭く効かせたい場合は、リソースグループやリソースをスコープにした割り当ても可能です。本書は分かりやすさを優先し、まずサブスクリプション スコープで割り当てます。

## 設計のポイント ── 最小権限とグループ経由の原則

RBAC の設計は、突き詰めると「**必要な人に、必要な権限を、必要な範囲だけ**」に尽きます。この最小権限を、運用が破綻しない形で実現するための原則を、チェックリストにまとめます。

| 分類 | 原則 | 理由 |
|---|---|---|
| セキュリティ | 個人に直接割り当てず、**グループ経由**で付与する | 退職・異動時にグループ出入りだけで済む |
| セキュリティ | 既定は**共同作成者**、所有者は絞る | 権限を配れる所有者を増やさない |
| セキュリティ | 本番の特権は**常時持たせず PIM で一時昇格** | 常時特権の乗っ取りリスクを下げる |
| 統制 | スコープは**必要な最小範囲**に置く | 上位に付けると継承で効きすぎる |
| 統制 | ロール割り当てを**棚卸し**する | 使われない権限を放置しない |
| コスト | 権限自体は無料。P1/P2 機能に注意 | 動的グループ・PIM はライセンスが要る |

<p class="caption">図2-9 RBAC 設計のチェックリスト</p>

### グループ経由の原則 ── 個人に直接割り当てない

いちばん大事な原則が、**権限を個人へ直接割り当てないこと**です。理由は運用にあります。個人へ直接割り当てると、その人が異動・退職したとき、どのスコープに何を割り当てたかを 1 件ずつ探して剥がす必要があります。割り当てを見落とせば、権限が残り続けます。

グループ経由なら、権限はグループに固定し、人はグループに出入りするだけです。退職時はグループから外せば、そのグループが持つすべての権限が一度に外れます。第 1 章で決めた「必須タグ」と同じく、**最初に原則を決めておくと後がずっと楽**になる典型例です。

### 段階的に強める ── 最小権限は「後から絞る」より「最初から絞る」

最小権限は、緩く始めて後から絞るのが最も難しい運用です。いったん配った権限を回収するのは、心理的にも運用面でも抵抗が大きいからです。だからこそ、**最初から共同作成者を既定にし、所有者は数を絞る**という初期設計が効きます。

本番の管理者権限は、さらに一歩進めて PIM で扱います。管理者グループには本番の所有者ロールを**割り当て可能(Eligible)な状態**にしておき、実際に強い操作が必要になったときだけ、申請と承認を経て一時的に**有効化(Activate)**します。有効化には多要素認証や理由の記録を必須にでき、期限が来れば自動で権限が外れます。「強い権限が常時ぶら下がっている」状態そのものを無くすのが狙いです。<!-- 要検証: 公式Doc確認 — PIM の Eligible/Active の用語と有効化フローの最新仕様 -->

<div class="column">
<span class="column-title">コラム:ライセンスで変わる機能</span>

本章で紹介した機能の一部は、Entra ID の上位ライセンスを前提とします。動的グループは P1、PIM は P2 で提供される、といった区分があります。無料枠やベーシックなライセンスでは、これらが使えないか制限されます。

学習段階では、まず「割り当て済みグループ+組み込みロール」だけで RBAC の型を体験すれば十分です。動的グループや PIM は「本番運用で効いてくる上位機能」と捉え、必要になった段階でライセンスを検討してください。個人の学習用テナントでいきなり有償ライセンスを買う必要はありません。<!-- 要検証: 公式Doc確認 — 動的グループ(P1)・PIM(P2)のライセンス要件と無料枠での制限 -->

</div>

## 構築手順 ── グループ作成とRBAC割り当ての動作確認

ここからは手を動かします。目的は、**3 つのグループを作り、dev と prod に強さの違う RBAC を割り当て、権限差が実際に効くことを確かめる**ことです。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

### 1. グループを作成する

まず、管理者・開発者・閲覧者の 3 つのセキュリティグループを Entra ID に作成します。

```bash
# 開発者グループ
az ad group create \
  --display-name grp-foundation-developers \
  --mail-nickname grp-foundation-developers

# 閲覧者グループ
az ad group create \
  --display-name grp-foundation-viewers \
  --mail-nickname grp-foundation-viewers

# 管理者グループ
az ad group create \
  --display-name grp-foundation-admins \
  --mail-nickname grp-foundation-admins
```

作成したグループの一覧を確認します。

```bash
az ad group list --output table
```

### 2. グループにユーザーを追加する

権限は個人ではなくグループに割り当てるため、まず自分(または対象ユーザー)を該当グループのメンバーにします。ここでは開発者グループに追加する例を示します。

```bash
# 対象ユーザーのオブジェクト ID を控える
az ad user show --id "you@example.com" --query id --output tsv

# 開発者グループにメンバーとして追加する
az ad group member add \
  --group grp-foundation-developers \
  --member-id "<上で取得したオブジェクト ID>"
```

### 3. dev サブスクリプションに RBAC を割り当てる

開発者グループに、dev サブスクリプションの**共同作成者**を割り当てます。スコープにはサブスクリプション ID を指定します。

```bash
# dev サブスクリプションの ID を控える
az account show --subscription "sub-foundation-dev" \
  --query id --output tsv

# 開発者グループ = 共同作成者(dev サブスクリプション スコープ)
az role assignment create \
  --assignee-object-id "<開発者グループのオブジェクト ID>" \
  --assignee-principal-type Group \
  --role "Contributor" \
  --scope "/subscriptions/<dev サブスクリプションの ID>"
```

### 4. prod サブスクリプションに RBAC を割り当てる

同じ開発者グループを、prod サブスクリプションでは**閲覧者**に絞ります。これで「dev では作れるが、prod では見るだけ」という差が生まれます。

```bash
# 開発者グループ = 閲覧者(prod サブスクリプション スコープ)
az role assignment create \
  --assignee-object-id "<開発者グループのオブジェクト ID>" \
  --assignee-principal-type Group \
  --role "Reader" \
  --scope "/subscriptions/<prod サブスクリプションの ID>"
```

閲覧者グループは dev/prod の両方に閲覧者を割り当てます(コマンドは上と同じ要領のため省略します)。管理者グループへの本番所有者の割り当ては、常時付与を避けて PIM で扱うため、ここでは行いません。

### 5. 権限差を動作確認する

割り当てが効いているかを、実際の操作で確かめます。開発者グループのメンバーとしてサインインし、dev ではリソースグループを作れること、prod では作成が拒否されることを確認します。

```bash
# 現在のサインインユーザーが持つロール割り当てを確認
az role assignment list \
  --assignee "you@example.com" \
  --all --output table

# dev では作成できる(共同作成者のため成功する想定)
az group create \
  --name rg-foundation-dev-test \
  --location japaneast \
  --subscription "sub-foundation-dev"

# prod では拒否される(閲覧者のため権限エラーになる想定)
az group create \
  --name rg-foundation-prod-test \
  --location japaneast \
  --subscription "sub-foundation-prod"
```

prod 側では、権限不足を示すエラー(認可に失敗した旨のメッセージ)が返る想定です。実際のエラー文言は検証ログ取得時に差し替えます。<!-- 要検証: handson-logs/ch02 の検証ログ未取得。prod 拒否時のエラーメッセージ実文言を確認する -->

<div class="note">
<span class="note-title">注意:確認用リソースは消しておく</span>

動作確認で dev に作ったテスト用リソースグループは、確認できたら削除します。削除はリソースグループごと行え、中のリソースもまとめて消えます。名前をよく確認してから実行してください(第 1 章の後片付けと同じ要領です)。

```bash
az group delete --name rg-foundation-dev-test --yes --no-wait
```

以上で、テナントに 3 つのグループができ、dev と prod に強さの違う RBAC が割り当たりました。「dev は緩く、prod は厳しく」を、権限の言葉で表現できたことになります。次章では、この権限の層に**ガバナンス**(管理グループと Azure Policy)を重ね、「そもそも何を作ってよいか」をルールで縛っていきます。

**この章のまとめ**

- **Entra ID** はテナントに一元管理される ID の基盤で、ユーザー・グループ・サービスプリンシパル・マネージド ID を保持します。IAM と違い、ID はアカウント(サブスクリプション)の外側にあります。
- **RBAC** は「**プリンシパル × ロール × スコープ**」を結ぶロール割り当てで表します。JSON を書くのではなく、組み込みロール(所有者・共同作成者・閲覧者)をスコープに割り当てて考えます。
- 権限は**個人でなくグループ経由**で与え、既定は**共同作成者**、本番の特権は**PIM で一時昇格**——最小権限を運用が破綻しない形で実現します。
- 第 2 章では、管理者/開発者/閲覧者の 3 グループを作成しました。dev と prod で強さの違う RBAC を割り当て、環境ごとの権限差を表現しています。
