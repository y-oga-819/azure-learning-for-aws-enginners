## 概要 ── 管理グループと Azure Policy

第 1 章で器の骨格(4 階層と ARM)を、第 2 章でそこに ID と権限の層(Entra ID と RBAC)を重ねました。ここまでで「**誰が**何をできるか」は表現できました。この章で足すのは、その一段上のルール——「**そもそも何を作ってよいか**」を組織として縛る層、すなわち **ガバナンス** です。使う道具は 2 つ、**管理グループ**と **Azure Policy** です。

第 1 章で、Azure のすべての操作は ARM という「一枚の関所」を通ると確認しました。そこで予告したとおり、第 2 章の RBAC も、この章の Azure Policy も、ARM を通る要求に対して効きます。両者の役割分担は明快です。

- **RBAC** は「**誰が**」を見ます。要求してきたプリンシパルに、その操作の権限があるかを判定します。
- **Azure Policy** は「**何を**」を見ます。作られようとしているリソースそのものが、組織のルール(このリージョンか、必須タグは付いているか、この SKU は許されるか)に合っているかを判定します。

権限があっても、ポリシーに反するリソースは作れません。逆に、ポリシー上は問題なくても、権限がなければ作れません。ID とガバナンスは、ARM という同じ通り道の上で、別の観点から要求を検査する二重のゲートなのです。

### 管理グループ:ルールを継承させる器

**管理グループ**は、第 1 章で作った統制の器です。複数のサブスクリプションを束ね、上位で 1 度決めたルールを配下すべてへ**継承**させます。第 1 章で `mg-foundation` を作り、その下に dev/prod のサブスクリプションを並べました。この章では、その管理グループを**ポリシーを効かせる足場**として本格的に使います。

ポイントは、ポリシーを**どのスコープに割り当てるか**です。管理グループに割り当てれば配下の全サブスクリプションに効き、特定のサブスクリプションに割り当てればそこだけに効きます。この継承の仕組みが、後述する「dev は緩く、prod は厳しく」を実現する土台になります。

### Azure Policy:定義・割り当て・効果

**Azure Policy** は、リソースが従うべきルールを定義し、それを ARM の要求に対して評価する仕組みです。構成要素を、要求が処理される順に並べておきます。

| 要素 | 役割 |
|---|---|
| ポリシー定義(Policy Definition) | 「何を・どう評価するか」1 つのルール。組み込み定義が多数用意される |
| イニシアティブ(Initiative) | 複数のポリシー定義を束ねた集合。まとめて割り当てる単位 |
| 割り当て(Assignment) | 定義またはイニシアティブを、あるスコープへ適用する行為 |
| 効果(Effect) | 評価結果に対して何をするか(監査・拒否・自動修復など) |

<p class="caption">図3-1 Azure Policy の構成要素</p>

つまり、**定義**でルールを決めます。必要なら**イニシアティブ**で束ね、対象スコープへ**割り当て**て、違反時の**効果**を選ぶという流れです。必須タグ・許可リージョン・許可 SKU といった典型的なルールは**組み込み定義**が用意されており、自分でルールを書かずに割り当てるだけで使えます。

この章で最も重要なのが、最後の**効果**です。同じルールでも、効果を変えれば「見つけるだけ」にも「作らせない」にもできます。

| 効果 | 挙動 | 使いどころ |
|---|---|---|
| 監査(Audit) | 違反しても**作成は許す**が、非準拠として記録する | まず現状把握。既存への影響がない |
| 拒否(Deny) | 違反するリソースの**作成・更新を止める** | ルールを強制する。関所として弾く |
| 自動修復(DeployIfNotExists / Modify) | 不足設定を**自動で足す・直す** | 診断設定の付与などを取りこぼさない |

<p class="caption">図3-2 ポリシーの主な効果</p>

この「監査で見つけ、拒否で止め、自動修復で直す」という 3 段が Azure Policy の核心です。AWS では複数サービスにまたがっていた統制を、1 つに束ねます。次節で AWS と対応づけながら、その守備範囲の広さを確認します。

## AWSとの対応と相違 ── SCP と Policy の守備範囲の違い

ガバナンスまわりを AWS と対応づけます。ここは 1 対 1 で綺麗に対応しない部分で、AWS では**複数のサービスに分かれていた役割を、Azure Policy が 1 つで担う**のが最大の違いです。

| 分類 | AWS | Azure | 似て非なる点 |
|---|---|---|---|
| 組織の器 | Organizations / OU | 管理グループ | 階層で統制を継承する点は同じ |
| 予防的統制 | SCP(Service Control Policy) | Azure Policy の拒否効果 | SCP は「上限の設定」、Policy はリソースの中身を見る |
| 発見的統制 | AWS Config / Config Rules | Azure Policy の監査効果 | 非準拠を検出して記録する |
| 自動修復 | Config の修復アクション / SSM | Azure Policy の自動修復効果 | 不足設定を自動で足す |

<p class="caption">図3-3 AWS のガバナンス群と Azure Policy の対応</p>

AWS 経験者は、**SCP**(Service Control Policy、サービスコントロールポリシー)と **AWS Config** を別々の道具として使い分けていたはずです。SCP で「そもそも禁止」を、Config で「作られたものが規約に沿うか」を見ていました。Azure ではこの両方を Azure Policy が兼ねます。順に、決定的な違いを 3 点掘り下げます。

### 決定的な違い1 ── SCP は「上限」、Policy は「中身」を見る

SCP は、アカウントに対する**権限の上限(ガードレール)**を設定する仕組みでした。「この OU では EC2 の起動を一切許可しない」のように、**どのアクションを許すか**の天井を下げます。SCP 自体は権限を**与えません**。あくまで、IAM で許可された操作の範囲をさらに狭めるフィルターです。

Azure Policy の拒否効果は、似て見えますが観点が違います。ポリシーが見るのは**アクションの可否**ではなく、**作られるリソースのプロパティ**です。「リージョンが東日本以外なら拒否」「`env` タグが無ければ拒否」「指定外の仮想マシン SKU なら拒否」——このように、リソースの**中身**を検査して判定します。

```text
【AWS】SCP:許可されたアクションの「上限」を下げる
  IAM が Allow ──▶ SCP のフィルタ ──▶ 実際に可能な操作
                  (Deny でさらに絞る)

【Azure】Azure Policy:作られるリソースの「中身」を見る
  作成要求 ──▶ RBAC で「誰が」を判定 ──▶ Policy で「何を」を判定 ──▶ 作成
                                        (リージョン・タグ・SKU …)
```

<p class="caption">図3-4 SCP は権限の上限、Azure Policy はリソースの中身を見る</p>

この違いから、RBAC と Azure Policy が**役割分担している**ことも改めて見えてきます。AWS では権限の話(IAM)と上限の話(SCP)が、どちらも「アクションの許可」という同じ土俵にありました。Azure では、アクションの許可は RBAC、リソースの中身の規約は Policy、と土俵が分かれています。

### 決定的な違い2 ── 検出だけの Config、止められる Policy

AWS Config は、**発見的統制**の道具でした。リソースの構成を継続的に記録し、Config Rules で「規約に沿っているか」を評価します。ただし Config は基本的に**作られた後に検出する**もので、非準拠を見つけても、その作成自体を止められませんでした。作成を止めるのは、SCP の役目でした。

Azure Policy は、この「検出」と「予防」を**効果の切り替えだけ**で行き来できます。同じ「東日本以外を禁止」というルールを、**監査**にすれば Config のように非準拠を記録し、**拒否**にすれば SCP のように作成を止めます。道具を替えるのではなく、1 つのポリシーの効果を替えるだけで守備範囲まで変わります。これが Azure の特徴です。

この性質が、次節と本章の核である「**監査から拒否への段階導入**」を自然にします。まず監査で影響範囲を把握し、問題なければ同じルールを拒否へ引き上げる——道具の乗り換えなしに、統制を段階的に強められます。

### 決定的な違い3 ── 自動修復まで一続きで担う

Azure Policy には、**自動修復**の効果があります。「診断設定が無ければ自動で付ける」「必要なタグが無ければ既定値で補う」といった修正を、評価と同時に走らせられます。AWS でも Config の修復アクションや Systems Manager で同様のことはできましたが、複数サービスの組み合わせが必要でした。Azure では、検出・予防・修復がポリシーという 1 つの体系の中に収まっています。

## 全体構成に組み込む ── devは緩く・prodは厳しく

第 2 章までで、器(管理グループ+サブスクリプション+リソースグループ)と、そこに割り当てた RBAC がそろいました。ただし、この時点では「**どんなリソースでも、権限さえあれば作れる**」状態です。タグの無いリソースも、海外リージョンのリソースも、権限があれば通ってしまいます。第 3 章では、ここに**ポリシーの層**を足し、作れるリソースそのものを規約で縛ります。

この章の核は、第 2 章と同じ合言葉——「**dev は緩く、prod は厳しく**」——を、今度は**ポリシー**で表現することです。第 2 章では権限の強さで差を付けました。この章では、**作ってよいリソースの範囲**で差を付けます。

### 何をどこに割り当てるか

割り当てるルールは 3 つです。継承を使い、**共通のルールは管理グループへ、prod だけ厳しくしたいルールは prod サブスクリプションへ**割り当てます。

| ルール | 効果 | 割り当てスコープ | ねらい |
|---|---|---|---|
| 必須タグ(`env` / `owner`) | まず監査、のちに拒否 | 管理グループ `mg-foundation` | dev/prod 共通。持ち主不明のリソースを無くす |
| 許可リージョンを東日本に限定 | 拒否 | 管理グループ `mg-foundation` | dev/prod 共通。想定外リージョンの事故を防ぐ |
| パブリック IP の作成を禁止 | 拒否 | サブスクリプション `sub-foundation-prod` | **prod のみ**。本番の不用意な公開を止める |

<p class="caption">図3-5 ポリシーの割り当て設計。共通は管理グループ、prod 限定は prod サブスクリプションへ</p>

必須タグとリージョン制限は、環境を問わず守りたい共通ルールなので、管理グループに割り当てて dev/prod の両方へ継承させます。一方、パブリック IP の作成禁止は、開発中は付けたい場面もあるため dev では緩め、**prod にだけ**割り当てます。継承の仕組みがあるからこそ、「共通は上で 1 回、環境差は下で個別」と書き分けられます。

### Before / After ── ガバナンスの層が加わる

第 2 章を終えた時点では、権限はあってもリソースの中身を縛るものはありませんでした。

```text
【Before】第2章の到達状態(権限はあるが、作れるものは無制限)
テナント(Entra ID)
├─ グループ: admins / developers / viewers(RBAC 割り当て済み)
│
└─ 管理グループ: mg-foundation
   ├─ サブスクリプション: sub-foundation-dev
   │  └─ リソースグループ: rg-foundation-dev
   └─ サブスクリプション: sub-foundation-prod
      └─ リソースグループ: rg-foundation-prod
```

<p class="caption">図3-6 第2章の到達状態(Before)</p>

第 3 章を終えると、管理グループに共通ポリシーが、prod サブスクリプションに追加の厳しいポリシーが割り当たり、配下へ継承されます。

```text
【After】第3章の到達状態(ポリシーの層を追加)
テナント(Entra ID)
├─ グループ: admins / developers / viewers(RBAC 割り当て済み)
│
└─ 管理グループ: mg-foundation
   │  ◆ ポリシー: 必須タグ(env/owner) … 監査→拒否
   │  ◆ ポリシー: 許可リージョン=東日本 … 拒否
   │  (配下の dev/prod 両方へ継承)
   │
   ├─ サブスクリプション: sub-foundation-dev
   │  └─ リソースグループ: rg-foundation-dev(共通ポリシーが継承)
   └─ サブスクリプション: sub-foundation-prod
      │  ◆ ポリシー: パブリック IP 作成の禁止 … 拒否(prod 限定)
      └─ リソースグループ: rg-foundation-prod(共通+prod 限定が継承)
```

<p class="caption">図3-7 第3章の到達状態(After)。共通は管理グループ、prod 限定は prod サブスクリプションへ</p>

これで、dev と prod の差が**権限**(第 2 章)と**作れるリソースの範囲**(第 3 章)の両面から表現されました。開発者は dev では自由に作れますが、それでも「タグ無し」「海外リージョン」は共通ポリシーに弾かれます。prod ではさらにパブリック IP まで塞がれ、本番の不用意な公開が構造的に起きなくなります。

## 設計のポイント ── 監査から拒否への段階導入

ポリシー設計でいちばん事故が多いのは、**いきなり拒否から始めること**です。既存リソースや進行中の作業を巻き込み、「昨日まで作れたものが今日から作れない」という混乱を生みます。これを避けるための原則を、チェックリストにまとめます。

| 分類 | 原則 | 理由 |
|---|---|---|
| 統制 | **監査 → 拒否**の順で段階導入する | いきなり拒否は既存・進行中の作業を壊す |
| 統制 | 割り当てスコープは**継承**を意識して選ぶ | 上位に付けると配下すべてに効く |
| 統制 | 例外は**除外(Exemption)**で期限付きに管理する | 恒久的な穴を作らない |
| セキュリティ | prod ほどルールを厚くする | 本番の事故の影響が大きい |
| コスト | ポリシー機能自体は**無料** | 割り当て・評価に課金はない |

<p class="caption">図3-8 ポリシー設計のチェックリスト</p>

### 監査から始める ── まず現状を知る

新しいルールを導入するときは、まず**監査**で割り当てます。監査モードは作成を止めないため、既存リソースにも進行中の作業にも影響を与えず、「**いま何件が違反しているか**」だけを可視化します。第 1 章・第 2 章で `env` / `owner` タグを付ける約束にしていても、実際には付け忘れが混ざっているものです。監査でその全体像を把握します。

違反が把握でき、対象を直し終えたら、同じルールの効果を**拒否**へ引き上げます。ここで初めて「タグの無いリソースは作らせない」という強制が効きます。前節で見たとおり、Azure Policy はこの切り替えを**効果を替えるだけ**で行えます。道具を乗り換える必要はありません。

```text
段階導入の流れ
  ① 監査で割り当てる    → 違反を可視化(作成は止めない)
  ② 違反を洗い出し直す  → 既存リソースにタグを補う
  ③ 拒否へ引き上げる    → 以後は違反リソースを作らせない
```

<p class="caption">図3-9 監査から拒否への段階導入</p>

### 例外は除外で、恒久ではなく期限付きに

運用していると、「このリソースだけは一時的にルールから外したい」場面が必ず出ます。このとき、ポリシーの割り当て自体を消したり効果を緩めたりすると、**全体の穴**になります。Azure Policy には、特定のスコープをそのポリシーの評価対象から外す**除外(Exemption)**の仕組みがあります。

除外は、対象・理由・そして**期限**を添えて登録するのが原則です。期限を切らない除外は、いつしか「なぜ外れているのか誰も分からない恒久的な穴」になります。第 2 章の PIM と同じ発想で、**例外は一時的に、期限付きで**扱います。

<div class="note">
<span class="note-title">注意:拒否ポリシーは既存リソースを消さない</span>

拒否効果のポリシーが止めるのは、**これから行われる作成・更新**です。すでに存在する違反リソースを、ポリシーが自動で削除することはありません。そのため、監査で見つけた既存の違反は、拒否へ引き上げる前に**手で直しておく**必要があります。「拒否にすれば古い違反も消える」と誤解しないでください。

</div>

### 複数ルールはイニシアティブで束ねる

ルールが増えてくると、割り当てを 1 つずつ管理するのは煩雑になります。関連するポリシー定義を**イニシアティブ**にまとめておけば、割り当て・除外・準拠状況の確認を**束ねた単位**で行えます。本書では扱うルールが少ないため個別に割り当てますが、実運用では「本番ベースライン」のようなイニシアティブを 1 つ作り、それを管理グループへ割り当てる形が管理しやすくなります。

## 構築手順 ── 必須タグポリシーで違反を弾く

ここからは手を動かします。目的は、**必須タグ(`env`)のポリシーを prod へ割り当て、タグの無いリソースの作成が実際に拒否されることを確認する**ことです。前節の段階導入に沿い、まず監査で現状を見てから拒否へ引き上げます。操作は原則 Azure CLI(`az`)で示し、ポータルでも同じことができます。

以下のコマンド・出力は、`handson-logs/ch03/` の検証ログ取得後に実値へ差し替えます。<!-- 要検証: handson-logs/ch03 の検証ログ未取得。コマンド・出力を実機で確認する -->

### 1. 使う組み込みポリシー定義を確認する

「指定したタグが無ければ弾く」ルールには、組み込みのポリシー定義があります。まず、その表示名で検索して定義 ID を控えます。

```bash
# 「Require a tag」系の組み込み定義を探す
az policy definition list \
  --query "[?policyType=='BuiltIn' && contains(displayName, 'Require a tag')].{name:name, displayName:displayName}" \
  --output table
```

<!-- 要検証: 公式Doc確認 — 必須タグの組み込みポリシー定義の正式な表示名と定義 ID -->

### 2. まず監査モードで割り当てる(現状把握)

段階導入の第 1 歩として、prod サブスクリプションを対象に**監査**で割り当てます。ここでは必須タグとして `env` を指定します。監査なので、既存リソースの作成は止まりません。

```bash
# prod サブスクリプションを対象にする
az account set --subscription "sub-foundation-prod"

# 必須タグ(env)を「監査」で割り当てる
az policy assignment create \
  --name "require-tag-env-audit" \
  --display-name "必須タグ env(監査)" \
  --scope "/subscriptions/<prod サブスクリプションの ID>" \
  --policy "<手順1で控えた定義 ID>" \
  --params '{ "tagName": { "value": "env" } }'
```

### 3. 準拠状況を確認する

割り当て後、準拠状況の評価が走ります(反映には少し時間がかかります)。非準拠(タグの無い既存リソース)がどれだけあるかを確認します。

```bash
az policy state summarize \
  --subscription "sub-foundation-prod" \
  --output table
```

ここで見つかった既存の違反リソースには、拒否へ引き上げる前に `env` タグを補っておきます。

### 4. 拒否モードへ引き上げる

現状を把握し、既存の違反を直し終えたら、同じルールを**拒否**へ引き上げます。ここでは監査用の割り当てを削除し、拒否効果で割り当て直します。

```bash
# 監査用の割り当てを削除する
az policy assignment delete \
  --name "require-tag-env-audit" \
  --scope "/subscriptions/<prod サブスクリプションの ID>"

# 必須タグ(env)を「拒否」で割り当て直す
az policy assignment create \
  --name "require-tag-env-deny" \
  --display-name "必須タグ env(拒否)" \
  --scope "/subscriptions/<prod サブスクリプションの ID>" \
  --policy "<手順1で控えた定義 ID>" \
  --params '{ "tagName": { "value": "env" } }'
```

<!-- 要検証: handson-logs/ch03 の検証ログ未取得。組み込みの必須タグ定義が拒否効果を持つか、効果指定が必要かを実機で確認する -->

### 5. 違反リソースが弾かれることを確認する

拒否が効いているかを、実際の作成で確かめます。`env` タグを**付けずに**リソースグループを作ろうとして、拒否されることを見ます。

```bash
# タグ無しで作成 → ポリシーに拒否される想定
az group create \
  --name rg-foundation-prod-policytest \
  --location japaneast \
  --subscription "sub-foundation-prod"
```

ポリシーに違反するため、どのポリシー割り当てに拒否されたかを含むエラーが返る想定です。実際のエラーコード・文言は検証ログ取得時に差し替えます。<!-- 要検証: handson-logs/ch03 の検証ログ未取得。拒否時のエラーコード・メッセージ実文言を確認する -->

続けて、`env` タグを**付ければ**同じ作成が通ることを確認します。

```bash
# env タグを付けて作成 → 今度は成功する想定
az group create \
  --name rg-foundation-prod-policytest \
  --location japaneast \
  --subscription "sub-foundation-prod" \
  --tags env=prod owner="you@example.com"
```

<div class="note">
<span class="note-title">注意:確認用リソースは消しておく</span>

動作確認で作ったテスト用リソースグループは、確認できたら削除します。名前をよく確認してから実行してください(第 1 章・第 2 章の後片付けと同じ要領です)。

```bash
az group delete --name rg-foundation-prod-policytest --yes --no-wait
```

</div>

以上で、必須タグのポリシーを監査から拒否へ段階的に導入し、違反リソースが実際に弾かれることを確認できました。管理グループに割り当てる共通ポリシー(必須タグ・リージョン制限)や、prod 限定のパブリック IP 禁止も、スコープと効果を替えれば同じ要領で割り当てられます。

**この章のまとめ**

- **ガバナンス**は「そもそも何を作ってよいか」を縛る層です。「誰が」を見る RBAC に対し、**Azure Policy** は作られるリソースの「**何を**(中身)」を見ます。両者は ARM の上で働く二重のゲートです。
- **管理グループ**にポリシーを割り当てると配下へ**継承**されます。共通ルールは管理グループへ、環境差は個別スコープへ。これで「dev は緩く、prod は厳しく」をポリシーで表現できます。
- AWS では **SCP**(上限設定)と **AWS Config**(検出)に分かれていた統制を、Azure Policy が**監査・拒否・自動修復**の効果切り替えで 1 つに束ねます。
- 導入は**監査 → 拒否**の順が原則です。まず監査で違反を可視化し、既存を直してから拒否へ引き上げます。例外は**除外**で期限付きに管理します。
